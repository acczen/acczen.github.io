两数之和

两数之和

核心思路

第一反应：暴力枚举

拿到这道题，我们的第一反应往往很直接——两层循环，挨个检查每一对数字的和是否等于 target。

这个解法没错，但让我们想想它的代价。对于数组中的每一个元素，我们都要遍历它后面所有的元素。如果数组有 n 个数，第一层循环跑 n 次，第二层平均跑 n/2 次，总的比较次数大约是 n^2 / 2。时间复杂度是 O(n^2)。

空间上我们只用了两个循环变量，额外空间是 O(1)。

当 n 很大的时候，O(n^2) 的算法就会变得很慢。LeetCode 的测试数据中数组长度可能达到 10^4 甚至 10^5 级别，n^2 就是 10^8 到 10^10 次操作——显然不可接受。

那么问题来了：有没有办法减少重复计算？

换个角度看问题

让我们重新审视一下题目。对于每个元素 nums[i]，我们真正想找的是：在数组中是否存在另一个数 x，使得 nums[i] + x == target。

移项一下：x = target - nums[i]。

所以问题变成了：对于当前元素，我们想知道 target - nums[i] 之前有没有出现过。

这个视角的转变很关键。我们不是在”找两个数的组合”，而是在”查一个数是否出现过”。

用空间换时间

“查一个数是否出现过”——什么数据结构最擅长做这件事？哈希表（Hash Table），它的查找时间复杂度是 O(1)。

于是我们可以这样改进：

遍历数组的过程中，把每个元素的值和它的下标存到哈希表里。对于当前元素 nums[i]，检查 target - nums[i] 是否已经在哈希表中。如果在，我们就找到了答案。

这里有一个细节需要注意：题目要求同一个元素不能使用两次。我们是在遍历到 nums[i] 的时候去查哈希表，而哈希表里存的是之前遍历过的元素，所以当前元素和哈希表中的元素一定是不同的下标，天然满足题目的要求。

让我们想一想：我们应该先把所有元素都放进哈希表再查找，还是边遍历边放？

如果先把所有元素都放进哈希表再查找，会有一个问题：如果数组中有重复元素（比如 nums = [3, 3], target = 6），后一个 3 会覆盖前一个 3 的下标。但更关键的是，我们可能错误地使用同一个元素两次——例如 nums = [3, 2, 4], target = 6，如果先全部放入，查 3 的时候哈希表里 target - 3 = 3 指向的就是它自己。

所以正确做法是边遍历边查，查完再放入。这样查的时候，哈希表里只有当前元素之前的元素，绝不会误用自身。

最优解的核心思想

最终的核心思路只有一句话：

在遍历数组的过程中，用哈希表记录已经遍历过的元素值及其下标，对于当前元素，检查 target - nums[i] 是否存在于哈希表中。

这个过程像不像是我们在玩一个配对游戏——每遇到一个新数字，我们就看看它的”另一半”是不是已经来过了。

复杂度分析表

版本 时间复杂度 空间复杂度 说明
暴力法 O(n^2) O(1) 两层循环遍历所有数对，不消耗额外空间
哈希表法（最优解） O(n) O(n) 单层遍历，哈希表最多存储 n 个元素

为什么哈希表法是 O(n)？因为遍历数组一次，每次操作（查表和插入）都是 O(1) 的哈希表操作。空间上哈希表最多存 n 个键值对，所以是 O(n)。

对比之下，暴力法虽然空间省，但时间上慢了一个数量级——这就是用空间换时间的经典案例。

延伸思考

这道题让我们学到了什么？

哈希表是一种强大的”以空间换时间”的工具。 当我们遇到”查找一个元素是否存在”或者”记录元素出现位置”这类需求时，首先想到的就应该是哈希表。

另外，这道题揭示了一个重要的思考方式：有时候把问题转化一下，答案就变得清晰了。 从”找两个数和为 target”到”查 target - nums[i] 是否出现过”，只是移了一下项，但解决问题的思路完全不同了。这种”换个角度定义问题”的能力，在算法学习中会反复用到。

下次遇到类似问题优先观察什么？

  1. 是否能通过等式变换，把”找组合”转化为”查存在性”？ 凡是”找两个元素满足某个条件”的问题，都值得尝试这个思路。
  2. 是否需要记录元素的额外信息？ 这道题需要返回下标，所以哈希表的值存的是下标。如果只问是否存在（比如变体题”两数之和是否存在”），哈希表的值只需要存布尔值，甚至用 HashSet 就够了。
  3. 同一个元素能否被重复使用？ 这个约束会影响我们是在遍历前全部放入，还是边遍历边放入。务必在动笔之前想清楚这个细节。

LeetCode 的第一题，看似简单，却承载了”暴力 -> 优化 -> 空间换时间”这一完整的思维训练。后面的很多题目，都会沿用这套思考框架。掌握它，远远比背下这道题的代码更重要。
