相交链表

相交链表

核心思路

从最直观的想法开始

题目给了两个单链表的头节点 headA 和 headB，让我们判断它们是否相交。换句话说，是否存在某个节点，同时出现在两个链表中？

最直接的想法——我们先把其中一个链表的所有节点记录下来，然后遍历另一个链表，逐个检查当前节点是否在记录中出现过。这就像我们在一个陌生的体育馆找人：先记住所有人的座位，然后挨个去找，看谁坐在同一个座位上。

这个方案的时间复杂度是 O(m + n)，但空间复杂度也是 O(m + n)——我们用了一个哈希表来存储节点引用。

让我们问自己：能不能不用额外空间呢？

空间 O(1) 的方向在哪里

我们不让用哈希表，那就意味着只能靠移动指针来”记住”位置。两个链表各自走一遍，怎么对齐它们的步伐呢？

不妨想一下：如果两个链表相交，那么相交之后的节点是完全一样的。也就是说，两个链表的尾部是共享的。我们如果能先走到两个链表的尾部，再从尾部对齐往前找，问题就解决了……可惜这是单向链表，没法从后往前走。

换个角度——如果两个链表长度相等呢？

如果长度相等，事情就简单了：两个指针分别从 headA 和 headB 出发，每次各走一步，第一次相遇的节点就是交点。如果走到 null 都没相遇，说明不相交。

但题目没有说长度相等。一个链表可能比另一个长出一截，长的那个前面多出来的节点肯定不可能是交点（因为交点之后的部分两个链表共享）。所以长链表的前面多出来的部分可以直接跳过。

思路出现了：先让长的链表走掉长度差，然后两个指针同步前进。

更优雅的解法：双指针交替遍历

上面的思路需要先计算两个链表的长度，需要两次遍历。能不能一次遍历就解决？

让我们想象一下：两个指针 pA 和 pB 分别从 headA 和 headB 出发。pA 走完 A 链表后，让它从 headB 重新出发；pB 走完 B 链表后，让它从 headA 重新出发。这样会发生什么？

假设链表 A 的长度为 m，链表 B 的长度为 n，相交部分长度为 c（不相交则为 0）。

  • pA 走过的路径：A 链表（m 步）+ B 链表的非相交部分（n - c 步）
  • pB 走过的路径：B 链表（n 步）+ A 链表的非相交部分（m - c 步）

两个指针走过的总步数分别是 m + (n - c) 和 n + (m - c)。当它们都走完 m + n - c 步时，两个指针指向的节点相等——正是交点。

如果链表不相交，c = 0，两个指针都会同时走到 null，此时返回 null。

这个解法巧妙之处在于，它通过交换路径来”消除”长度差，让两个指针在交点处自然相遇。

图解直觉

当 pA 走到了 null（走完 A 的 5 个节点），它跳到 B 的开头 b1；当 pB 走到了 null（走完 B 的 6 个节点），它跳到 A 的开头 a1。之后它们会在 c1 相遇，因为从此刻开始，两个指针剩下的路径长度相等。

延伸思考

这道题让我们学到了什么？

路径叠加消除差异。 这道题的核心洞察是：当两个实体走过的”总路程”不一致时，让它们互相交换到对方的轨迹上继续走，最终会在共同的终点相遇。这种思想在很多问题中都会出现——比如在环形赛道上的追逐问题。

代码的简洁不等于思路的简单。 双指针解法只有寥寥几行，但背后蕴含的观察——两个链表相交后尾部重合、通过交换路径来对齐步长——并不直观。理解这道题的过程告诉我们，写出好的代码之前，先要有一个好的思路。

下次遇到类似问题优先观察什么？

  1. 链表的结构特征。 单向链表只能向前走，所以”对齐”是常见的操作模式。
  2. “走完自己的路，再走别人的路”这个模式。 当问题涉及两个序列的比较或寻找共同点时，交换路径是一个值得尝试的策略。
  3. 能否转化为环的问题？ 如果我们将 B 的尾部连到 B 的头部，那么问题就变成了”在 A 链表中寻找环的入口”，这是另一个巧妙的角度（虽然会修改链表结构，需要额外恢复）。

从暴力到优雅，从直观到精妙——这正是算法学习的乐趣所在。
