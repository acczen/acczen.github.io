每日温度

每日温度

核心思路

从题目说起

给定一个整数数组 temperatures，表示每天的温度。我们需要返回一个等长的数组 answer，其中 answer[i] 表示在第 i 天之后，需要等多少天才能遇到更高的温度。如果之后的气温都不会再升高，则该位置为 0。

举个例子：

第 0 天 73°，第二天 74° 就更高了，所以答案是 1；第 2 天 75°，要等到第 6 天 76° 才更高，相隔 4 天，所以答案是 4。最后两天之后没有更高的温度了，填 0。

看起来不算复杂，那我们从最直觉的方式开始。

第一反应：双层循环

最直接的想法是：对于每一天，往后遍历，找到第一个比它高的温度，记录距离。

这个写法几乎不需要思考。但问题也很明显——每一天都要往后扫描，最坏情况下（温度单调递减）内层循环要走完整个数组，总复杂度是 O(n²)。当输入规模达到 10^5 时，这会超时。

那么，能不能让每一天不重复扫描那些已经看过的元素？

让我们停下来想一个问题：当我们处理第 i 天时，我们真正关心的是什么？是未来第一个比它大的值的位置。而未来尚未处理，我们不知道。

但换个方向呢？如果我们从右往左处理，已经处理过的部分就是”未来”的信息——这听起来有道理，但似乎还是需要某种方式快速找到”第一个更大的”。

让我们换一个更常见的视角。

换个角度：不是为每一天找未来，而是为未来记录今天

假设我们正从左向右遍历。当我们站在第 i 天时，我们不知道未来会发生什么。但是，对于之前那些还没有遇到更高温度的日子来说，今天的温度可能就是它们的答案。

换句话说，我们维护一个集合，里面存放那些”还在等待更高温度”的日子。每遇到一个新温度，就去看看这个集合里有没有比它还矮的——如果有，那些矮日子的答案就是今天。

这个”还在等待的”日子有什么特征呢？如果我们把它们的温度画出来，会发现它们一定是递减的。为什么？因为如果某一天的温度高于它后面的某天，那么后面的那天在等待时，前面那天也在等待——而一旦遇到更高的温度，会先满足前面那天（因为它离得更近？不对，是更高的那天会先满足栈顶的元素）。

等等，让我们重新整理一下直觉。

单调递减栈

我们维护一个栈，栈里存的是还没有找到更高温度的日子的下标，并且这些下标对应的温度在栈中从底到顶是递减的。

为什么保持递减？假设栈里有下标 a 和 b，a 在 b 下面（a 先入栈），那么 temperatures[a] >= temperatures[b]。如果新来的温度 t 大于栈顶的温度，那么栈顶就找到了答案——它的答案就是 i - stack.pop()。

弹出栈顶后，新的栈顶温度仍然大于等于之前那个（因为栈是递减的），如果 t 比它还大，继续弹出并更新答案。直到栈为空或者栈顶温度大于等于 t，我们再把当前下标 i 入栈。

这样，每个元素最多入栈一次、出栈一次，总复杂度 O(n)。

让我们用手上的例子走一遍来验证：

步骤 当前温度 栈（下标 -> 温度） 动作 更新结果
0 73 [] 栈空，下标0入栈
1 74 [0 -> 73] 74 > 73，弹出0，answer[0]=1-0=1，下标1入栈 [1]
2 75 [1 -> 74] 75 > 74，弹出1，answer[1]=2-1=1，下标2入栈 [1,1]
3 71 [2 -> 75] 71 < 75，直接入栈
4 69 [2 -> 75, 3 -> 71] 69 < 71，直接入栈
5 72 [2 -> 75, 3 -> 71, 4 -> 69] 72 > 69，弹出4，answer[4]=5-4=1 [,,,,1]
[2 -> 75, 3 -> 71] 72 > 71，弹出3，answer[3]=5-3=2 [,,_,2,1]
[2 -> 75] 72 < 75，下标5入栈
6 76 [2 -> 75, 5 -> 72] 76 > 72，弹出5，answer[5]=6-5=1 [,,,,_,1]
[2 -> 75] 76 > 75，弹出2，answer[2]=6-2=4 [,,4,,,1]
[] 栈空，下标6入栈
7 73 [6 -> 76] 73 < 76，直接入栈

遍历结束，栈里剩下的下标6、7对应的答案保持为0。最终结果：[1, 1, 4, 2, 1, 1, 0, 0]。

完美匹配。

核心洞察： 单调栈的核心不在于栈本身，而在于我们利用了信息的延迟匹配——当温度 t 到来时，它一次性解决了栈中所有比它小的温度的”宿命”。栈保证了那些等待中的温度按降序排列，新来的温度只需要与栈顶比较即可。

为什么栈是递减的，而不是递增的？

这是一个很好的问题。如果我们维护一个递增栈，栈底最小、栈顶最大。那么新来的元素如果小于栈顶，栈顶永远不会等到更高的温度（因为新来的更小），所以递增栈帮不了我们。

我们需要的是：栈顶是最”迫切”的那一天——它温度最低，离当前最近，最容易找到更高的温度。递减栈正好做到这一点：栈顶是最小的（最晚入栈的），它只要遇到一个比它大的就能出栈。

完整 Java 代码

关键细节：

  • 栈里存的是下标而非温度——因为我们要计算距离 i - prevIndex，存下标一举两得。
  • 用 ArrayDeque 而非 Stack（Stack 是线程安全的遗留类，性能较差）。
  • 循环结束后栈中剩余元素直接保留默认值 0。

完整 Python 代码

Python 直接用列表模拟栈即可，stack[-1] 取栈顶，stack.pop() 弹出栈顶。

复杂度分析表

版本 时间复杂度 空间复杂度 说明
暴力双层循环 O(n²) O(1)（不计输出数组） 每个元素向后扫描；最坏情况温度严格递减
单调栈 O(n) O(n)（栈 + 输出数组） 每个元素入栈一次、出栈一次，n 为天数

时间：单调栈版本中，外层循环遍历 n 个元素，内层 while 在整个算法中累计执行不超过 n 次（每个元素至多出栈一次），因此总操作次数为 O(n)。

空间：输出数组 answer 占 O(n)；栈在最坏情况下（温度严格递减）会存储所有 n 个下标，因此额外空间 O(n)。

延伸思考

这道题让我们学到了什么？

“下一个更大元素”的通用解法就是单调栈。

本题是”下一个更大元素”系列的经典原型（LeetCode 496、503、556 等都是同类问题）。核心思想是：在遍历数组的过程中，利用栈暂存”等待匹配”的元素，当前元素一次性解决所有可以匹配的等待者。这种”延迟匹配”的思路在括号匹配、直方图最大矩形（LeetCode 84）等问题中反复出现。

换个方向看问题往往豁然开朗。

从”为每一天往后找”变成”为栈顶元素找答案”，仅仅是把遍历和处理的方向调转了一下，复杂度就从 O(n²) 降到了 O(n)。这种视角转换是算法优化中最有力的武器之一。

维护栈的单调性是为了保证算法的正确性。

递减栈保证了栈顶永远是当前”最急迫”（温度最低、离当前位置最近）的元素。新来的温度只需要跟栈顶比较就能决定是否要弹出——如果比栈顶大，那么栈顶找到了答案；如果不比栈顶大，那栈中所有元素都不会被它满足（因为栈中温度 >= 栈顶温度），所以直接入栈即可。

下次遇到类似问题优先观察什么？

  1. 问题是否在问”下一个更大的 / 更小的 / 更远的”？ 如果是，单调栈应当立即进入候选列表。
  2. 是否每个元素都需要跟它后面的元素比较？ 如果是，尝试反向思考——让当前元素去”解决”前面等待的元素。
  3. 如果数组改为循环数组怎么办？ 思考一下：如何用”拼接”或”取模”的方式扩展单调栈解法？（这就是 LeetCode 503 下一个更大元素 II 的解法思路。）
