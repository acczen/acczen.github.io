单词拆分

题目描述

给定一个非空字符串 s 和一个包含非空单词列表的字典 wordDict，判定 s 是否可以被空格拆分为一个或多个在字典中出现的单词。

说明：

  • 拆分时可以重复使用字典中的单词。
  • 可以假设字典中没有重复的单词。

核心思路

从第一直觉出发：暴力枚举

拿到这道题，我们最直接的想法是什么？

不妨想一下：我们从左到右扫描字符串，每次切出一个前缀，检查它是否在字典中。如果在，就对剩下的后缀重复同样的过程。只要有一条路径能走到字符串末尾，就说明可以成功拆分。

这本质上是一棵决策树：在每个位置，我们都可以选择切一刀，或者不切（继续延伸当前单词）。那么，一共有多少种切法呢？对于长度为 n 的字符串，有 n-1 个可以切的位置，每个位置都有切与不切两种选择，所以最坏情况下的时间复杂度是 O(2^n)。

让我们用一个例子来感受一下：

你会发现大量的子问题被反复计算：”sandog” 可能从 “cat”+”sandog” 这条路径进入，也可能从其他切法进入。每次都要重新探索一遍，太浪费了。

于是我们反问自己：有没有办法让每个子问题只计算一次？

这自然把我们引向了记忆化搜索（Memoization）。

加个缓存：自顶向下 DP

我们定义一个递归函数 canBreak(start)，表示从 start 位置开始的后缀 s[start:] 能否被拆分。

对于每个 start，我们枚举下一个单词的结束位置 end（从 start+1 到 n）。如果 s[start:end] 在字典中，并且 canBreak(end) 为真，那么 canBreak(start) 就为真。

每次计算出结果后，用数组 memo 缓存起来。下次再遇到同样的 start，直接返回缓存值。

这样，每个 start 位置只计算一次，状态总数为 O(n)，每次转移需要枚举 O(n) 个结束位置，时间复杂度降到了 O(n²)。

从递归到递推：自底向上 DP

既然有了递归+记忆化的思路，我们很自然地会想：能不能直接用递推来写？这样不需要递归栈，也更直观。

定义 dp[i] 表示字符串的前缀 s[0:i]（长度为 i 的前缀）是否可以拆分成字典中的单词。

边界条件：dp[0] = true，空串总是可以拆分的。

状态转移方程：

$$ dp[i] = \bigvee_{j=0}^{i-1} \big( dp[j] ; \land ; s[j:i] \in \text{wordSet} \big) $$

也就是说，对于每个位置 i，我们回头看所有 j < i，如果 s[0:j] 可拆分（dp[j] = true）且 s[j:i] 在字典中，那么 s[0:i] 就可拆分。

还能再快一点吗？

上面的双层循环是 O(n²)。当字符串长度很大时（比如长达几百甚至上千），这个复杂度可能不太理想。

让我们换个角度思考：我们真的需要枚举所有的 j 吗？如果我们反过来，从字典里的单词出发呢？

对于每个位置 i，我们不枚举所有可能的切分点，而是遍历字典中的每一个单词 w。如果 i >= len(w)，且 s[i-len(w):i] 恰好等于 w，并且 dp[i-len(w)] 为真，那么 dp[i] 就可以设为真。

这样，每次转移只需要 O(m) 次检查（m 是字典大小），而不是 O(n) 次。当字典中的单词数量远小于字符串长度时，这个优化非常显著。

另一种视角：BFS 图遍历

等等，我们能不能把这个问题看成一张图？

把字符串的每个位置 0, 1, 2, ..., n 看作图中的节点。如果 s[i:j] 是字典中的单词，就有一条从 i 到 j 的边。那么问题就转化为：是否存在一条从节点 0 到节点 n 的路径？

这不就是一个标准的图可达性问题吗？BFS 或 DFS 都可以解决。

这种视角的好处在于，一旦到达位置 n，就可以立即返回，不需要遍历所有状态。在最好情况下，如果最短路很短，BFS 可以提前退出。

让我们总结一下思考的脉络：

  1. 暴力回溯：枚举所有切分方式 —— 指数爆炸，不可行
  2. 发现重叠子问题：相同的后缀被反复计算
  3. 记忆化搜索：缓存子问题结果，自顶向下，O(n²)
  4. 自底向上 DP：将递归转为递推，从短前缀逐步推导出完整结果
  5. 转移优化：从遍历切分点改为遍历字典单词，利用字典规模小的特点
  6. BFS 图视角：将问题抽象为图的路径搜索，可以提前退出

每一步都是基于前一步的不足产生的自然思考，而不是凭空跳出来的技巧。这就是动态规划的核心思想：定义状态、分解子问题、缓存结果、优化转移。

复杂度分析

版本 时间复杂度 空间复杂度 说明
暴力回溯 O(2^n) O(n) 递归栈深度，指数级不可行
记忆化搜索 O(n²) O(n) 每个位置计算一次，每次枚举 O(n) 个结束位置
自底向上 DP（遍历切分点） O(n²) O(n) 双层循环，n 为字符串长度
自底向上 DP（遍历单词） O(n × m × L) O(n) m 为字典大小，L 为单词最大长度。当 m 较小时优于 O(n²)
BFS 图遍历 O(n²) O(n) 最坏仍需遍历所有状态，但可提前终止

空间复杂度补充： 所有 DP 解法都需要 O(n) 的 DP 数组，加上 O(m) 的哈希集合（m 为字典大小），总空间 O(n + m)。记忆化搜索额外需要递归栈空间 O(n)。

延伸思考

这道题让我们学到了什么？

1. 状态定义的艺术

这道题最精妙的地方在于 dp[i] 的定义——它表示的是前缀 s[0:i] 的可拆分性，而不是后缀。这个”前缀视角”让我们可以从短到长、从左到右递推，非常自然。如果反过来定义后缀，虽然也可以做，但递推的方向就不那么直观了。

2. 子问题分解的边界

dp[0] = true 这个边界条件值得玩味。它代表”空串”是可拆分的，这看似平凡，却为整个递推提供了起点。很多 DP 问题的边界条件都是这样设计的——一个平凡的”什么都没有”的状态作为地基。

3. 转移的两种视角

  • “回头看”：枚举所有切分点 j，检查前缀 + 当前子串
  • “向前匹配”：枚举字典中的单词，检查是否能拼接在当前前缀末尾

前者是”位置驱动”，后者是”模式驱动”。在实践中，当字典规模小时，模式驱动更快；当字典规模大而字符串长度小时，位置驱动更直接。根据数据特点选择转移策略，是我们从这道题中得到的实用启发。

4. 同一问题，多种抽象

DP、记忆化搜索、BFS 图遍历——三种方法看似不同，实则都建立在同一张”状态图”上：

  • DP 是拓扑序的递推
  • 记忆化搜索是带缓存的 DFS
  • BFS 是按层遍历搜索最短路径

理解它们之间的等价关系，能让我们在面对新问题时更灵活地选择最自然的实现方式。

下次遇到类似问题优先观察什么？

  • 是否可分解为独立子问题？ — 如果能将大问题切成若干独立的小问题求解，DP 通常可行
  • 子问题是否有重叠？ — 如果有重复计算，就需要缓存（记忆化或 DP）
  • 状态如何定义？ — 前缀/后缀、区间、位置组合…找到合适的维度定义状态是关键
  • 转移方程怎么推导？ — 最后一步做了什么？从哪些更小的状态可以走到当前状态？
  • 能否优化转移？ — 枚举所有可能的转移还是只枚举一部分？有没有利用题目特性（比如字典大小很小）来加速？
