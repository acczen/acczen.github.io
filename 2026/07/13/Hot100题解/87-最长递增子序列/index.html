最长递增子序列

最长递增子序列

核心思路

从暴力开始

题目很直接：给定一个无序的整数数组 nums，让我们找出最长严格递增子序列的长度。

子序列（subsequence）和子数组（subarray）不同——子序列不要求连续，只要保持相对顺序即可。

最朴素的想法是什么？枚举所有子序列，检查每个子序列是否递增，记录最长的长度。对于一个长度为 n 的数组，一共有 2^n 个子序列（每个元素选或不选），复杂度爆炸，显然不可行。

有没有办法减少搜索空间？

让我们换个角度。如果已知以某个元素结尾的最长递增子序列长度，能不能递推出后续元素的结果？

贪心 + 二分查找——革命性的优化

思考一个直觉问题：同样长度的递增子序列，我们更希望末尾元素尽可能小。因为末尾越小，后续元素越容易接上去，越有可能形成更长的序列。

比如 nums = [2, 3, 1]：

  • 长度为 1 的递增子序列有 [2]、[3]、[1]
  • 显然 [1] 最有”前途”，因为它末尾最小，后面随便来个大于 1 的数就能扩展

这个直觉引导出一个巧妙的数据结构——tails 数组：

我们遍历 nums 中的每个元素 x，在 tails 中二分查找第一个大于等于 x 的位置：

  • 如果找到，说明可以用 x 替换该位置的值，使该长度的末尾更小
  • 如果没找到（x 比所有末尾都大），说明可以扩展出一个更长的子序列，追加到 tails 末尾

延伸思考

这道题让我们学到了什么？

一、从”以谁结尾”的角度定义状态。
子序列问题的经典 DP 套路——dp[i] 定义为”以第 i 个元素结尾的某性质”。对于子序列类问题（最长递增子序列、最长公共子序列等），这种定义方式天然保证了子问题的独立性，因为子序列必须保持原顺序，而”结尾位置”正好划清了问题边界。

二、贪心思想 + 二分查找可以优化”最长递增”类问题。
当 DP 中的内层循环只是在找”满足条件的最大值/最小值”时，如果被查找的数组具有单调性，就可以用二分查找优化。tails 数组的单调递增性质，恰好让二分查找有机可乘。

三、有时我们不需要具体的序列，只需要长度。
tails 数组存储的不是真正的 LIS，但足以给出正确的长度。这提醒我们：根据问题需求设计数据结构——问什么就存什么，不追求完整信息反而可能更高效。

下次遇到类似问题优先观察什么？

  1. 先判断子序列还是子数组。 连续 → 滑动窗口/前缀和；不连续 → 动态规划。
  2. 尝试 dp[i] 以 i 结尾的定义。 对于每个 i，回头看 0..i-1 能否递推。
  3. 如果 O(n^2) 太慢，看单调性。 能否用二分查找优化内层循环？tails 数组的单调性、最长递增子序列的”末尾越小越好”的贪心性质，都是经典的优化突破口。
  4. 变形题思路。 如果题目改成”非递减”（允许相等），只需要把二分查找从 bisect_left 改成 bisect_right（插入位置从第一个 >= x 变成第一个 > x）。如果要求输出具体的 LIS 序列，可以在贪心的同时用 parent 数组记录每个元素来自哪个前驱，最后回溯构造。
