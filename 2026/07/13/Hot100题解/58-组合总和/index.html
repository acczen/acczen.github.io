组合总和

组合总和

核心思路

从一个朴素的问题开始

让我们先想一想：给定一组无重复的候选数字 [2,3,6,7]，要从中选出若干个（可以重复选），使得它们的和恰好等于目标值 7，该怎么找全所有组合？

最直觉的想法是什么？枚举所有可能的组合，检查每个组合的和是否等于 7。

但这里有一个麻烦：每个数字可以被无限次选取，这意味着组合的”长度”是不确定的——可能是 1 个数字（[7]），也可能是 3 个（[2,2,3]），甚至更多。如果我们想用循环来枚举，应该嵌套多少层循环呢？我们事先并不知道，因为组合的长度是未知的。

这让我们遇到了第一个困境。

如何枚举长度不固定的组合？

不妨换一个角度思考。我们不是在”选固定数量的数字”，而是在做一系列连续的”选择”：先选一个数，然后基于这个结果再选下一个数，直到总和达到目标或者超过目标。

这听起来像什么？像是一棵不断分叉的决策树。

让我们以 candidates = [2,3,6,7]、target = 7 为例，画一画这棵树：

  • 从根节点（空组合，和为 0）出发，我们可以选 2、3、6 或 7。
  • 如果选了 2，当前和为 2，还差 5。接下来又可以选 2、3、6、7……
  • 如果选了 3，当前和为 3，还差 4……

这棵树会不断生长，直到某条路径的和等于 7（找到了一个答案）或大于 7（此路不通）。

这正是回溯算法的基本形态：深度优先搜索 + 状态回退。

等等，这里有一个细节——起始索引是用来做什么的？

让我们想一下：题目说组合中数字的顺序不重要，[2,2,3] 和 [2,3,2] 被认为是同一个组合。如果我们每次递归都从 candidates 的开头遍历，就会出现重复。例如，选 2 -> 2 -> 3 和 2 -> 3 -> 2 都会被搜索到。

为了避免这种重复，我们规定：每次只能从当前数字或之后的位置开始选择。这样一来，如果先选了 2，后面只能选 2、3、6、7（可以重复选 2）；如果先选了 3，后面就只能选 3、6、7（不能回头选 2）。这样就不会产生重复组合了。这就是为什么我们需要传入一个 startIndex 参数。

但这个朴素版本有一个问题：对于某些输入，搜索树可能非常庞大，做了很多无用功。

如何剪枝？——思考何时可以提前停止

让我们观察一下：如果当前的 candidates 是 [2,3,6,7]，当前和已经是 6，还差 1。此时再去尝试 [6, ...] 这种组合还有意义吗？6 本身就已经大于剩余需要的 1 了，何况更大的数字？显然没有意义。

所以一个很自然的优化是：先对 candidates 排序，这样在递归过程中，一旦发现当前数字已经大于剩余需要的和，就可以直接跳出循环，后面的数字更大，更不可能满足条件。

这就是剪枝——提前终止那些注定找不到解的搜索分支。

完整思路总结

把上面的思考串起来，完整的算法流程就是：

  1. 排序 candidates，方便后续剪枝。
  2. 定义回溯函数 dfs(startIndex, currentCombo, currentSum)：
    • 如果 currentSum == target：将 currentCombo 的副本加入答案，返回。
    • 遍历 i 从 startIndex 到 candidates.length - 1：
      • 如果 currentSum + candidates[i] > target：剪枝，直接 break（因为数组已排序，后面的更大）。
      • 将 candidates[i] 加入 currentCombo。
      • 递归调用 dfs(i, currentCombo, currentSum + candidates[i])（注意起始索引仍然是 i，因为可以重复选）。
      • 回溯：将最后一个元素移除。
  3. 从 dfs(0, [], 0) 开始搜索。

让我们对比一下排序前后朴素回溯的效率：如果不排序，即使当前数字已经超过剩余和，我们还必须继续遍历后面的所有数字；排序后一趟 break 就跳过了整个后续分支。

复杂度分析

版本 时间复杂度 空间复杂度 说明
朴素回溯 $O(N^{T/M})$，其中 $N$ 是候选数组长度，$T$ 是目标值，$M$ 是候选数字的最小值 $O(T/M)$ 搜索树的高度最坏为 $T/M$（每次都选最小数），每层最多 $N$ 个分支。未剪枝时可能遍历大量无效分支
排序 + 剪枝回溯 最坏仍为指数级，但实际远小于朴素版本 $O(T/M)$ 排序预处理 $O(N \log N)$。剪枝大幅减少搜索空间，实际运行效率显著提升

说明：组合总和问题的本质是 NP-Complete 的子集和问题的变体，因此最坏情况下时间复杂度仍然是指数级的。但在题目给定的约束下（组合数不超过 150 个），回溯 + 剪枝已经足够高效。空间复杂度主要取决于递归栈的深度（最多 $T / \min(candidates)$ 层）和存储当前路径的临时列表。

延伸思考

这道题让我们学到了什么？

回溯算法的本质是暴力搜索 + 状态回退。面对”枚举所有可能性”的问题时，回溯是我们最有力的工具之一。但回溯本身只是骨架，真正的灵魂在于剪枝——通过深入分析问题的结构，提前砍掉那些注定无解的搜索分支，让指数级的搜索空间变得可行。

回看这道题，剪枝只用了两个简单的观察：

  1. 排序后，一旦当前数字超过剩余目标，后面的更大，不可能满足条件。
  2. 通过 startIndex 控制搜索方向，避免了重复组合的产生。

很多时候，好的算法并不是凭空创造出来的，而是从暴力解法出发，不断问自己”这里有必要搜索吗？””那里能提前终止吗？”，一步步优化出来的。

下次遇到类似问题优先观察什么？

当我们遇到组合、排列、子集这类”所有可能解”的问题时，不妨先问自己几个问题：

  1. 要不要去重？ 如果数字可以重复选取（本题），起始索引保持 i；如果每个数字只能用一次，起始索引变为 i+1。
  2. 能不能剪枝？ 排序往往是剪枝的第一步。排序后，如果当前选择已经违规，后续更大的选择必然同样违规，可以 break。
  3. 状态怎么回退？ 在递归前后做好添加和移除的操作——这是回溯的”起手式”，写熟了就不会忘。

回溯 + 剪枝 = 暴力搜索的优雅进化。掌握这个思想，就能从容应对一大类”枚举所有可能”的问题了。
