合并区间

核心思路

0. 问题理解

先看题目：给定一个区间集合 intervals，其中每个区间 [starti, endi]，合并所有重叠的区间，返回不重叠的区间数组。

比如 [[1,3],[2,6],[8,10],[15,18]]，[1,3] 和 [2,6] 有重叠，合并成 [1,6]，最终返回 [[1,6],[8,10],[15,18]]。

1. 暴力解法 —— 每次合并后重新检查

如果没有任何技巧，我们第一反应会怎么做？

最直觉的做法：遍历所有区间，对于当前区间，检查它和后面每一个区间是否有重叠。如果有，就合并，然后从头开始重新检查——因为合并产生的新区间可能又和前面的区间重叠了。

这个做法的问题在哪里？每次合并都要重新扫描，最坏情况下我们可能要扫描 $O(n)$ 次，每次扫描又需要 $O(n^2)$ 的检查——时间复杂度高达 $O(n^3)$。即使我们用一些标记来优化，也很难降到 $O(n^2)$ 以下。

不妨想一下：为什么我们需要反复扫描？ 核心原因在于，合并后的新区间可能向前”覆盖”——它可能变长了，和前面已经检查过的区间又重叠了。如果我们能提前安排好顺序，让每个区间只需要和它后面的区间比较一次，是不是就不用回头了？

2. 排序优化 —— 让重叠自然呈现

带着上面的问题思考：有什么办法让合并只需要一次遍历？

让我们换个角度。假设我们先把所有区间按左端点从小到大排序。排序之后，一定有以下性质：

  • 如果区间 $A$ 和区间 $B$ 重叠，那么 $A$ 的右端点一定 $\ge$ $B$ 的左端点（因为 $A$ 在 $B$ 前面，$A$ 的左端点 $\le$ $B$ 的左端点）。
  • 更重要的是：合并后的新区间，左端点就是当前区间的左端点，右端点是当前区间右端点和下一个区间右端点的较大值。

这个性质让我们不需要回头。因为排序后，新区间的左端点不会比前面的区间更小（不会向左覆盖），只需要关心向右扩展。

3. 最终解法 —— 一次遍历合并

有了排序的保证，算法就变得非常直接了：

  1. 将所有区间按左端点升序排序
  2. 初始化结果列表，将第一个区间加入结果
  3. 遍历剩下的区间，对于每个区间 [l, r]：
    • 如果 l <= lastR（上一个区间的右端点），说明有重叠 → 更新 lastR = max(lastR, r)
    • 否则没有重叠 → 将新区间加入结果列表

版本 时间复杂度 空间复杂度 说明
暴力法 $O(n^3)$ $O(n)$ 每次合并后重新扫描所有区间，最坏情况反复合并
排序优化 $O(n \log n)$ $O(\log n)$ 或 $O(n)$ 排序 $O(n \log n)$，遍历 $O(n)$；空间取决于排序算法的栈空间（Java 的 Arrays.sort() 对原数组是 $O(\log n)$）或结果列表（$O(n)$）

为什么排序能解决这个问题？ 排序把”二维”的重叠判断，变成了”一维”的右端点比较。排序后每个区间只需要和它紧挨着的前一个区间比较，不需要回头——这就是排序的力量。

延伸思考

这道题让我们学到了什么？

区间问题的核心套路——排序预处理。 很多区间类问题（合并区间、插入区间、无重叠区间、会议室等）都有一个共同点：区间之间的关系（重叠、包含、相离）在排序后变得极其清晰。排序让我们可以把 $O(n^2)$ 的暴力比较压缩成 $O(n)$ 的一次扫描。

更抽象地说，这体现了预处理简化问题的思想：不要一上来就在原始数据上硬做，问问自己——如果先排个序、如果先建个哈希表、如果先做前缀和，数据之间的关系会不会变得一目了然？

下次遇到类似问题优先观察什么？

  1. 问题的输入是否是区间/线段/时间范围？ 如果是，排序大概率是第一步。
  2. 重叠的定义是什么？ [1,3] 和 [3,5] 算不算重叠？这道题里，3 <= 3 算重叠（闭区间），但有些题目会明确排除相接的情况。
  3. 比较的标准是左端点还是右端点？ 合并区间按左端排序；无重叠区间按右端排序（贪心）——不同的目标对应不同的排序策略。
  4. 合并后是否可能产生新的连锁重叠？ 只要排序方向正确，一次遍历就够，不需要回溯——这是我们可以放心的地方。

最后留一个小思考：如果输入的区间是无序的流式数据（不能排序），我们还能高效合并吗？这时候可以考虑用区间树（Interval Tree） 或线段树来维护——这又是另一个话题了。
