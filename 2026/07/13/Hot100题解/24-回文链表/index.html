回文链表

回文链表

核心思路

题目： 给定一个单链表的头节点，判断它是否为回文链表。回文的定义是正着读和反着读一样。

让我们先停一下——看到”回文”这两个字，你的第一反应是什么？

第一反应：数组双指针

如果是数组，判断回文太简单了：左指针从开头、右指针从结尾，向中间走，一旦遇到不相等就返回 false。但这次是链表——我们没法直接从尾巴往前走。

最直接的想法浮出水面：把链表复制到数组中，然后用数组双指针判断。

这个解法没问题，但空间复杂度是 O(n)。题目下方的进阶挑战写着：”你能否用 O(n) 时间复杂度和 O(1) 空间复杂度解决？”

为什么进阶挑战总喜欢卡空间？——因为链表的魅力就在于它只占用了必要的存储，我们不应该为了判断回文就额外拷贝一整份数据。

追问：不用额外空间，怎么比较？

如果我们只能操作原链表，那不得不问自己一个问题：想要比较首尾元素是否相等，而链表又只能从前往后走，该怎么办？

一个直觉：如果我们能把后半段链表”反过来”，让它也能从后往前遍历，问题就迎刃而解了。

这就引出了链表的经典操作——反转链表。

但反转整条链表没有意义，反转后我们就丢失了原始顺序。我们需要的是只反转后半部分，然后前半部分和后半部分逐节点比较。

再追问：怎么找到”后半部分”？

等等，我们先要确定”后半部分”的起点在哪里。

我们需要找到链表的中间节点。对于偶数长度的链表，中间有两个节点，前半部分的结尾是第一个中间节点；对于奇数长度链表，中间那个节点在回文判断中可以被忽略（它和自己对称）。

寻找链表中间节点——这个问题我们见过，快慢指针恰好就是为此而生。慢指针每次走一步，快指针每次走两步，当快指针走到末尾时，慢指针正好在中间。

如果链表长度是奇数，比如 1 -> 2 -> 3 -> 2 -> 1，快指针结束时慢指针指向 3，前半部分应该是 1 -> 2，后半部分是 2 -> 1（反转后）。

现在整理一下完整方案

到这里，整个思路已经清晰了：

  1. 快慢指针找中点——慢指针边走边收集前半段？不，我们更简单的做法是先找到中点。
  2. 反转后半段链表——从中点下一个节点开始反转。
  3. 逐一比较——前半段和反转后的后半段逐节点比对。
  4. 恢复链表（可选项）——如果不想破坏原链表结构，可以把后半段再反转回来。

让我们仔细推敲一下边界：

  • 当链表为空或只有一个节点时，直接返回 true。
  • 快慢指针的停止条件：快指针走到 null 或快指针的 next 为 null 时停止。
  • 对于奇数长度链表，中间节点可以跳过——如果我们让慢指针最终停在中间节点，那么后半部分的起点就是 slow.next。
  • 偶数长度的情况，慢指针停在第一个中间节点，后半部分起点也是 slow.next。

也就是说，不管奇数还是偶数，slow.next 都是后半部分的起点——因为快指针停止时，慢指针总是停在前半部分的最后一个节点上。

延伸思考

这道题让我们学到了什么？

判断回文不一定需要从两端同时向中间遍历。 当数据结构不支持双向访问时（如单链表），我们可以换一个思路：把一半的数据”翻转”过来，然后单向比较。

这种”以反转换方向”的思路在链表中反复出现。比如：

  • 反转链表本身——让我们能从后往前看链表。
  • 两数相加 II——先把两个链表分别反转，然后从个位开始加。
  • 重排链表——找到中点后反转后半部分，然后交替合并。

下次遇到类似问题优先观察什么？

数据结构的访问特性决定算法方向。

拿到一道题，先问自己三个问题：

  1. 这个数据结构支持什么方向的遍历？（单向？双向？随机访问？）
  2. 如果只能单向，是否能通过局部反转来获得逆向访问的能力？
  3. 是不是一定要完整反转？还是只反转一半就够了？

回到回文本题，我们的本质需求是将第 i 个节点与第 n-i+1 个节点做比较。对于数组，下标访问 O(1) 解决了；对于链表，我们等价地实现了”从两端向中间比较”，只不过以修改链表结构为代价。如果题目不允许修改原链表，那在返回结果前把后半段再反转一次还原回去即可——时间仍然是 O(n)，常数因子稍大一点而已。

小结： 回文链表这道题看似简单，实则巧妙地把”数组双指针”和”链表反转”两个基础技能组合在了一起。它提醒我们：复杂问题往往是基础操作的有机组合，吃透每一个基础操作，才能在组合时游刃有余。
