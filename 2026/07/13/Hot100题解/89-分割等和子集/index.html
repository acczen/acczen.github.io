分割等和子集

分割等和子集

核心思路

从问题本质开始

题目要求我们把一个数组分成两个子集，使得两个子集的元素之和相等。

让我们先想一个最基本的问题：如果两个子集的和相等，那么每个子集的和应该是多少？

假设整个数组的总和为 sum，那么每个子集的目标和就是 sum / 2。这立刻引出一个关键观察：如果 sum 是奇数，答案直接就是 false——因为整数不可能平分成两个相等的一半。

所以问题转化为：给定一个数组，能否选出一些元素（一个子集），使得它们的和恰好等于 sum / 2？

第一反应：暴力枚举

最朴素的想法是枚举所有子集，检查每个子集的和是否等于目标值。

一个大小为 n 的数组有 2^n 个子集，每个子集求和需要 O(n) 时间，总复杂度 O(n · 2^n)。当 n 达到 200 时，这完全不现实。

但我们不妨从这个暴力思路出发，想想看它有什么可以优化的地方？

这个问题的本质是什么？

让我们换一个角度思考：每个元素我们要么选、要么不选——这不就是0-1 背包问题吗？

回顾一下 0-1 背包的模型：

  • 有 n 个物品，每个物品有一定重量
  • 有一个容量为 C 的背包
  • 问能否恰好装满背包？

在这个问题中，每个数字就是物品的”重量”，目标值 sum / 2 就是背包的容量。我们想知道是否存在一种选择方式，恰好装满这个背包。

这样一想，思路就清晰了。

二维 DP：逐步构建

定义 dp[i][j] 表示从数组的前 i 个元素中选，能否凑出和为 j。

状态转移方程：

初始化：dp[0][0] = true，表示前 0 个元素可以凑出和为 0（空集）。对于 j > 0，dp[0][j] = false。

让我们走一个例子来验证：nums = [1, 5, 11, 5]，总和为 22，目标值为 11。

  • 处理 1：可以凑出 0, 1
  • 处理 5：可以凑出 0, 1, 5, 6
  • 处理 11：可以凑出 0, 1, 5, 6, 11, 12, 16, 17
  • 处理 5：可以凑出 0, 1, 5, 6, 10, 11, … 最终得到 11 ✓

压缩到一维

观察二维 DP 的转移，dp[i][j] 只依赖 dp[i-1][...] 这一行的数据。因此我们可以用一个一维数组，从后向前更新来节省空间。

为什么从后向前？因为 dp[j] 依赖的是上一轮的 dp[j] 和 dp[j - num]。如果从前向后更新，dp[j - num] 可能已经被本轮更新覆盖了，就不再是上一轮的状态了。

复杂度分析

版本 时间复杂度 空间复杂度 说明
暴力枚举 O(n · 2^n) O(n) 枚举所有子集并求和，n=200 时不可行
二维 DP O(n · target) O(n · target) target = sum/2，最坏情况 n·sum/2
一维 DP O(n · target) O(target) 滚动数组优化，只保留一行

其中 target = sum / 2。在最坏情况下，数组元素都接近 10^4，n = 200，target 最大为 10^6。此时 O(n · target) 约为 2×10^8，在 Java 中仍可通过（配合剪枝），但 Python 可能需要进一步优化。

延伸思考

这道题让我们学到了什么？

识别问题的本质模式比技巧更重要。

当我们看到”选或不选”的决策问题时，第一反应不应该是枚举所有可能性，而是问自己：这是不是一个背包问题？

背包问题的标志性特征是：

  • 每个元素有”选”和”不选”两种状态
  • 有一个容量/目标约束
  • 问是否存在某种选择方案

下次遇到类似问题优先观察什么？

  1. 总和的性质——如果涉及平分，先检查总和的奇偶性，这是一个 O(1) 的过滤条件
  2. 数值范围——target 的大小决定了 DP 是否可行；如果 target 很大但 n 很小，或许可以换用回溯+剪枝
  3. 能否剪枝——这道题的早期剪枝（一旦 target 可达就返回）在实践中能大幅提升性能
  4. 一维还是二维——只要转移只依赖上一行，就可以用一维数组+逆序更新来优化空间

进一步思考

这道题是 0-1 背包问题的变体，但它只问”能否”而不问”最多能装多少”。如果题目改成”能否用 k 个子集平分数组”（比如 LeetCode 698「划分为k个相等的子集」），我们就需要更复杂的搜索+回溯+剪枝策略了。

另外，这道题还有一个有趣的变种：如果数组元素可以重复使用（完全背包），解法又会有不同。这些细微的差异提醒我们，读懂题目中每个字的限制条件，往往比直接套模板更重要。
