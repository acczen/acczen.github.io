搜索旋转排序数组

搜索旋转排序数组

核心思路

从最朴素的想法开始

题目给了我们一个排序数组，但它在某个未知的 pivot 处旋转了。比如 [0,1,2,4,5,6,7] 可能变成 [4,5,6,7,0,1,2]。要求在 O(log n) 时间内找到 target。

先想想最简单的方法：遍历一遍数组，逐个比较，找到就返回下标。时间复杂度 O(n)，空间复杂度 O(1)。

但题目要求 O(log n)，这让我们想起了什么？二分查找。

二分查找在这里遇到了什么困难？

常规的二分查找要求数组整体有序，我们才能通过比较中间值和 target 来决定往左还是往右。但现在数组被旋转了——它不再是整体有序的。

不过，让我们仔细观察一下旋转后的数组。虽然整体无序，但如果我们从中间切一刀，会发生什么？

以 [4,5,6,7,0,1,2] 为例，mid = 3，nums[3] = 7。左半部分是 [4,5,6,7]——它是完全有序的！右半部分是 [7,0,1,2]——虽然包含旋转点，但如果我们从 mid+1 看，[0,1,2] 也是有序的。

关键发现：无论数组在哪个位置旋转，在任意一个 mid 处切割后，至少有一半是有序的。

如何判断哪一半是有序的？

这很简单：比较 nums[left] 和 nums[mid]。

  • 如果 nums[left] <= nums[mid]，说明左半部分没有旋转点，左半部分是有序的。
  • 否则，右半部分是有序的。

这里有个小细节：为什么要用 <= 而不是 <？考虑到数组长度为 1 或 2 的情况，当 left == mid 时，我们仍然认为左半部分是有序的。

判断 target 在不在有序的那一半

假设左半部分有序。我们检查 target 是否落在这个区间里：

  • 如果 nums[left] <= target < nums[mid]，说明 target 在左半部分，我们往左搜。
  • 否则，target 在右半部分，我们往右搜。

同理，如果右半部分有序：

  • 如果 nums[mid] < target <= nums[right]，说明 target 在右半部分，往右搜。
  • 否则，往左搜。

整体流程

让我们走一遍完整的例子：nums = [4,5,6,7,0,1,2], target = 0

  1. left = 0, right = 6, mid = 3, nums[3] = 7

    • nums[0] <= nums[3] → 左半部分有序
    • target = 0 不在 [4, 7) 范围内 → 往右走
    • left = 4

  2. left = 4, right = 6, mid = 5, nums[5] = 1

    • nums[4] <= nums[5] → 左半部分有序
    • target = 0 在 [0, 1) 范围内吗？nums[4]=0 <= target=0 < nums[5]=1 → 是的！
    • 往左走，right = 4

  3. left = 4, right = 4, mid = 4, nums[4] = 0

    • nums[4] == target → 返回 4

找到了！

边界情况

  • 数组只有一个元素：nums = [1]，mid = 0，nums[left] == nums[mid]，左半部分有序，正常比较即可。
  • 数组没有旋转：相当于 nums[left] <= nums[mid] 始终成立，退化为标准二分查找。
  • 旋转点在开头或末尾：例如 [1,2,3,4] 或 [2,3,4,1]，算法同样正确处理。

复杂度分析表

版本 时间复杂度 空间复杂度 说明
暴力遍历 O(n) O(1) 逐个扫描，不满足题目 O(log n) 要求
二分查找 O(log n) O(1) 每次缩小一半搜索范围，仅使用常数空间

两种实现的空间复杂度都是 O(1)，但时间复杂度上的差距在数据量较大时会非常明显——当 n = 10^8 时，二分查找最多只需约 27 次比较，而暴力遍历可能需要上亿次。

延伸思考

这道题让我们学到了什么？

旋转数组的核心性质：从任意位置切一刀，至少有一半是有序的。

这个性质让二分查找在看似无序的结构中找到了立足之地。类似的思路还可以用到很多问题上——比如在旋转数组中找最小值（LeetCode 153），或者在某个值未知的单调函数上做查找。

还有一个更抽象的收获：当标准算法因为条件不满足而无法使用时，不要直接放弃。想想条件被破坏后，数据还保留了哪些结构？旋转打乱了整体有序性，但没有完全消灭有序性——它只是把有序性分裂成了两段。

下次遇到类似问题优先观察什么？

  1. 题目的时间复杂度要求：O(log n) 几乎总是二分查找或其变体。
  2. 数据的结构特征：有序数组被旋转，本质上还是两段有序数组拼接而成。抓住”至少一半有序”这个性质。
  3. 边界条件的处理：二分查找的边界最容易被忽视。<= 还是 <，left = mid 还是 mid + 1——差之毫厘，谬以千里。写完后一定要用单元素和两元素的数组验证一下。

最后留一个小思考：如果数组中有重复元素，上面的算法还能用吗？如果不能，问题出在哪里？如何才能修复它？（提示：这道题的进阶版本是 LeetCode 81）
