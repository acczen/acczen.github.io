和为 K 的子数组

和为 K 的子数组

核心思路

第一反应：暴力枚举

拿到这道题，如果没有任何技巧，我们第一反应会怎么做？

“找出所有连续子数组，分别求和，然后数一数有多少个和为 k。”

这个思路非常直接。我们可以写两层循环，外层枚举子数组的起点 i，内层枚举终点 j，同时累加 i 到 j 的和。

这个解法的问题在哪里？

时间复杂度是 O(n²)——对于 n = 10⁵ 的规模，大约需要 10¹⁰ 次操作，显然会超时。

空间复杂度是 O(1)，时间却是硬伤。

停下来想一想：有没有办法减少重复计算？

我们观察一下暴力法的计算过程：对于同一个终点 j，我们需要从每个起点 i 重新累加到 j。这里面其实有很多重复劳动。

不妨想一下——如果我们能快速算出任意一个连续子数组 nums[i..j] 的和，是不是就能省掉内层的累加？

这个改写非常关键。

它告诉我们：当我们遍历到位置 j 时（终点），我们不是在找起点 i，而是在找之前出现过多少次值为 prefix[j+1] - k 的前缀和。

换句话说，我们只需要一边遍历、一边用哈希表记录每个前缀和出现的次数，然后对于当前位置，查一下 当前前缀和 - k 之前出现过几次，累加即可。

最终的核心思想

  1. 维护一个哈希表 map，key 是前缀和，value 是该前缀和出现的次数。
  2. 初始化 map[0] = 1，表示前缀和为 0 出现过一次（一个元素都不取的情况），这是因为子数组可以从第一个元素开始，此时 prefix[j+1] - k = 0。
  3. 遍历数组，计算当前前缀和 curSum。
  4. 检查 curSum - k 在哈希表中出现的次数，累加到答案中。
  5. 将 curSum 的出现次数加一。

这样，我们只遍历了一次数组，时间复杂度 O(n)，空间复杂度 O(n)。

让我们用一个小例子走一遍：

位置 元素 当前前缀和 curSum - k map 中已有次数 累加 map
初始 - 0 - - 0 {0:1}
0 1 1 -2 0 0 {0:1, 1:1}
1 2 3 0 1 1 {0:1, 1:1, 3:1}
2 3 6 3 1 2 {0:1, 1:1, 3:1, 6:1}

最终结果为 2，对应的子数组是 [1,2] 和 [3]。

延伸思考

这道题让我们学到了什么？

将”连续子数组的和”转化为”两个前缀和的差”，是一种极其经典的优化思路。当我们需要快速判断一个连续区间的性质时，前缀和往往是第一把钥匙。而加上哈希表之后，我们又把”找差等于 k 的对”转化成了”找值为某个固定值的元素出现次数”，这本质上是用空间换时间——用 O(n) 的额外空间，把 O(n²) 的枚举降到了 O(n)。

另一个值得注意的点是初始化 prefixSumMap[0] = 1。这个细节很容易被忽略，但恰恰是它保证了从第一个元素开始的子数组也能被正确计数。边界条件的处理，往往藏在最简单的直觉背后。

下次遇到类似问题优先观察什么？

  • “连续子数组”+”和为某个值”：立刻想到前缀和。这是最强烈的信号。
  • “子数组”+”计数”：当问题要求的是”有多少个”而不是”最长/最短”时，哈希表统计出现次数往往是关键。
  • “连续子数组”+”可正可负”：这就排除了滑动窗口（双指针）的解法，因为滑动窗口依赖单调性。此时前缀和 + 哈希表是首选。

这道题是前缀和系列的基础题。掌握它之后，不妨继续挑战 “和可被 K 整除的子数组”（974） 和 “连续的子数组和”（523）——它们的思路一脉相承，都是前缀和 + 哈希表的不同变体。
