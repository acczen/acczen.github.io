跳跃游戏

跳跃游戏

核心思路

从问题本身开始

给定一个非负整数数组 nums，我们一开始站在下标 0 的位置，每个位置上的数字代表我们最多能向前跳多远。问题是：我们能否跳到最后一个下标？

让我们先理解一下”最多能跳多远”的含义。如果 nums[i] = 3，意味着从位置 i 出发，我们可以选择跳到 i+1、i+2 或 i+3 这三个位置中的任意一个。我们的目标是在一系列跳跃之后，到达或超过 nums.length - 1。

看起来就是一个路径探索问题，对吧？

暴力尝试：把所有可能性都走一遍

最直接的想法是：从起点出发，列举所有可能的跳跃步数，一条路一条路地尝试，只要能到达终点就返回 true。

这不就像 DFS 吗？从根节点出发，每个位置分支出若干条路径，只要有任意一条路径能走到终点就成功。

但等一下——这个算法的时间复杂度是多少？每个位置最多可以分支 nums[i] 条路径，最坏情况下（比如数组全为 1），复杂度是 $O(2^n)$ 级别的。当数组长度达到 $10^4$ 或 $10^5$ 时，这显然跑不动。

有没有什么办法可以避免重复探索？

记忆化：记下已经走过的路

我们之所以会慢，是因为同一个位置可能被从不同的路径多次到达。比如从 0→1→2 和从 0→2 都到达了位置 2，但我们没有必要在位置 2 重复计算两次。

我们可以用一个 memo 数组记录每个位置是否可达终点：

  • memo[i] = 1 表示从 i 出发可以到达终点
  • memo[i] = -1 表示从 i 出发无法到达终点
  • memo[i] = 0 表示还没计算过

这样每个位置只会被计算一次，时间复杂度降为 $O(n \times \max(nums[i]))$。

但再想一想——我们需要记录每个位置的可达性吗？或者说，我们真的需要关心”具体从哪个位置跳”吗？

换个角度：关注”能到达的最远位置”

让我们换一种视角来思考这个问题。

从起点开始，我们不断扩展自己的”可达范围”。只要这个范围覆盖到了终点，就说明可以抵达。

什么意思呢？一开始我们在位置 0，能跳的最远距离是 nums[0]，所以目前能到达的最远下标是 0 + nums[0]。

当我们走到位置 1 时，如果位置 1 本身是可达的（即 1 <= 最远可达距离），那么从位置 1 我们又可以跳 nums[1] 步，最远可达距离就可以更新为 max(当前最远距离, 1 + nums[1])。

这个过程可以一直持续下去——我们遍历数组，不断更新”能到达的最远位置”，只要发现最远位置 ≥ 最后下标，就返回 true。

如果在某个位置 i，我们发现 i 已经超出了当前能到达的最远距离，说明这个位置是”不可达”的，后面的位置自然也不可达，直接返回 false。

让我们用一个例子走一遍：

nums = [2, 3, 1, 1, 4]

  • 起点 i=0，最远距离 maxReach = 0 + 2 = 2
  • i=1，1 <= 2 可达，maxReach = max(2, 1 + 3) = 4，已经到达终点 4，返回 true

nums = [3, 2, 1, 0, 4]

  • 起点 i=0，最远距离 maxReach = 0 + 3 = 3
  • i=1，1 <= 3 可达，maxReach = max(3, 1 + 2) = 3
  • i=2，2 <= 3 可达，maxReach = max(3, 2 + 1) = 3
  • i=3，3 <= 3 可达，maxReach = max(3, 3 + 0) = 3
  • i=4，4 > 3 不可达，返回 false

完美吻合。

这个思路为什么是对的？

我们不需要知道具体的跳跃路径，只需要知道最远能到哪里。因为在遍历过程中，只要位置 i 是可达的，从 i 能跳到的所有位置 [i+1, i+nums[i]] 就都变成了可达的。我们只需要用 maxReach 来维护所有可达位置中的最大下标，而不需要逐一标记每个位置。

这就是贪心的思想——每一步都只关心”最远能到哪里”，不做多余的计算。

复杂度如何？只需要一次遍历，$O(n)$ 时间，$O(1)$ 额外空间。这才是这道题的最优解。

延伸思考

这道题让我们学到了什么？

贪心不是”拍脑袋”的策略——很多时候，贪心解法来自于对问题的重新定义。我们从”找一条路径”变成了”追踪最远可达边界”，问题立刻简化了。

一个重要的洞察是：当我们不需要路径本身，只需要判断”是否可达”时，往往可以放弃具体的路径探索，转而去维护一个范围或边界。这有点像”跳房子”游戏——我们不需要记住每次踩的是哪块砖，只需要知道自己的最远落点在哪里。

下次遇到类似问题，优先观察什么？

  1. 问自己：我真的需要知道路径细节吗？ 如果只需要判断可行性（是否可达、是否存在解），往往有比搜索更高效的方法。
  2. 能不能维护一个边界？ 比如”最远能到达哪里”、”当前能覆盖的范围”——这类问题常常可以用贪心或双指针处理。
  3. 遍历方向有讲究吗？ 这道题从左到右自然，但有些类似问题从右到左贪心更直观（比如”跳跃游戏 II”），多想一想哪个方向的信息更容易利用。

下次你遇到一个数组上的”能否到达”或”能否覆盖”的问题时，不妨先问问自己：我们能维护一个什么边界，不断地把它往前推？
