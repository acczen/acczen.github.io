无重复字符的最长子串

无重复字符的最长子串

核心思路

让我们从题目本身开始。给定一个字符串，我们要找出其中不含有重复字符的最长子串的长度。

举个例子，"abcabcbb" 中最长的无重复子串是 "abc"，长度为 3。"bbbbb" 就是 "b"，长度 1。"pwwkew" 是 "wke" 或 "kew"，长度 3。

听起来不难，不是吗？不妨先想一下，如果没有任何技巧，我们第一反应会怎么做？

第一反应：暴力枚举

最朴素的想法是：枚举所有可能的子串，对每个子串检查它是否包含重复字符，记录所有合法子串的最大长度。

具体来说：

  1. 用两层循环枚举所有起点 i 和终点 j（i <= j），得到子串 s[i..j]
  2. 写一个辅助函数 hasRepeat(s, i, j)，用 Set 判断这个子串内是否有重复字符
  3. 取所有无重复子串的最大长度

这个解法很直观，但问题也很明显——效率太低了。hasRepeat 本身是 O(k) 的，乘以两层循环，总时间复杂度来到了 O(n^3)。对于长度为 n 的字符串，这个开销是无法接受的。

即使我们对 hasRepeat 做一点优化——在枚举 j 的同时维护 Set，让检查变成 O(1) 均摊——整体依然是 O(n^2)。当 n 达到 10^5 级别时，O(n^2) 仍然不可接受。

那么，有没有办法把复杂度降到 O(n)？

关键洞察：哪些信息是可以复用的？

让我们换个角度思考。假设我们已经找到了一个以 i 开头的无重复子串，终点在 j。现在我们要把起点往右移一位，变成 i+1——难道需要重新从 i+1 开始遍历吗？

不妨想一想，子串 s[i..j] 已经保证了无重复，当我们把 i 右移到 i+1 时，s[i+1..j] 这个子串天然也是无重复的！因为我们只是砍掉了最左边的一个字符，不会引入新的重复。

也就是说，我们不需要每次移动左边界后都从零开始构建窗口。如果能”记住”当前窗口的状态，每次只需要处理右边界移动时引入的新字符，以及左边界移动时移除的旧字符，就能避免大量的重复计算。

这引出了滑动窗口（Sliding Window）的思想。

滑动窗口 + HashSet

我们可以用两个指针 left 和 right 维护一个窗口，再用一个 HashSet 记录窗口内有哪些字符：

  1. 不断右移 right，将新字符 s[right] 加入窗口
  2. 如果 s[right] 已经在 HashSet 中，说明窗口内有重复了——我们需要右移 left，并同步从 HashSet 中移除 s[left]，直到 s[right] 不再重复
  3. 每次调整后，更新最大长度

这里有一个关键点：为什么是移动 left 而不是跳过整个窗口？

假设我们有 "abca"，当 right 指向第二个 'a' 时，重复了。我们不能简单地认为”从下一个位置重新开始”——因为 "bca" 依然是一个合法的无重复子串。所以我们需要逐步收缩左边界，直到窗口内不再包含重复字符为止。

这个版本的时间复杂度是 O(2n) = O(n)——每个字符最多被 left 和 right 各访问一次。空间复杂度 O(k)，其中 k 是字符集大小（ASCII 最多 128，Unicode 场景下也不超过字符串长度）。

还能再优化吗？

进一步优化：HashMap 记录字符位置

刚才的方法中，while 循环可能一次只移动 left 一位。在某些极端情况下，比如 "abccccdef"，当遇到重复的 'c' 时，我们需要一步一步地把 left 从第一个 'c' 挪到第二个 'c' 后面。这虽然不影响总体的 O(n) 复杂度，但每一步的 while 循环确实做了些重复劳动。

不妨想：我们能不能一步到位——当发现字符重复时，直接把 left 跳到重复字符上一次出现位置的下一个位置？

这就需要我们记住每个字符最近一次出现的下标。HashMap 刚好可以做到这一点，映射关系是：字符 -> 它最近一次出现的位置。

  • 遍历字符串时，如果 s[right] 已经存在于 HashMap 中，说明遇到了重复
  • 此时，我们可以直接把 left 更新为 max(left, map.get(s[right]) + 1)——为什么要取 max？因为 HashMap 中存储的是历史所有位置，有些位置可能在当前 left 的左边，这些已经被窗口排除的旧位置不应该影响我们
  • 然后更新 map 中 s[right] 的位置为当前 right
  • 更新最大长度

这样一来，每个字符只需要处理一次，left 也不需要一步一步挪了。

这就完成了从 O(n^3) –> O(n^2) –> O(2n) –> O(n) 的优化路径。

用小例子跑一遍

让我们用手算一下 "abcabcbb"，直观感受流程：

初始：left=0，maxLen=0，map={}

right s[right] map 中已存在？ left 更新后 map 更新 窗口 长度 maxLen
0 ‘a’ 否 0 {a:0} a 1 1
1 ‘b’ 否 0 {a:0, b:1} ab 2 2
2 ‘c’ 否 0 {a:0, b:1, c:2} abc 3 3
3 ‘a’ 是，位置 0 max(0, 0+1)=1 {a:3, b:1, c:2} bca 3 3
4 ‘b’ 是，位置 1 max(1, 1+1)=2 {a:3, b:4, c:2} cab 3 3
5 ‘c’ 是，位置 2 max(2, 2+1)=3 {a:3, b:4, c:5} abc 3 3
6 ‘b’ 是，位置 4 max(3, 4+1)=5 {a:3, b:6, c:5} b 1 3
7 ‘b’ 是，位置 6 max(5, 6+1)=7 {a:3, b:7, c:5} (空) 0 3

最终结果为 3，答案正确。

关键步骤在第 6 步：left 没有跳到 map.get('b') + 1 = 5 吗？跳了，但此时 left 本身就是 3，所以 max(3, 5) = 5 将 left 更新为 5。窗口从 "abc" 收缩为只包含 "b"——这很合理，因为前一个 'b' 在位置 4，而位置 3 的 'a' 也必须被排除，才能使窗口内无重复。

延伸思考

这道题让我们学到了什么？

滑动窗口是处理子串/子数组问题最经典的工具之一。它的核心思想并不复杂：当右边界扩大时，如果当前窗口不再满足条件，我们不是从头开始，而是收缩左边界，直到条件重新满足。

与二分法不同，滑动窗口不是靠”猜中间值”来缩小搜索范围，而是利用相邻状态之间的连续性：窗口 [i+1, j] 和窗口 [i, j] 只差了一个元素，这个增量关系让我们可以 O(1) 地复用前一状态的信息。

HashMap 优化版本的巧妙之处在于，它把”逐步收缩”变成了”一步到位”——因为我们可以通过哈希表直接查询到冲突字符的历史位置，从而一次性将左边界跳到正确的地方。这是一个典型的空间换时间应用。

下次遇到类似问题优先观察什么？

如果题目涉及”子串””子数组””连续区间”等关键词，并且条件是关于”是否存在重复””是否满足某个约束”，可以优先考虑：

  1. 是否能用滑动窗口？ 判断依据：当右边界扩大的时候，左边界是否需要移动来维持某种约束？如果约束是单调的（左边界右移只会使约束更宽松或更严格），滑动窗口基本适用。
  2. 窗口内需要维护什么信息？ 频率计数、字符集合、还是位置索引？这取决于题目要求的是”是否存在”还是”具体位置”。
  3. 左边界如何移动？ 是一次一步，还是可以跳跃？如果可以跳跃，往往需要额外的数据结构（如 HashMap 记录位置）来支持。

类似的题目有很多，比如：

  • 长度最小的子数组（LeetCode 209）：同样是滑动窗口，但条件变成了”和 >= target”
  • K 个不同整数的子数组（LeetCode 992）：需要维护窗口内不同元素的个数
  • 最小覆盖子串（LeetCode 76）：滑动窗口的经典变体，需要用两个指针分别寻找可行解和优化解

掌握滑动窗口的核心在于理解单调性——为什么左指针只需要向右移动，永远不会回头。想通了这一点，滑动窗口就不再是”背模板”，而是一个自然而然的思考工具了。
