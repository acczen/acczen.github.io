有效的括号

有效的括号

核心思路

第一步：先理解问题

题目给了我们一个只包含 ( ) { } [ ] 六种字符的字符串，让我们判断它是否是”有效的括号”。

什么是有效的？让我们观察几个例子：

从这些例子中，我们总结出规则：每个左括号都必须有一个同类型的右括号与之匹配，并且括号的嵌套顺序必须正确。换句话说，后打开的括号要先关闭——这听起来像什么数据结构？

第二步：从暴力解法开始

如果我们不考虑任何技巧，最直接的方法是什么？

不妨想一下：我们可以不断地在字符串中寻找成对的 ()、[]、{}，找到一对就删掉，然后继续检查剩下的字符串。如果最后字符串被删空了，说明它是有效的。

这个思路很直观，但让我们想想它的效率。每次 replace 都要扫描整个字符串，在最坏情况下（如 ((((()))))）需要 O(n²) 的时间。能不能做得更好？

第三步：观察规律，引入栈

让我们再仔细看看括号匹配的本质：

当我们从左到右扫描字符串时，遇到的每个右括号必须和它最近的一个未匹配的左括号配对。这个”最近”很关键——它揭示了**后进先出（LIFO）**的特性。

我们来手动模拟一个例子 "{[]}"：

  1. 遇到 { — 记下来，等后面的匹配
  2. 遇到 [ — 记下来，等后面的匹配
  3. 遇到 ] — 最近记录的左括号是 [，匹配！移除这条记录
  4. 遇到 } — 最近记录的左括号是 {，匹配！移除这条记录
  5. 所有记录都移除了，有效！

反过来看 "([)]"：

  1. 遇到 ( — 记下来
  2. 遇到 [ — 记下来
  3. 遇到 ) — 最近记录的是 [，不匹配！无效

这个”记下来”和”从最近开始匹配”的过程，正是**栈（Stack）**的典型应用场景。

第四步：用栈实现

算法流程非常清晰：

  1. 创建一个空栈
  2. 遍历字符串的每个字符：
    • 如果是左括号（(, [, {），将其入栈
    • 如果是右括号（), ], }）：
      • 如果栈为空，说明没有对应的左括号，返回 false
      • 取出栈顶元素，检查是否与当前右括号匹配
      • 如果不匹配，返回 false
  3. 遍历结束后，检查栈是否为空。如果为空说明全部匹配，否则说明有未关闭的左括号

这里有一个小技巧：我们可以在遇到左括号时，将对应的右括号入栈。这样遇到右括号时，只需检查栈顶元素是否等于当前字符即可，省去了写多个配对判断的麻烦。

第五步：边界情况

  • 空字符串：题目通常认为空字符串是有效的（条件允许）
  • 奇数长度的字符串：一定无效，可以提前返回
  • 第一个字符就是右括号：栈为空，直接返回 false

复杂度分析

版本 时间复杂度 空间复杂度 说明
暴力替换 O(n²) O(n) 每次 replace 扫描 O(n)，最坏执行 n/2 次
栈（最优） O(n) O(n) 一次遍历，每个字符入栈或出栈一次

时间复杂度 O(n)：只需遍历字符串一次，每个字符入栈或出栈操作都是 O(1)。

空间复杂度 O(n)：最坏情况下（如 (((((），所有字符都是左括号，栈中需要存储 n 个元素。

延伸思考

这道题让我们学到了什么？

栈是处理”嵌套匹配”问题的利器。 凡是涉及”后出现的要先处理”、”最近匹配”这类场景，栈几乎总是第一选择。

回想一下，我们的思考路径是从”不断替换”的暴力法开始的，这个思路虽然低效，但它帮助我们牢牢抓住了问题的本质——括号必须成对消除。而正是这个”成对消除”的过程，让我们自然地想到了栈。

这也是 LeetCode 中一条重要的解题经验：先想暴力解，再想优化。 暴力解虽然通常不是最终答案，但它能帮我们理清问题结构，找到优化的突破口。

下次遇到类似问题优先观察什么？

  • 是否具有 LIFO 特性？ 检查最近的元素是否影响当前决策
  • 是否涉及嵌套结构？ HTML 标签、数学表达式求值、路径解析等
  • 能否用映射简化逻辑？ 用 Map 或条件判断减少重复代码

括号匹配的变体问题还有很多，比如在字符串中嵌入普通字符、要求计算最大有效括号长度、或者支持自定义优先级规则。但无论怎么变，栈始终是解题的核心武器。
