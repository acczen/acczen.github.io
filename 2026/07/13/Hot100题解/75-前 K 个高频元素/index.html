前 K 个高频元素

前 K 个高频元素

题目：给你一个整数数组 nums 和一个整数 k，请你返回其中出现频率前 k 高的元素。你可以按任意顺序返回答案。

例如：nums = [1,1,1,2,2,3], k = 2，输出 [1, 2]，因为 1 出现 3 次，2 出现 2 次，是频率最高的两个元素。

核心思路

第一反应：数频率，再排序

这道题的第一直觉非常清晰——我们要找”频率最高的几个数”，那自然分两步走：

  1. 先统计每个数出现的频率——用哈希表轻松搞定。
  2. 再按频率排序，取前 k 个。

统计频率的环节是免不了的，遍历整个数组是 O(n)。但排序呢？如果直接用排序算法，时间复杂度是 O(n log n)，其中 n 是数组中不同元素的个数。

对于 n 可能达到 10⁵ 甚至更大的数据来说，O(n log n) 的排序虽然不算太慢，但让我们想一个问题——我们真的需要把所有元素都排好序吗？

我们的目标仅仅是”前 k 个”，不是”全部排好序”

这是一个关键的观察点。如果我们要的是全班前 3 名的成绩，难道要把全班 50 个人的成绩都精确排好序吗？完全不需要——我们只需要知道前 3 名是谁，剩下 47 个人的具体顺序跟我们无关。

在算法中，处理这种”只关心前 k 大/前 k 小”的场景，有一个经典的数据结构——堆（Heap）。

用最小堆保留前 k 大频率

思路是这样：我们维护一个大小为 k 的最小堆（Min-Heap）。

  • 遍历每个不同的数字及其频率时，把 (频率, 数字) 这个对放进堆里。
  • 如果堆的大小超过 k，就把堆顶（当前堆中最小的频率）弹出去。
  • 遍历结束后，堆里剩下的就是频率最高的 k 个元素。

用最小堆而不是最大堆，原因不妨想一下：最小堆的堆顶是当前第 k 大的门槛，新来的频率只要比堆顶大，就替换掉堆顶。这样遍历完，堆里自然就是 top k 了。

这样，时间复杂度从 O(n log n) 降到了 O(n log k)。因为堆的大小被限制在 k，每次入堆和出堆操作都是 O(log k)，n 个元素入堆一次就是 O(n log k)。

当 k 远小于 n 时（比如 n = 10⁵, k = 10），O(n log k) 相比 O(n log n) 优势非常明显。

还能不能再快一点？O(n) 有可能吗？

让我们再深挖一步——O(n log k) 已经不错了，但有没有可能做到 O(n)？

直觉上，频率是一个整数，而且它的取值范围是有限的——一个数在数组中出现的最多次数不会超过数组长度 n。如果我们把”频率”当作下标，能不能用一种类似于桶排序的思想来做？

核心思路是这样的：

  1. 先用哈希表统计每个数字的频率。
  2. 创建一个长度为 n + 1 的数组 bucket，bucket[i] 用来存放所有频率恰好为 i 的数字。
  3. 然后从后往前遍历 bucket（频率从高到低），依次取出元素，直到取满 k 个。

之所以用 n + 1，是因为频率最大就是 n（整个数组全是同一个数），最小的频率是 1。

这个算法的时间复杂度是 O(n)——遍历数组统计频率 O(n)，遍历频率桶取结果 O(n)，没有排序，没有堆操作。

空间复杂度是 O(n)——哈希表存 n 个键值对，桶数组长度 n + 1，每个元素只在一个桶中出现一次。

两种最优解，哪种更好？

桶排序是 O(n)，堆排是 O(n log k)。看起来桶排序更优，但桶排序需要额外的 n + 1 个桶的空间，而且每个桶是一个列表，当频率分布很分散时，空间上不一定比堆更省。

在实际工程中：

  • k 很小时（比如 k = 1, 2, 3），堆的 O(n log k) 几乎等同于 O(n)，而且代码更简洁。
  • n 非常大且 k 不明显小于 n时，桶排序的 O(n) 更有优势。
  • 桶排序的另一个好处是：它不涉及元素之间的比较，纯数组访问，常数因子非常小。

两种都是面试中值得掌握的解法。

延伸思考

这道题让我们学到了什么？

“只关心前 k 个，不需要全排序”——这个思维在很多题目中都能复用。当我们遇到”top k”类问题时，第一反应先想能不能用堆（最小堆或最大堆），因为它在时间和空间上往往比全排序更优。

更进一步，这道题展示了利用数据的特性突破理论下界的巧妙思路。常规的排序下限是 O(n log n)，但当我们发现频率是整数且有上界（不超过数组长度）时，就可以用桶排序做到 O(n)。这种”信息越多，解法越优”的思想，和计数排序、基数排序等非比较排序是相通的。

另外，本题也是一个多步骤组合问题的典型：统计频率 + 选择 top k。它提醒我们，当一道题涉及多个子任务时，每个子任务都可以独立优化，最终组合出最优方案。

下次遇到类似问题优先观察什么？

  1. 问题是不是”top k”类型？ 如果是，堆是首选——不用纠结，直接往堆的方向思考。
  2. 数据的范围有没有特殊性质？ 如果频率、分数、年龄等指标是整数且有界，桶排序可能带来惊喜的 O(n) 解法。
  3. 面试中先写堆还是先写桶？ 建议先写堆，因为它的思路更通用且代码不易出错；有余力再展示桶排序的优化版本，体现出对数据特性的敏感度。
  4. 注意审题：k 是否合法？ 题目保证 k 在合理范围内，但在实际工程中要考虑 k 大于不同元素个数的情况。
  5. 结果是否有顺序要求？ 这道题不要求按频率高低输出，只要求包含正确元素即可，这给了我们更大的灵活性。
