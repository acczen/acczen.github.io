乘积最大子数组

乘积最大子数组

核心思路

从一道”老朋友”说起

在 Hot 100 中，我们之前遇到过一道题叫”最大子数组和”——给定一个整数数组，找一个连续子数组，使其和最大。那道题我们用 Kadane 算法 轻松解决：每到一个位置，要么从当前位置重新开始，要么带着之前的累积和继续往前走，取两者较大值即可。

现在，问题换了一个字：

给定一个整数数组 nums，请你找出一个连续的非空子数组，使得子数组的乘积最大。

求和变成了求积。听起来只是运算符的替换，对吧？那暴力解法自然也顺理成章：

暴力枚举： 枚举所有起点和终点，计算每个子数组的乘积，记录最大值。三重循环，或者说两重循环内层边走边乘，时间复杂度 O(n^2)。

但是且慢——乘积和求和，真的有我们想象的那么相似吗？

一个关键差异：负数与零

让我们想一个问题：

在最大子数组和中，如果一个数让我们”变差了”，我们直接丢弃它，从当前元素重新开始。这能行，因为加法是单调的——负数只会让和变小，我们不要它就是了。

但是乘法呢？

计算一下：

  • 子数组 [-2] 乘积 = -2
  • 子数组 [-2, 3] 乘积 = -6
  • 子数组 [-2, 3, -4] 乘积 = 24 —— 这才是最大的

有意思的事情发生了：前两个数 -2 和 3 乘在一起是 -6，已经比 -2 小了。如果按照求和的思路，到了 3 这个位置，我们会丢弃前面的 -2，只保留 3。那最终结果就是 3，错失了后面的 24。

乘积的”反转”效应是我们面临的核心挑战：一个绝对值很大的负数，可能在遇到另一个负数后变成最大的正数。

如何应对这种”反转”？

不妨逆向思考：如果我们同时跟踪当前能得到的最大值和最小值呢？

当一个负数到来时：

  • 最大值 * 负数 –> 变成最小值
  • 最小值 * 负数 –> 变成最大值

当一个正数到来时：

  • 最大值 * 正数 –> 更大的最大值
  • 最小值 * 正数 –> 更小（或更大）的最小值

所以，我们需要的不是只维护一个”当前最大乘积”，而是同时维护两个状态：

每到一个新元素 nums[i]，我们面临三个选择：

  1. 重新开始：只取 nums[i] 本身
  2. 延续最大：nums[i] * maxSoFar
  3. 反转最小：nums[i] * minSoFar —— 当 nums[i] 为负数且 minSoFar 为负数时，这个乘积可能成为新的最大值

于是转移方程就清晰了：

全局答案就是所有 maxEndingHere 中的最大值。

这里我们不必纠结负数到底”反转”了谁——因为三个候选都列出来了，最值自然会落到正确的那一个上。

零这个特殊角色

还有一个细节：零。

当遇到 0 时，任何乘积都会变成 0。如果我们把 0 纳入子数组，后续的乘积就永远停在 0 了。

但观察我们的转移方程：

遇到 0 之后，maxSoFar 和 minSoFar 都会被归零。下一个元素来时，它只能选择”重新开始”（取自身）。这恰好是我们想要的效果——零天然地充当了切割点，子数组不会跨越零。

整体的算法流程

综上所述，我们可以这样描述整个算法：

  1. 初始化 maxSoFar = nums[0]，minSoFar = nums[0]，result = nums[0]
  2. 从索引 1 开始遍历数组：
    • 如果当前元素为负数，交换 maxSoFar 和 minSoFar（因为负数会反转大小关系）
    • 更新 maxSoFar = max(nums[i], nums[i] * maxSoFar)
    • 更新 minSoFar = min(nums[i], nums[i] * minSoFar)
    • 更新 result = max(result, maxSoFar)
  3. 返回 result

这里引入了一个小优化：当 nums[i] 为负数时，我们先交换 maxSoFar 和 minSoFar，这样后续的更新方程就简化为两个选择（重新开始 vs 延续），不用每次写三个候选。

延伸思考

这道题让我们学到了什么？

一、减法思维与乘法思维不同。 最大子数组和的成功经验不能直接套用到乘积上。和的单调性让我们只需要记住一个值（当前最大和），但乘积的”符号反转”特性要求我们同时记住最大和最小。这是一个非常重要的思维跳跃——当问题从加法变为乘法时，状态表示的维度可能翻倍。

二、负负得正——数据结构化思维。 如果直接面试遇到这道题，很多人的第一反应是”不好处理负号”。但如果我们把问题结构化——列出所有可能的选择（延续、反转、重新开始）——答案自然浮现。不要凭直觉跳跃，用穷举法把所有可能性摆出来，然后看哪些是多余的。

三、交换技巧。 当涉及符号反转时，先交换最大最小值再统一处理，可以让代码更简洁优雅。这是一种常见的编码技巧，在涉及正负号反转的场景中经常用到。

下次遇到类似问题优先观察什么？

  • 加法和乘法的本质区别是什么？ 面对乘法题，先问自己：负数会带来什么影响？需不需要同时维护最小值和最大值？
  • 数列中有没有零？ 零会让子数组”断裂”，但同时也简化了问题——你可以把数组按零分段处理。
  • 是否能扩展状态维度？ 如果只维护一个状态不够，不要犹豫，增加状态维度。从一维到二维（最大/最小）是这类问题的经典升级路径。

最后，我们不妨记住一句话：求和靠延续，求积靠反转。理解了这个”反转”，乘积类动态规划就不再神秘了。
