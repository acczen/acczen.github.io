零钱兑换

零钱兑换

核心思路

题目初印象

LeetCode 322「零钱兑换」——给定一个整数数组 coins 表示不同面额的硬币，以及一个整数 amount 表示总金额。我们需要计算凑成总金额所需的最少硬币个数。如果没有任何一种组合能凑出总金额，返回 -1。

每种硬币的数量是无限的——这提示我们，这是一个完全背包问题。

不妨先想一个最简单的例子：coins = [1, 2, 5]，amount = 11。直观上我们会想，先尽量用大面额硬币：5 + 5 = 10，剩下 1，用 1 元硬币，一共 3 枚。那这是不是最优的呢？似乎是的。但让我们换一个例子：coins = [1, 3, 4]，amount = 6。贪心选最大面额会得到 4 + 1 + 1 = 3 枚，但实际最优解是 3 + 3 = 2 枚。贪心失效了。

那么，怎么找到真正的答案？

从暴力递归出发

当我们不知道最优解是什么的时候，有一个朴素的方法：枚举所有可能的组合。

对于总金额 amount，我们可以尝试每种硬币作为”最后一枚”：

  • 如果最后一枚是 1，那么问题变成凑出 amount - 1 的最少硬币数；
  • 如果最后一枚是 2，那么问题变成凑出 amount - 2 的最少硬币数；
  • …
  • 如果最后一枚是 c，那么问题变成凑出 amount - c 的最少硬币数。

这个解法的时间复杂度是多少？每一层递归会分叉出 len(coins) 个分支，深度最多为 amount / min(coins)，指数级的复杂度，无法接受。

但我们可以问自己一个问题：这个递归中，有多少重复计算？

以 coins = [1, 2, 5]，amount = 11 为例，f(6) 可能被 f(7) - 1、f(8) - 2、f(11) - 5 等多个路径反复计算。重叠子问题出现了——这是我们熟悉的动态规划信号。

加缓存：自顶向下记忆化

既然有大量重复子问题，我们用一个数组把 f(i) 的结果存下来，遇到直接取，不重复算。

这样，每个 f(i) 只计算一次，子问题个数是 amount + 1，每个子问题需要遍历 k = len(coins) 种面额，时间复杂度降到 O(amount * k)，空间复杂度 O(amount)。

很好，但这还不是终点。递归有栈溢出的风险，我们能不能改成迭代形式？

回顾思考路径

我们从贪心开始，发现它不能保证最优解；于是退回到暴力枚举，发现指数级不可行；观察到重叠子问题后，先用记忆化搜索优化到多项式时间；最后发现子问题之间是递推关系，自然地写出了自底向上的 DP。

每一步都是对前一步的观察和反思，没有一步是凭空产生的。

复杂度分析

版本 时间复杂度 空间复杂度 说明
暴力递归 $O(k^{amount/min(coins)})$ $O(amount)$ (递归栈) 指数级，不可行
记忆化搜索 $O(amount \times k)$ $O(amount)$ 递归栈 + memo 数组
自底向上 DP $O(amount \times k)$ $O(amount)$ 迭代无栈开销，推荐

其中 $k = len(coins)$。

空间复杂度上，由于 dp[i] 只依赖更小的子问题，但依赖的跨度不固定（取决于硬币面额），我们无法进一步压缩为一维滚动变量。不过 $O(amount)$ 的空间通常可以接受。

延伸思考

这道题让我们学到了什么？

动态规划不是一种玄学，而是一套思维方法。 它背后的逻辑链条非常清晰：

  1. 先想暴力怎么做（枚举所有组合）；
  2. 观察有没有重叠子问题（画递归树）；
  3. 有重叠子问题 → 加缓存（记忆化搜索）；
  4. 看子问题是否按顺序依赖 → 改成迭代（自底向上 DP）。

很多同学觉得 DP 难，是因为跳过了前两步，直接去背状态转移方程。当我们从暴力递归出发，一步一步推理过来，状态转移方程其实是水到渠成的。

下次遇到类似问题优先观察什么？

零钱兑换是完全背包问题的变体——每种物品（硬币）可以无限取，求凑满背包的最小数量。下次遇到这类问题，我们可以从两个维度快速定位：

  • 物品能不能重复取？ 能重复 → 完全背包；不能重复 → 0/1 背包。循环顺序会不同（完全背包内层正序，0/1 背包内层倒序）。
  • 求什么？ 求最小值 / 最大值 / 方案数？这决定了初始化和转移方程的形式。

另外，这道题还有一个值得玩味的变种：零钱兑换 II（LeetCode 518）——不求最少硬币数，而求凑成 amount 有多少种不同的组合。本质完全一样，只是转移方程从 Math.min 变成了累加。一通百通。

最后，一个小提醒：当 amount 很大（比如 $10^4$ 以上）而 coins 中有很多小面额时，$O(amount \times k)$ 可能达到百万甚至千万级别——但通常还在 LeetCode 的时间限制内。如果实在需要优化，可以对硬币排序后用 BFS 或分支限界剪枝，但那是另一个故事了，留待我们以后探讨。
