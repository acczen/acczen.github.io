最长连续序列

最长连续序列

难度：MEDIUM | 题号：128 | 英文名：Longest Consecutive Sequence

核心思路

第一反应：暴力法

题目给了我们一个未排序的整数数组，让我们找出最长连续序列的长度。

不妨想一下，如果没有任何技巧，我们第一反应会怎么做？

最直接的想法是：对于数组中的每一个元素 x，我们逐个检查 x+1、x+2、x+3……是否也在数组中，直到找不到下一个数为止，然后统计这段连续序列的长度。

这个做法的问题在哪里呢？"数组中包含" 这个操作本身，如果在原数组里线性查找，那么每个 while 循环都要 O(n) 的时间，最坏情况下我们会对每个元素都向后查找一大段，总复杂度会达到 O(n³)。即使我们用一些优化手段，这依然是一个不可接受的方案。

排序之后呢？

暴力法太慢了，让我们换个角度想一想。

连续序列有一个天然的性质：有序。如果我们先把数组排好序，那么连续的数值就会乖乖地排在一起，我们只需要从左到右扫描一遍，统计相邻的递增段就行了。

这样只需要 O(n log n) 的排序时间加上 O(n) 的扫描时间，比暴力法好多了。

但是！题目明确要求了 时间复杂度 O(n)，排序法虽然简单直观，却不满足题目的硬性要求。

那么问题来了：有没有办法在不排序的情况下，也能快速判断一个数的”下一个”是否存在？

哈希表带来转机

我们需要一种数据结构，可以 O(1) 地判断一个数是否在数组中——这正是哈希表的拿手好戏。

把数组中的所有元素放入一个 HashSet，然后我们就可以在 O(1) 时间内回答”数组中是否包含某个数”这个问题了。

但这里还有一个隐藏的陷阱：想一想，如果数组是 [1, 2, 3, 4]，当我们遍历到 1 的时候，会查到 2, 3, 4，得到长度 4；然后遍历到 2 的时候，又会查到 3, 4，得到长度 3……我们做了大量重复的计算。

有没有办法避免这种重复？

关键洞察是：我们只需要从连续序列的”起点”开始统计。换句话说，如果一个数 x 在数组中存在前驱 x-1，那么 x 一定不是某个连续序列的开头，我们就不必以它为起点向后统计。

还是 [1, 2, 3, 4] 这个例子：

  • 遍历到 1：检查 0 不在 set 中 → 1 是起点 → 向后查到 2, 3, 4 → 长度 4
  • 遍历到 2：检查 1 在 set 中 → 2 不是起点 → 跳过
  • 遍历到 3：检查 2 在 set 中 → 3 不是起点 → 跳过
  • 遍历到 4：检查 3 在 set 中 → 4 不是起点 → 跳过

这样一来，每个数最多只会被访问两次（一次作为起点建入 set，一次作为某个起点的后续被查到），总复杂度是 O(n)。

最优解核心思想总结

  1. 将所有数字存入 HashSet，实现 O(1) 的查找
  2. 遍历每个数字，只对没有前驱（即 num-1 不在 set 中）的数字开始统计连续序列的长度
  3. 不断向后查找 num+1、num+2……直到找不到为止，更新最长长度

整个过程只需要遍历数组常数次，时间复杂度 O(n)，空间复杂度 O(n)。

复杂度分析

版本 时间复杂度 空间复杂度 说明
暴力法 O(n³) O(1) 对每个元素线性查找其后继，每次查找需 O(n)，最多查找 O(n) 层
排序法 O(n log n) O(1) 或 O(n) 排序后扫描，排序本身决定了时间复杂度的下限
哈希表最优解 O(n) O(n) 每个元素至多被访问两次，一次存入 set，一次作为起点或后继被查

延伸思考

这道题让我们学到了什么？

哈希表不仅仅是用来做查找的，它更是”降维打击”的利器。 当我们需要频繁地判断”某个元素是否存在于集合中”时，哈希表可以将 O(n) 的查找变成 O(1)，从而让整个算法的时间复杂度跃升一个量级。

更深入一层，这道题教会了我们如何识别”冗余计算”。在暴力枚举中，以每个元素为起点的统计覆盖了大量重复的区间。通过 num-1 存在性判断来过滤非起点，本质上是一种剪枝策略——只做那些真正有意义的计算。

下次遇到类似问题优先观察什么？

  • “是否存在”类的查询：如果一个问题中反复出现”判断某个值是否在集合中”的操作，第一反应就考虑用哈希表来加速。
  • 连续/相邻关系：当问题涉及连续数值、相邻元素或某种”链条”关系时，可以先思考是否能用 HashSet 来构建 O(1) 的连接判断。
  • 起点过滤：如果链条/序列可以从多个位置开始统计，找到唯一的最小起点（或等价条件）来避免重复统计，往往是将复杂度从 O(n²) 降到 O(n) 的关键。

举一反三：类似的思路还可以用在”最长连续递增子序列”、图的连通分量统计等场景中——核心都是找到那个”不被包含的更小值”作为起点。
