随机链表的复制

随机链表的复制

核心思路

从最直观的想法开始

题目要求我们”深拷贝”一个链表。普通的单链表复制很简单——遍历一次原链表，每遇到一个节点就创建新节点，串联起来即可。但这道题的每个节点多了一个 random 指针：

random 可以指向链表中的任意节点，也可以指向 null。

如果我们先按 next 顺序复制一条”主链”，然后回头处理 random 指针，会遇到一个问题：当我们遍历原链表的某个节点时，知道它的 random 指向谁，但我们怎么知道新链表中对应的那个节点在哪里？

新节点是我们逐个创建的，它们还没有被”索引”过。所以一上来就迎面碰上一个问题：

问题：已知原链表中节点 A 的 random 指向原链表中的节点 B，如何快速找到新链表中对应的节点 B’？

思路一：用 HashMap 建立映射关系

最直接的想法是：在创建新节点的同时，用一个字典记录原节点到新节点的对应关系。

这样一来，第一次遍历只复制节点值并建立映射，第二次遍历利用映射来连接 next 和 random。

这个解法正确吗？是的。时间复杂度 O(N)，空间复杂度 O(N)。

但我们不禁要问：

问题：能否省掉这个 HashMap？

如果能省掉，就能把空间复杂度降到 O(1)，这在面试场景中是一个很有分量的优化点。

思路二：原地交织法（O(1) 额外空间）

我们观察一下 HashMap 的核心作用：它建立了一条从”原节点地址”到”新节点地址”的快速通道。如果没有这个通道，当我们在原链表中看到 node.random = target 时，我们不知道新链表中哪个节点对应 target。

如果换一个思路呢？把新节点直接”织入”原链表中，让它们的物理位置建立一一对应的关系。

具体来说：

让我们具体验证一下这个逻辑：

假设原链表：节点1(val=7) -> 节点2(val=13) -> 节点3(val=11)
其中 节点2.random = 节点1

第一步插入复制节点后：
节点1 -> 节点1’ -> 节点2 -> 节点2’ -> 节点3 -> 节点3’ -> null

第二步处理 random：

  • 节点1 的 random 为 null，所以 节点1’.random = null
  • 节点2 的 random 指向 节点1，所以 节点2’.random = 节点2.random.next = 节点1.next = 节点1’
    这正是我们希望的结果！
  • 节点3 的 random 同理

第三步分离，我们只需要交错拆分即可。

核心洞察：当复制节点恰好位于原节点的 next 位置时，”原节点到复制节点”的映射就隐含在链表的 next 指针中，无需额外空间。

边界条件

几个值得注意的点：

  • 如果原链表为 null，直接返回 null
  • 如果某个节点的 random 为 null，则复制节点的 random 也设为 null（在第二步中需要特判）
  • 第三步分离时要注意不要破坏原链表的结构（虽然题目没有强制要求，但好习惯是恢复原链表）

延伸思考

这道题让我们学到了什么？

将隐式映射转化为物理结构。 当我们说”我要记住 A 对应 A’”时，最自然的想法是用哈希表。但哈希表不是唯一的记忆方式——我们也可以把这种对应关系”写入”链表本身的拓扑结构中。将复制节点放在原节点的 next 位置，就是一种物理化的”记忆”。

这种思想在很多题目中都有体现：例如，二叉树的”填充每个节点的下一个右侧节点指针”也可以用类似的层序或递归思路。但”在原结构中嵌入新信息，处理完再恢复”这个模式，其实是一种更通用的解题手法。

下次遇到类似问题优先观察什么？

  1. 数据之间的关系能否通过物理位置表达？ 如果题目要求”保持某种对应关系”，先想想这个关系是否可以隐含在索引、指针或位置中，而非必须显式存储。
  2. 是否能分阶段处理？ 这道题分三步走（复制插入 → 设置 random → 分离）。有些复杂操作拆成多个遍历后，每一步都变得很简单。不要试图在一个循环里做完所有事。
  3. 注意”记住”和”查找”的代价。 HashMap 能让我们 O(1) 地找到任意对应关系，但代价是 O(N) 的额外空间。在空间敏感的场景中（比如嵌入式设备或面试追问），原地算法往往是加分的思考方向。

最后，回到题目本身：随机链表的复制之所以被列为”中等”题，不是因为它涉及的编程技巧有多难，而是因为它考察的是——你能不能从一个看似需要额外存储空间的问题中，找到那个令人眼前一亮的”无额外空间”解法。不妨记住这个”三步交织法”，它在链表相关题目中是一个经典且优雅的模式。
