二叉树的右视图

二叉树的右视图

核心思路

题目：给定一棵二叉树，站在它的右侧，从上到下能看到哪些节点？

不妨想一下，”站在右侧看”这件事，本质上是在问：每一层最右边的那个节点是谁。这个直觉直接引导我们走向第一种解法。

从暴力开始：层序遍历，取每层最后一个

最容易想到的方案是什么？让我们把二叉树按层拆开。如果我们做一次层序遍历（BFS），把每一层的节点都列出来，那么每层的最后一个节点自然就是从右侧看到的那个节点。

这个思路直接、正确，而且时间复杂度已经是 O(n)——每个节点恰好访问一次。那么问题来了：我们能做得更好吗？

严格来说，在时间上无法超越 O(n)，因为我们必须至少看一眼每个节点才能知道它是不是某层的最右节点。所以 BFS 解法本身就是最优时间复杂度的。但是，空间上还有优化的空间吗？

BFS 层序遍历需要维护一个队列，队列中最多同时存放一层的节点。在最坏情况下（满二叉树的最底层），队列大小可以达到 n/2，也就是 O(n) 的空间。对于一棵很宽的树，这可能会消耗不少内存。

那么，有没有空间效率更高的做法呢？

自问：不用队列，只用递归能不能做？

如果我们换一个角度：DFS（深度优先搜索）是沿着一条路径走到黑，再回溯。它不会像 BFS 那样同时持有大量节点——递归栈的深度等于树的高度 h，最坏情况 O(n)（链状树），但在平衡树中只有 O(log n)。

关键在于，DFS 能否也识别出”每层最右边的节点”？

让我们想一下：如果我们按照根节点 -> 右子树 -> 左子树的顺序遍历（也就是”右优先的前序遍历”），会发生什么？

看到规律了吗？对于每一层，当我们第一次到达该层深度时，遇到的节点正好就是从右侧看到的节点。我们只需要在递归时维护一个 depth 变量，当 depth == result.size() 时，说明这是该层第一个被访问到的节点，加入结果即可。

这样我们就不需要显式的队列了，只用递归栈。

两种方案对比

维度 BFS 层序取末 DFS 右优先前序
遍历顺序 按层从左到右 根 → 右 → 左
核心数据结构 队列 递归栈
如何确定右视图节点 取每层最后一个节点 每层第一个被访问的节点
空间复杂度 O(n)（队列最宽处） O(h)（递归深度，h 为树高）

两种方法都是 O(n) 时间，在平衡树上 DFS 的空间更优，而在链状树上二者持平。

版本 时间复杂度 空间复杂度 说明

两种算法的时间复杂度均为 O(n)，因为每个节点恰好被访问一次。空间上的差异在于：BFS 的空间消耗取决于树的宽度（最宽一层的节点数），而 DFS 的空间消耗取决于树的高度（递归栈的深度）。

延伸思考

这道题让我们学到了什么？

“站在右侧看”这个问题，本质上是对层内节点的筛选条件做了一个巧妙的转化。BFS 的视角是”我已知每一层所有节点，从中选最后一个”；DFS 的视角是”我改变遍历顺序，让每层第一个被访问的天然就是答案”。同一个需求，从不同的遍历顺序切入，得到了差异巨大的代码风格——DFS 解法只有寥寥几行，核心逻辑只有一个 if 判断。

下次遇到类似问题优先观察什么？

如果题目要求我们”从某个特定角度观察树”（左视图、俯视图、垂直遍历等），不妨先问自己两个问题：

  1. 能不能通过调整遍历顺序来简化问题？ 右视图用”右优先前序”，左视图用”左优先前序”——遍历顺序本身就是武器。
  2. 用 BFS 还是 DFS？ BFS 天然按层处理，适合”每层取某些节点”的题目；DFS 空间更省，适合树比较高但不太宽的场合。

树的问题，归根结底是对遍历方式的灵活运用。掌握了 BFS 和 DFS 各自的性格特点，就能在它们之间自如切换，为每一道题找到最趁手的工具。
