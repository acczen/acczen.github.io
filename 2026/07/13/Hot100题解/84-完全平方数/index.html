完全平方数

完全平方数

核心思路

从题目本身出发

给定正整数 n，找到若干个完全平方数（比如 1, 4, 9, 16, ...）使得它们的和等于 n。需要让组成和的完全平方数的个数最少。

题目本身并不复杂——给定一个数，用最少的完全平方数把它凑出来。

第一直觉：贪心能行吗？

让我们先想一个最直觉的策略：每次选尽可能大的完全平方数。

比如 n = 12，小于 12 的最大完全平方数是 9，剩下 3；然后 3 只能拆成 1+1+1。于是得到 9 + 1 + 1 + 1，用了 4 个数。

但最优解是 4 + 4 + 4，只用 3 个数。贪心在这里翻车了——局部最优并不能引导我们到达全局最优。

暴力递归：枚举所有可能

既然贪心不行，那我们就老老实实枚举：对于当前的 n，尝试减去每一个不超过它的完全平方数 j²，然后递归求解剩下的数，最后取最小值。

这里的 k 满足 k² <= n。

但——我们还能不能更快？

换个角度看问题：最短路模型

让我们换一种视角。

把 0 到 n 的每个整数看作一个节点。从节点 i，我们可以减去一个完全平方数 j² 到达节点 i - j²。边的权重都是 1。那么问题就转化成了：从节点 n 到节点 0 的最短路径。

最短路 → BFS（因为边权为 1）。

为什么不直接用 DP 而要用 BFS？因为 BFS 具有早停特性——一旦第一次到达 0，就找到了最短路径，可以立即返回。理论上，对于某些输入，BFS 可以比 DP 更快，因为它不需要计算所有状态。

数学定理：一锤定音

有没有可能我们根本不需要搜索？

拉格朗日四平方和定理（Lagrange’s Four-Square Theorem） 告诉我们：每个正整数都可以表示为至多 4 个完全平方数的和。

也就是说，答案只可能是 1、2、3、4 之一。

那么问题就变成了：如何判断答案是 1、2、3 还是 4？

判断是否为 1：n 本身是完全平方数，则 answer = 1。

判断是否为 4：根据四平方和定理的推论，如果一个数形如 4^a × (8b + 7)，则它必须用 4 个平方数表示。所以检查：不断除以 4，直到不能被 4 整除，然后检查 n % 8 == 7。

判断是否为 2：检查是否存在两个完全平方数之和等于 n。即枚举所有 j² 使得 j² < n，检查 n - j² 是否为完全平方数。

其余情况：答案为 3。

这个解法将时间复杂度降到了 O(√n)，空间复杂度降到了 O(1)。相当优雅。

总结思考路径

从指数到多项式再到常数级别的定理判断——这条路本身，比答案更有价值。

延伸思考

这道题让我们学到了什么？

1. 从暴力到最优是一步一步推导出来的。

没有哪个最优解是凭空出现的。贪心不行就暴力，暴力太慢就发现重叠子问题，引入 DP；DP 遍历所有状态觉得慢，就用 BFS 早停；BFS 还不够，最终借助数论定理直接回答。每一步的驱动力都是前一个方案的局限性——这种「发现问题 → 寻找改进」的节奏，才是解题中最值得反复练习的能力。

2. 同一个问题，可以套进不同的框架去理解。

当我们将「数字拆成平方数之和」看作「图上最短路径」时，BFS 自然就出现了。换一个框架，就意味着换一套可用的工具。如果我们只把问题锁死在「递推」这个框架里，永远不会想到 BFS。

3. 数学知识可以带来降维打击。

四平方和定理直接告诉我们答案在 1~4 之间，把问题从一个搜索问题变成了一个判断问题。类似的例子还有很多：判断质数可以用 Miller-Rabin，求最大公约数有辗转相除法——数论中那些看似没有实用价值的定理，恰恰是某些算法问题的终解。

下次遇到类似问题优先观察什么？

  • 数据范围：n 是 10⁴ 还是 10⁹？这决定了 O(n√n) 够不够用，还是需要 O(√n) 甚至 O(log n)。
  • 问题结构：是否具有「最优子结构」和「重叠子问题」？如果是，考虑 DP。
  • 能否转化为图：状态之间的转移关系是不是均匀的、无环的、边权相等的？如果是，考虑 BFS 双向 BFS。
  • 有没有现成的数学结论：这道题是平方数，那「立方数」「阶乘」「质数」相关的题是否有对应的定理？养成查阅数学背景的习惯，有时可以省去大量计算。

归根结底，算法竞赛或面试中遇到的绝大多数题目，考察的都不是「你是否见过这个结论」，而是「你是否能沿着一条清晰的思考路径，从最基本的方法一直推进到足够好的解法」。这比背模板重要得多。
