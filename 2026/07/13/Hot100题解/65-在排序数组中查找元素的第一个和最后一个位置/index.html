在排序数组中查找元素的第一个和最后一个位置

在排序数组中查找元素的第一个和最后一个位置

核心思路

第一步：最直观的解法 — 线性扫描

题目要求我们在一个非递减排序的数组中，找到某个目标值的第一个和最后一个位置。

最直接的想法是：从左到右遍历数组，找到第一个等于 target 的元素就记录左边界；再从右到左遍历一遍，找到第一个等于 target 的元素记录右边界。

这个方法非常朴素，两趟遍历，时间复杂度是 O(n)。可当我们看到题目描述里写着**「你必须设计并实现时间复杂度为 O(log n) 的算法」**时，心里就该有数了——这道题考察的不是线性扫描。

那么问题来了：一看到 O(log n)，我们自然想到什么？

二分查找。

但二分查找通常只找一个目标值，怎么用它去找”第一个”和”最后一个”呢？

第二步：用二分查找找边界

给定一个有序数组，二分查找每次都能将搜索范围缩小一半。关键在于：当 nums[mid] == target 时，我们不急着返回，而是继续向一侧收缩。

  • 找第一个位置：当 nums[mid] == target 时，把右边界向左移动（right = mid - 1），继续在左半部分搜索，直到区间耗尽。最后 left 指向的就是第一个目标值。
  • 找最后一个位置：当 nums[mid] == target 时，把左边界向右移动（left = mid + 1），继续在右半部分搜索。最后 right 指向的就是最后一个目标值。

让我们用一个具体的例子来感受一下这个过程。

假设 nums = [1, 2, 3, 3, 3, 4, 5]，target = 3。

查找第一个位置：

步骤 left right mid nums[mid] 动作
1 0 6 3 3 == target 收缩右边界：right = mid - 1 = 2
2 0 2 1 2 < target 收缩左边界：left = mid + 1 = 2
3 2 2 2 3 == target 收缩右边界：right = mid - 1 = 1
4 2 1 — — left > right，退出，first = left = 2

查找最后一个位置：

步骤 left right mid nums[mid] 动作
1 0 6 3 3 == target 收缩左边界：left = mid + 1 = 4
2 4 6 5 4 > target 收缩右边界：right = mid - 1 = 4
3 4 4 4 3 == target 收缩左边界：left = mid + 1 = 5
4 5 4 — — left > right，退出，last = right = 4

结果：[2, 4]，完全正确。

第三步：边界检查与优雅统一

这里有一个细节值得注意：如果 target 根本不存在于数组中怎么办？

在上面的找第一个位置的流程结束后，left 可能越界（等于数组长度），或者 nums[left] != target。此时直接返回 [-1, -1] 即可。只有在确认第一个位置存在之后，才需要去找最后一个位置。

这样我们就得到了一个两趟二分查找、每次 O(log n) 的解法，整体 O(log n)，符合题目要求。

延伸思考

这道题让我们学到了什么？

  1. 二分查找的灵活运用。二分查找绝不只是在有序数组中找一个确定的值。当相等时如何处理边界，决定了你是找到第一个、最后一个，还是任意一个。掌握了这种变体，你就掌握了二分查找的精髓。

  2. “区间”即”两个边界”。在一个有序数组中，所有相等的目标值构成一个连续的区间。找这个区间，就是分别找它的左右边界。这种”分而治之”——把一个问题拆成两个相同的子问题——是算法设计中非常经典的思路。

  3. 边界检查的重要性。很多时候算法的主体逻辑并不复杂，但边界条件——空数组、没找到、数组越界——往往最容易出 Bug。在写完核心逻辑后，花一分钟思考所有边界情况，是专业程序员的好习惯。

下次遇到类似问题，优先观察什么？

  • 输入是不是有序的？有序 + 查找 ≈ 二分查找（或者双指针）。
  • 是不是要找边界？”第一个大于等于”、”最后一个小于等于”、”第一个出现的位置”、”最后一个出现的位置”——这些本质上都是二分查找的变体。
  • 能不能用标准库？Python 的 bisect、Java 的 Arrays.binarySearch、C++ 的 lower_bound / upper_bound 都能帮我们减少重复造轮子。

下次当你在一道题中看到 “sorted” 和 “O(log n)” 这两个关键词时，不妨先想想：问题能不能拆成两个二分查找？
