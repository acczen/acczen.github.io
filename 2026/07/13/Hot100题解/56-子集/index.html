子集

子集

核心思路

从”选还是不选”开始

先看题目：给定 nums = [1, 2, 3]，输出它的所有子集。

答案长这样：

观察一下，一共有 8 个子集。为什么是 8？

不妨想一下：对于数组中的每个元素，它只有两种命运——在子集中，或者不在。3 个元素，每个 2 种选择，总组合数就是 $$2^3 = 8$$。

推而广之，对于长度为 n 的数组，子集总数一定是 $$2^n$$ 个。

那么问题来了：如何系统性地枚举出这 $$2^n$$ 种组合，不重复、不遗漏？

沿着这个问题往下走，我们会发现三条不同的路，而它们的内核惊人地一致。

解法一：回溯法 —— 在决策树上散步

把枚举过程想象成一棵二叉树：

每个节点就是一个子集。从根（空集）出发，走到每个节点时记录路径，正好覆盖所有子集。

这是我们最常见的递归写法：

  1. 用一个 path 列表记录当前正在构造的子集
  2. 每次进入递归，先把当前 path 的快照加入结果集
  3. 从当前位置 start 开始，依次选择每个元素加入 path
  4. 递归到下一层（start + 1）
  5. 撤销刚才的选择，尝试下一个元素

为什么一进入递归就记录结果？

因为子集不要求”选满”，空集、只选一个、选两个……每一步的中间状态都是合法的子集。所以我们在每个节点都收集一次。

为什么从 start 开始而不是从 0 开始？

这是为了避免重复。当我们已经决定跳过前面的元素，就不应该再回头选它们——否则 [1, 2] 和 [2, 1] 都会被生成。用 start 控制遍历的起点，保证了子集中元素的相对顺序和原数组一致，从而天然消除了重复。

回溯法的代码非常优雅，只有十几行，却把 $$2^n$$ 条路径全部走了一遍。

解法二：迭代法 —— 从空集开始，逐层扩张

换一个视角：如果你已经知道前 k 个元素的所有子集，当第 k+1 个元素到来时，会发生什么？

不妨想一下：新的元素可以和现有的每一个子集组合，形成一批新的子集。

每一步都像一个同心圆在不断扩张。

为什么这个方法是正确的？

数学归纳法：假设前 k 个元素已经生成了所有子集（共 $$2^k$$ 个）。第 k+1 个元素加入后，新的子集 = 旧的 $$2^k$$ 个子集 + 旧子集各加上新元素得到的 $$2^k$$ 个新子集 = $$2^{k+1}$$ 个，恰好覆盖。

和回溯法有什么联系？

迭代法其实是回溯法”展开”后的样子。回溯法通过递归隐式地维护了”当前元素前已有的子集”，迭代法则显式地用列表保存它们。

解法三：位运算法 —— 用二进制写答案

既然每个元素只有”选”和”不选”两种状态，那为什么不用二进制位来表示呢？

假设用 0 表示”不选”，1 表示”选”：

从 0 遍历到 $$2^n - 1$$，每个数字对应一个子集。检查每个数字的第 j 位是否为 1，就能确定 nums[j] 是否在当前子集中。

这看起来有点炫技，但它的思想很有价值——将组合问题映射到二进制运算上，用数学的方式理解枚举。

三种解法本质

解法 核心思想 遍历顺序
回溯法 深度优先搜索决策树 按元素顺序深度探索
迭代法 逐层扩展，每次”复制+添加” 广度推进
位运算法 用二进制编码表示选择状态 按数字顺序线性遍历

三种方法的时间复杂度都是 $$O(n \times 2^n)$$，空间复杂度各有千秋。回溯法最灵活，迭代法最直观，位运算法最省辅助空间。在实际面试中，回溯法是最常被考察和扩展的写法。

版本 时间复杂度 空间复杂度 说明
回溯法 $$O(n \times 2^n)$$ $$O(n)$$ 递归栈深度为 n，每条路径复制到结果需 O(n)
迭代法 $$O(n \times 2^n)$$ $$O(n)$$ 每轮遍历已有子集并复制添加新元素
位运算法 $$O(n \times 2^n)$$ $$O(1)$$ 不计输出空间，仅需常数辅助空间

时间复杂度详解：无论哪种方法，都要生成 $$2^n$$ 个子集，每个子集平均长度约为 $$n/2$$，将子集加入结果时需要进行一次拷贝，因此总时间复杂度为 $$O(n \times 2^n)$$。对于 n ≤ 10（题目约束），这个复杂度完全可行。

延伸思考

这道题让我们学到了什么？

  • “选或不选”是组合类问题的基本原子。子集、组合、分割……几乎所有回溯问题都可以归结为”对每个元素做决策”。
  • 从暴力到结构化。最原始的暴力是手动枚举，而回溯法给出了一个优雅的框架——用递归和剪枝系统性地遍历解空间。
  • 一题多解。同一个问题可以有回溯、迭代、位运算三种完全不同的视角，它们看似不同，本质上都在枚举 $$2^n$$ 种选择状态。这种”殊途同归”的感觉正是算法的魅力。
  • space-time tradeoff 的体现。位运算法省去了递归栈的开销，但思路更抽象；回溯法代码最自然，但有递归深度限制的风险。

下次遇到类似问题优先观察什么？

  • 是不是”从集合中选出若干元素”的问题？如果是，优先考虑回溯法。
  • 元素是否唯一？如果不唯一，需要先排序再加剪枝去重（参考 LeetCode 90. 子集 II）。
  • 数据规模多大？如果 n ≤ 20 左右，$$O(2^n)$$ 的复杂度通常可接受；再大就需要更高效的策略（如双向搜索）。
  • 是否要求顺序？如果不要求，用 start 控制遍历顺序来去重是经典套路。

掌握”子集”这道题，其实就掌握了回溯法最核心的骨架。接下来无论是组合总和、全排列还是分割回文串，你会发现它们的代码结构和这里几乎一模一样——只不过决策规则稍作调整罢了。
