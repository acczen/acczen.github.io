删除链表的倒数第 N 个结点

删除链表的倒数第 N 个结点

核心思路

最朴素的想法：先数长度，再删

让我们先放下所有的技巧，从最直觉的角度出发。给你一个单链表，要删掉倒数第 N 个结点，你第一反应是什么？

大多数人会这样想：我要是知道链表的总长度就好了。知道了总长度 L，倒数第 N 个就是从正着数的第 L - N + 1 个。这样问题就转化成了”删除正数第 k 个结点”——这是我们熟悉的问题。

按照这个思路，我们需要两趟遍历：

  • 第一趟：从头走到尾，数出链表的总长度 L。
  • 第二趟：从头走到第 L - N 个结点（即待删结点的前驱），跳过第 L - N + 1 个结点。

这么做当然是对的，但一个很自然的疑问就浮上来了：

我们能不能只遍历一遍就完成删除？

链表题里，”遍历两遍”往往意味着还有优化空间。让我们想一想，有什么办法能在一次扫描中就知道”这个结点是倒数第 N 个”？

快慢指针：让一个指针提前出发

不妨换个角度思考。倒数第 N 个结点，本质上是说从它到链表末尾恰好还有 N - 1 个结点。

那么，如果我们先让一个指针从头出发，向前走 N 步；再让另一个指针从头出发，两个指针以相同的速度同步前进——当第一个指针走到末尾（null）时，第二个指针所在的位置，是不是正好就是倒数第 N 个结点？

让我们验证一下：

  • 第一个指针先走 N 步，此时它领先第二个指针 N 个结点。
  • 然后两个指针一起每次走一步。当第一个指针走到 null 时，它又走了 L - N 步（因为链表总长 L，它已经走了 N 步，还剩 L - N 步到末尾）。
  • 第二个指针同样走了 L - N 步，从起点出发，它现在的位置就是正数第 L - N + 1 个结点，也就是倒数第 N 个结点。

完美！但我们仔细一想，删除一个结点需要找到它的前驱。上面的方法直接定位到了要删除的结点本身，而不是它的前驱。如果我们想删除它，还得额外记录前驱信息。

那能不能让第二个指针正好停在待删结点的前驱呢？

调整一下：让第一个指针先走 N + 1 步，然后两个指针再一起走。当第一个指针走到 null 时，第二个指针正好停在倒数第 N + 1 个结点——也就是待删结点的前驱。

这里还有一个边界情况需要我们小心：如果要删除的是头结点呢？头结点没有前驱。

一种很优雅的处理方式是引入一个哑结点（dummy node），让它指向头结点。这样一来，头结点也有了前驱，所有情况就统一了。

完整流程

让我们把思路整理成清晰的步骤：

  1. 创建一个哑结点 dummy，令 dummy.next = head。
  2. 初始化两个指针 first 和 second，都指向 dummy。
  3. 让 first 先走 N + 1 步。
  4. 然后 first 和 second 同步前进，直到 first 为 null。
  5. 此时 second 指向待删结点的前驱。执行 second.next = second.next.next 删除目标结点。
  6. 返回 dummy.next（新的头结点）。

为什么要走 N + 1 步而不是 N 步？因为 dummy 在 head 之前，从 dummy 到 head 是 1 步，要让 second 最终停在待删结点的前驱，first 需要多走 1 步。

小例子：链表 [1,2,3,4,5]，删除倒数第 2 个

我们用手动模拟来加深理解：

  • dummy -> 1 -> 2 -> 3 -> 4 -> 5 -> null
  • first 和 second 都指向 dummy
  • first 走 3 步（N+1=3）：指向 3
  • first 和 second 同步走：
    • first 到 4，second 到 1
    • first 到 5，second 到 2
    • first 到 null，second 到 3
  • second 在 3，它后面是 4（待删结点）。执行 second.next = second.next.next，链表变为：1 -> 2 -> 3 -> 5
  • 返回 dummy.next，即 1。完成！

复杂度分析

版本 时间复杂度 空间复杂度 说明
两趟遍历（暴力） $O(L)$ $O(1)$ 先数长度，再删除。L 为链表长度
一趟遍历（快慢指针） $O(L)$ $O(1)$ 快指针先走 N+1 步，再同步前进

两种方法的时间复杂度都是 $O(L)$，但快慢指针只需一次遍历，在链表较大时实际执行更快，且代码更简洁优雅。空间上两者都是 $O(1)$，只使用了常数级别的额外指针。

延伸思考

这道题让我们学到了什么？

哑结点（dummy node）是一个极其常用的技巧。当链表的头结点可能被删除时，哑结点可以让我们免于单独处理头结点删除的边界情况，代码更加统一和简洁。

快慢指针的”偏移量”思想也值得记在脑海里。”快指针先走 K 步”这个操作的本质，就是让快慢指针之间保持一个固定的间隔。利用这个间隔，我们可以定位到链表中的任意相对位置——不只是倒数第 N 个，还可以找链表中点（快指针走两步、慢指针走一步）、判断是否有环等等。

下次遇到类似问题优先观察什么？

遇到链表操作类问题，我们不妨先问自己三个问题：

  1. 头结点会不会变？ 如果头结点可能被删除、替换或修改，优先考虑哑结点。
  2. 能不能一次遍历解决？ 如果一次遍历不够，想想是否可以用两个指针来”记住”某些位置信息。
  3. 边界条件是什么？ 链表为空、只有一个结点、删除头结点、删除尾结点——这些情况是否都能被代码优雅地覆盖？

链表的问题往往代码不长，但边界陷阱多。把这些思考方式变成习惯，很多”中等”难度的链表题也就不那么”中等”了。
