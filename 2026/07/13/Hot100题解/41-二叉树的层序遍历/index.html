二叉树的层序遍历

二叉树的层序遍历

核心思路

第一反应：怎么把一棵树「按层」拆开？

题目要求我们返回一个二维数组，每一层对应一个子数组。举个例子：

输出是 [[3], [9,20], [15,7]]。

如果题目改成「前序遍历」，我们很快能写出递归：根 → 左 → 右。但现在是「层序遍历」，递归直觉好像派不上用场了——因为递归是一头扎到底再回溯的，天然就是深度优先。

那换个角度：如果我们手里有一棵树的全部节点，能不能像排队一样，一层一层地把它扫出来？

不妨想一下——排队的时候，先来的人先出去，后来的人站在队尾。我们访问完某个节点之后，它的左孩子和右孩子就应该排到队伍的后面去，等前面的人（同一层甚至上一层的节点）都处理完了，再轮到它们。这不就是先进先出吗？先进先出对应什么数据结构？队列。

用队列模拟层序：直观版

得到了访问顺序：3 → 9 → 20 → 15 → 7，刚好是按层从左到右。

可是这里有个问题——我们怎么知道 3 是第一层，9 和 20 是第二层，15 和 7 是第三层呢？直接顺序弹出，我们得到的只是一个一维数组 [3, 9, 20, 15, 7]，分不清边界。

关键一问：如何确定每一层的边界？

有没有更优雅的方式？

再想一下——当我们开始处理某一层的时候，队列里恰好装着这一层的全部节点。如果我们在一轮循环开始前，先记下队列的当前长度 size，然后只处理 size 个节点，那么这 size 个节点就是一整层。处理完这 size 个节点之后，队列里剩下的就是下一层的全部节点了。

第二种思路：DFS 也能做层序遍历？

等一下——如果我们真的想用递归呢？虽然递归天然是深度优先，但我们可以在递归时携带一个「层号」参数，把节点值塞到对应层号的数组里。

这样递归遍历的顺序虽然是深度优先（先走完左子树），但每个节点都会被分配到正确的层号上，最终结果自然是按层组织的。

对比一下两种思路：

方法 核心思想 优点 缺点
BFS + 队列 用 size 记录每层宽度 直观、标准 需要手动管理队列
DFS + 深度 递归时传递层号 代码短、无需额外数据结构 递归栈有深度限制

两种都可以 AC，面试时一般先写 BFS（因为层序遍历这个词本身就暗示了 BFS），再用 DFS 作为 follow-up 的优化或变体展示。

我们选哪个？

BFS 是层序遍历最自然的表达——题目叫”层序遍历”，我们就用层序的方法。如果面试官问”能不能不用队列”，再切换到 DFS 的递归写法。

版本 时间复杂度 空间复杂度 说明
BFS 队列 $O(n)$ $O(n)$ 每个节点入队出队各一次；最坏情况（满二叉树最后一层约 $n/2$ 个节点），队列占用 $O(n)$
DFS 递归 $O(n)$ $O(n)$ 每个节点访问一次；递归栈深度等于树高 $h$，最坏 $O(n)$（链状树），结果数组 $O(n)$

两种方法的时间都是 $O(n)$，因为我们”访问”了每个节点恰好一次。空间上，BFS 的队列峰值出现在最宽的一层（满二叉树的叶子层），DFS 的递归栈峰值出现在最深的一条路径（链状树），所以最坏情况都是 $O(n)$。平均情况下，对于平衡二叉树，BFS 的队列空间是 $O(n/2) \approx O(n)$，DFS 的递归栈是 $O(\log n)$，DFS 更优一些。但实际面试中，BFS 仍然是层序遍历的标准答案。

延伸思考

这道题让我们学到了什么？

「遍历」的顺序决定了数据结构的选型。

  • 前序/中序/后序是深度优先，天然适合递归（隐式栈）；
  • 层序是广度优先，天然适合队列（显式先进先出）。

当我们拿到一个遍历类的问题时，先问自己一句：这个顺序是纵向的还是横向的？ 纵向的想递归，横向的想队列。这个直觉一旦建立，后面遇到”之字形遍历””右视图””每层最大值”等题目时，都能快速找到切入点。

下次遇到类似问题优先观察什么？

  1. 是否需要分层输出？ 如果需要按层切分结果，一定在 while 循环里加一个 size = queue.size() 的步骤，这正是分层的关键。
  2. 顺序有没有变体？ 比如”之字形遍历”（Zigzag）就是偶数层反转一下列表，核心里还是层序遍历的那个框架。
  3. 能否用 DFS 替代？ 如果题目只是”按层分类”，不要求严格从左到右的顺序，DFS 加深度参数的写法往往更简洁。

本文以层序遍历为引子，展示了如何从”排队”的生活直觉抽象出队列数据结构，再到用 size 变量精确控制层边界的完整思考链条。这个模式可以推广到一大类树的广度优先问题，不妨在下一道题里亲自试试。
