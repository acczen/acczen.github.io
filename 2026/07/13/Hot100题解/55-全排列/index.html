核心思路

第 1 层：先想想人类怎么排

给定 [1, 2, 3]，让我们手动列出所有排列：

我们是怎么做的？先固定第一个位置选一个数，然后从剩下的数里选第二个位置，最后放最后一个数。选过的数不能再选。

把这个过程翻译成代码：3 层循环嵌套，每层排除之前选过的数。对长度固定的数组可以这么写，但如果 nums 长度不固定呢？如果 n = 10 呢？写 10 层 for 循环显然不现实。

我们遇到了第一个问题：循环层数不固定，怎么处理？

第 2 层：递归就是”可伸缩的循环”

既然循环层数由 n 决定，那我们就写一个函数，让它自己调用自己——每调用一次，就相当于多嵌套了一层循环。这就是递归的直觉：把”多层循环”的体力活交给函数调用栈。

第 3 层：回溯三问

写回溯算法时，我们习惯问自己三个问题：

  1. 路径：已经选了哪些数？
  2. 选择列表：当前还可以选哪些数？
  3. 结束条件：什么时候得到一个完整答案？

对应到本题：

  • 路径：track 列表，存放已选的数字。
  • 选择列表：nums 中尚未被选中的数字。我们用一个 boolean[] used 数组来标记。
  • 结束条件：track.size() == nums.length。

这就是回溯算法的标准骨架。让我们走一遍 [1, 2, 3] 的递归树（只画一条分支）：

第 4 层：有没有别的写法？

除了用 used 数组，还可以用 LinkedList 的 contains() 方法判断，但那样每次查找都是 O(n)，总复杂度退化为 O(n²·n!)。还可以通过交换元素的方式原地生成排列，避免额外的 used 数组和 track 列表——那就是另一种经典写法了，核心思想一样，实现更简洁，但理解门槛稍高一些。

我们这里优先掌握 used 数组版本，它是最直观、最通用的回溯模型，适用于绝大多数排列、组合、子集类问题。

复杂度分析

版本 时间复杂度 空间复杂度 说明
回溯（used 数组） O(n·n!) O(n) n! 个叶子节点，每个叶子拷贝路径 O(n)；递归栈深度 O(n)，used 数组 O(n)，不计输出空间

详细说明：

  • 全排列总数为 n!，每个排列需要 O(n) 时间加入结果集（拷贝）。
  • 递归树的内节点数量略少于叶子节点，同阶处理。
  • 空间复杂度指额外空间（不含答案所占的输出空间），包括递归调用栈深度 O(n) 和 used 数组 O(n)。

延伸思考

这道题让我们学到了什么？

回溯算法的通用模板。全排列是回溯思想最干净、最典型的应用题。做完这道题，我们就掌握了回溯的”三板斧”：

  1. 路径（track）
  2. 选择列表（通过标记数组或剩余集合）
  3. 结束条件（base case）

这个模板稍加变形就能解决子集（78）、组合（77）、N 皇后（51）、括号生成（22）等一系列经典问题。它们的区别仅仅在于”选择列表的剪枝规则”和”结束条件的定义”。

下次遇到类似问题优先观察什么？

  • 问题是否在问”所有可能的组合/排列/方案”？如果是，大概率是回溯或动态规划。回溯通常用于穷举所有解。
  • 是否可以区分”已选”和”未选”？对于排列类问题，明确的标记机制（used 数组或 swap 写法）是核心。
  • 有没有重复元素？如果有重复，需要在同一层递归中去重（排序后跳过相同值），这是下一道题”全排列 II”（47）的考点。

回溯本质上是一种有策略地暴力枚举——通过”做选择→递归→撤销选择”的模式，在不重不漏地遍历所有可能的同时，把代码写得优雅清晰。掌握了回溯，就等于掌握了算法面试中的半壁江山。
