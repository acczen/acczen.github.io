核心思路

让我们像做一道数学证明题一样，从最朴素的想法开始，一步步逼近最优解。

第一反应：暴力排序

如果没有任何技巧，我们第一反应会怎么做？

“把数组排序一下，然后从 1 开始往后找，第一个没出现的正整数不就是答案吗？”

确实，排序后数组变得有序，我们只需要这样扫描：

从 1 开始递增，检查当前数是否等于这个期望值，如果不等或者已经超过数组长度，就找到了。

这个解法很直观，但是——排序本身就需要 O(n log n) 的时间。题目明确要求 O(n) 时间复杂度，这条路走不通。

那能不能不排序？有没有办法在无序的数组中快速判断”某个数是否出现过”？

哈希集合：以空间换时间

如果需要快速判断某个数是否存在，我们自然会想到哈希表。

“把所有正数放进一个 HashSet，然后从 1 开始递增查询，第一个查不到的返回值就是答案。”

时间复杂度降到了 O(n)，完美满足时间要求。

但代价是 O(n) 的额外空间——题目要求的是常数级别额外空间。

问题来了：能不能既做到 O(n) 时间，又做到 O(1) 空间？

这看起来像是”既要马儿跑，又要马儿不吃草”。但让我们想一想：哈希表的本质是什么？

哈希表本质上是一个”值→位置”的映射。我们为什么需要额外开一个哈希表？因为我们没有一个天然的、可以直接使用的映射结构。

等一下——数组本身不就是一种天然的映射结构吗？

关键洞察：答案的范围有界

让我们停下来思考一个问题：

对于一个长度为 n 的数组，缺失的第一个正数最大可能是多少？

假设数组里包含了 1, 2, 3, …, n 所有正整数，那答案就是 n+1。否则，缺失的那个数一定在 1 到 n 之间。

所以答案一定在 [1, n+1] 这个范围内。

这意味着什么？意味着我们只需要关心 1 到 n 之间的数，超出这个范围的数（负数、零、大于 n 的数）对答案没有任何影响。

那我们是不是可以利用数组本身来”标记”哪些数字出现过？

原地哈希：把数组变成我们的哈希表

核心思想：用数组的索引（下标）来映射数字，用数组元素的值来标记该数字是否出现过。

具体来说：

  • 数字 x 应该对应索引 x-1
  • 如果 x 出现过，我们就在索引 x-1 上做一个”已出现”的标记

那么问题来了：怎么做标记才不会丢失数组原有的信息？

有两种经典做法：

做法一：置换法

既然数字 x 应该放在索引 x-1 的位置上，那我们直接把每个数”送回家”不就行了？

遍历数组，对于每个在 [1, n] 范围内的数，把它放到它应该在的位置上。然后再次遍历，第一个”位置不对”的地方就是答案。

好比整理书架：每本书上都印着它应该在的层架号，我们只需要把每本书放到正确的位置。整理完之后，第一个空着的书架位就是我们要找的答案。

做法二：正负标记法

我们换一种思路——不移动元素，而是用正负号来标记。

第一步：把所有非正数（<=0）和大于 n 的数替换成一个”无害”的值（比如 n+1）。

第二步：遍历数组，每遇到一个在 [1, n] 范围内的数 x，就把索引 x-1 上的数标记为负数（取绝对值后再取反）。

第三步：遍历数组，第一个正数所在的索引 +1 就是答案。

这个过程就像在考勤表上打勾——每个数出现一次，就在对应的格子上画一个勾（这里是把数变成负数）。

让我们用具体的例子走一遍”标记法”的流程，这样更清楚。

以 nums = [3, 4, -1, 1] 为例，n = 4：

第一步：清理
把 <= 0 或 > n 的数替换成 n+1 = 5：

第二步：标记

  • i=0: nums[0] = 3，标记索引 2 上的数：nums[2] = -|5| = -5 → [3, 4, -5, 1]
  • i=1: nums[1] = 4，标记索引 3 上的数：nums[3] = -|1| = -1 → [3, 4, -5, -1]
  • i=2: nums[2] = -5，取绝对值后是 5 > n，忽略
  • i=3: nums[3] = -1，取绝对值后是 1，标记索引 0 上的数：nums[0] = -|3| = -3 → [-3, 4, -5, -1]

第三步：扫描

索引 1 上是正数（4 > 0），说明数字 2 没有出现过。
答案是 1 + 1 = 2。✅

复杂度分析

版本 时间复杂度 空间复杂度 说明
排序暴力法 O(n log n) O(1)（原地排序） 排序后遍历查找，最简单但超时
哈希集合法 O(n) O(n) 用哈希集存储正数，不满足空间要求
原地置换法 O(n) O(1) 每个元素最多被交换一次（准确说每个索引最多被访问两次）
正负标记法 O(n) O(1) 三遍线性扫描，不使用额外空间

补充说明：置换法中看似用了 while 循环，但每个元素最多被交换到正确位置一次，所以整体仍然是 O(n) 时间复杂度。

延伸思考

这道题让我们学到了什么？

利用数组本身作为哈希表是一个极其重要的技巧。当我们被限制不能使用额外空间，却又需要”快速查找”的能力时，不妨看看数据本身是否提供了天然的索引结构。

这道题给我们的启示是：

  • 首先分析答案的可能范围——很多时候看似无限的问题，在给定输入规模下是有边界的
  • 在范围确定的前提下，数组的下标就是天然的哈希键，值是天然的标记位
  • 原地修改数组时，要确保修改是可逆的或不会丢失关键信息（置换法通过交换保留了所有信息，标记法通过符号位保留了数值的绝对值）

下次遇到类似问题优先观察什么？

当看到这些关键词时，可以优先考虑原地哈希的思路：

  1. “缺失的第一个 XXX”——答案范围往往有界，可以用索引映射
  2. “O(n) 时间，O(1) 额外空间”——极大概率需要原地哈希
  3. “数组中的数字在 [1, n] 范围内”或“数组中只包含 1 到 n”——这是最直接的信号

实际上，LeetCode 上还有好几道同类型题目：

  • 442. 数组中重复的数据——同样用索引标记法找出现两次的数
  • 448. 找到所有数组中消失的数字——同样是 [1, n] 范围，用索引找没出现的数

这三道题共享同一个核心思想：当数组元素的值本身可以被映射到数组的索引范围时，就用索引作为哈希表。

记住这个”索引映射”三板斧：置换归位、正负标记、遍历检测。下次遇到同类问题，可以让思路沿着这条线先跑一遍。
