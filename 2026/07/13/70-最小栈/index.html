最小栈

最小栈

核心思路

第一反应：暴力法

让我们先想想最直观的做法。题目要求我们实现一个栈，支持常规的 push、pop、top，外加一个 getMin() 返回当前栈中的最小值。

如果我们直接用数组或链表来实现栈，push、pop、top 都很容易做到 O(1)。但 getMin() 呢？

每次调用 getMin() 时遍历整个栈找最小值，时间复杂度是多少？

是 O(n)。如果 getMin() 被频繁调用，这样的性能是不行的。我们能不能让 getMin() 也达到 O(1)？

关键矛盾

要获取最小值，本质上需要”记住”历史中的最小值。栈是后进先出的，当栈顶元素被弹出时，最小值可能发生变化——我们如何知道变化后的最小值是什么？

举个例子，假设我们依次 push：5、3、7、2。当前最小值是 2。如果 pop 掉 2，最小值变成多少？是 3。如果继续 pop 掉 7，最小值还是 3。再 pop 掉 3，最小值变成 5。

也就是说，最小值的变化是有”记忆”的——每一层栈状态都对应一个”当前最小值”。

辅助栈：空间换时间

既然每一层栈状态都有一个对应的最小值，那我们可以用另一个栈来同步记录这个信息。

辅助栈思路：每当我们向主栈 push 一个元素时，辅助栈就 push 当前的最小值（即 min(新元素, 辅助栈栈顶)）。这样一来，主栈和辅助栈的长度始终保持一致，getMin() 直接取辅助栈的栈顶即可。

每一步的 getMin() 就是辅助栈的栈顶，O(1)。完美。

进一步优化：压缩辅助栈

上面的方案虽然简单，但有没有浪费空间？注意看：当 push 7 时，当前最小值是 3，并没有变化，但我们仍然向辅助栈 push 了一个 3。如果连续 push 大量比当前最小值大的元素，辅助栈里会重复存储很多相同的值。

能不能只在最小值发生变化时才记录？

可以的。我们改变辅助栈的规则：

  • push(x)：如果辅助栈为空，或者 x ≤ 辅助栈栈顶，才把 x push 进辅助栈
  • pop()：如果主栈弹出的元素等于辅助栈栈顶，辅助栈也 pop

这样辅助栈只存储”最小值变化时刻”的值，不会重复记录相同的最小值。

代价是什么？当连续 push 多个等于当前最小值的元素时，辅助栈只记录其中一个；但 pop 时，如果主栈弹出的是最小值，辅助栈也会弹出——如果后面还有相同的最小值没有被记录，就会丢失信息。

这个优化有一个特殊情况：如果 push 了重复的最小值怎么办？

所以严格的规则应该是：当 x ≤ 辅助栈栈顶时，push 进辅助栈。这样即使有重复的最小值，也能正确记录。让我们再走一遍：

如果第二个 3 没有 push 进辅助栈，那么在第一次 pop 后辅助栈的栈顶就会变成 1，最小值就错了。所以用 ≤ 而非 < 是关键细节。

不依赖辅助栈的实现

既然我们已经有了辅助栈方案，不妨再想一步：能否不用额外栈？

答案是可以的。我们可以在栈中存储一个”差值”或利用一个额外变量记录最小值。一个经典做法是：每个元素入栈时，栈中存储的是它与当前最小值的差值，同时用一个变量维护当前最小值。

但这种方法涉及数值运算的边界问题（整数溢出），可读性也差很多。在实际面试和工程中，辅助栈方案是最清晰、最不容易出错的实现，也是大多数场景下的首选。

延伸思考

这道题让我们学到了什么？

“状态快照” 的思想。当我们操作一个线性结构，并且需要快速查询”历史统计信息”时，可以额外维护一个同步结构来记录每一层的状态。这其实是一种 空间换时间 的缓存思维——把每次查询需要重新计算的结果，提前在每次修改时维护好。

这个思路在很多地方都有体现：

  • 单调栈：维护栈内元素的单调性，快速求解下一个更大/更小元素
  • 前缀和：一次遍历记录累积信息，支持 O(1) 区间求和
  • LRU 缓存：在 O(1) 内快速访问最近使用的数据

下次遇到类似问题优先观察什么？

  1. 操作是否具有”阶段关联性”——像栈这种后进先出的结构，当前状态和下一个状态之间有明确的继承关系，我们可以利用这种关系递推地维护信息
  2. 是否有冗余计算——每次 getMin 都遍历一遍整个栈显然做了重复工作，发现冗余的地方就是优化的入口
  3. 是否能分层记录——无论是同步辅助栈还是压缩辅助栈，本质都是把”查询时需要的信息”在”修改时同步更新”，让查询从 O(n) 降到 O(1)

最后，保持对边界情况的敏感——重复值、空栈操作、整型溢出——往往就是 bug 藏身的地方。
