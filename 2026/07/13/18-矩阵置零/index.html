矩阵置零

矩阵置零

核心思路

第一反应：暴力复制

拿到这道题，先别急着想技巧。不妨先问自己一个问题：如果没有任何限制，我们第一反应会怎么做？

最简单直接的想法是：再准备一个同样大小的矩阵，遍历原矩阵，遇到零就把新矩阵对应的整行整列置零。最后把新矩阵复制回去。

这个思路几乎不需要动脑筋，但它的代价是什么？

  • 空间上，我们需要一个 O(mn) 的额外矩阵。
  • 时间上，每个零都要扫一遍它所在的行和列，最坏情况下（全是零）会退化到 O(mn * (m+n))。

显然，这不是一个优雅的做法。

第一次优化：用数组记住「哪些行、哪些列需要置零」

上面的暴力法之所以慢，是因为重复计算——同一个位置可能被不同的零反复置零。那么我们能不能先把信息存下来，一次性地处理？

答案是肯定的。我们可以准备两个布尔数组：

  • rows[] —— 大小为 m，记录哪一行需要置零
  • cols[] —— 大小为 n，记录哪一列需要置零

第一遍遍历矩阵，如果 matrix[i][j] == 0，就把 rows[i] = true、cols[j] = true。

第二遍遍历，对于每个位置 (i, j)，如果 rows[i] 或 cols[j] 为 true，就把它设为 0。

这样一来，时间复杂度降到了 O(mn)，空间复杂度是 O(m + n)。

相比暴力法，我们进步在哪里？

我们付出了 O(m+n) 的空间，换来了一次遍历记录、一次遍历执行的线性时间。本质上是用空间换时间，把信息集中管理，避免了重复劳动。

还可以再省吗？—— O(1) 空间的终极优化

走到这一步，我们自然会追问：能不能连 O(m+n) 的额外空间都不要？

如果我们能把「哪行哪列需要置零」的信息直接存在矩阵本身，不就省掉额外数组了吗？

矩阵中哪些位置是”闲置”的？——第一行和第一列。

具体做法是：

  1. 先检查第一行和第一列本身是否包含 0，用两个布尔变量 firstRowZero、firstColZero 记下来。
  2. 然后，把第一行和第一列当作标记数组来用：遍历矩阵剩余部分（从 (1,1) 开始），如果 matrix[i][j] == 0，就让 matrix[i][0] = 0 和 matrix[0][j] = 0。
  3. 接着，根据第一行和第一列的标记，把对应行和列置零（同样从 (1,1) 开始）。
  4. 最后，处理第一行和第一列本身——如果之前的标记变量为 true，就把它们置零。

为什么要用两个变量保存第一行和第一列的原始信息？因为第一行和第一列在步骤 2 中被改写了，如果不提前保存，我们就丢失了它们原本是否包含 0 的信息。

这个思路的核心：整个矩阵的标记信息被压缩到了两条”边界线”上，我们不需要任何额外数组。这就是原地算法的精髓——在已有空间里巧妙地编码额外信息。

三种解法对比

版本 思路 空间 时间
暴力法 复制矩阵，遇到零就改副本 O(mn) O(mn(m+n))
标记数组 用两个数组记录需要置零的行和列 O(m+n) O(mn)
原地标记 用第一行和第一列做标记 O(1) O(mn)

延伸思考

这道题让我们学到了什么？

在原始数据上编码额外信息的技巧。 很多时候，我们需要记录一些”元信息”（比如哪些行该置零），并不一定要开辟新的空间——如果原来的数据结构中有某些”闲置”的位置，完全可以拿来做标记。本题中的第一行和第一列天然成了两个”标记数组”的替代品。

这个思想在很多原地算法中都有体现，比如：

  • 原地哈希（用数组本身记录某个数字是否出现过）
  • 原地 DP（在输入数组上直接做状态转移，节省空间）

下次遇到类似问题优先观察什么？

  • 是否允许修改原数据？ 如果允许，考虑利用原数据的存储空间。
  • 标记信息能否压缩到边界或少量位置？ 比如第一行、第一列、对角线等”特殊位置”。
  • 信息的流向是什么？ 本题中，信息从矩阵内部”汇聚”到边界（第一行/列），再从边界”扩散”到整个矩阵。画清楚信息流向，代码结构自然就清晰了。
  • 是否需要保留原始信息？ 我们在使用第一行/列前先检查了它们自身的零值，因为标记过程会覆盖这些位置——先保存，再复用是一个常见套路。

下次遇到”需要标记但希望省空间”的矩阵问题，不妨先想想：哪里可以当黑板用？
