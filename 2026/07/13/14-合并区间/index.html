LeetCode 56. 合并区间（merge-intervals）

题目

LeetCode 56. merge-intervals

以数组 intervals 表示若干个区间的集合，其中单个区间为 intervals[i] = [start_i, end_i]。请你合并所有重叠的区间，并返回 一个不重叠的区间数组，该数组需恰好覆盖输入中的所有区间。

核心思路

拿到这道题，我们先不急着写代码。假设你在一条数轴上画这些区间——有些区间互相覆盖，有些孤零零地待着。你的任务就是把所有”连在一起”的区间捏成一段。你会怎么做？

先想想最直接的做法：两两比较。拿每一个区间和其余所有区间比较，如果重叠就合并。但等等——合并之后，新产生的区间可能又和别的区间重叠了，你得反复扫描直到没有变化为止。这种暴力解法的时间复杂度最坏能达到 $O(n^3)$，显然不够优雅。

问题出在哪？输入的区间是乱序的。比如示例 3 里 [1,4] 在 [0,4] 前面，你扫到 [1,4] 时根本不知道后面还有个能吃掉它的大家伙。这让我们很难一次性判断”当前区间应该和谁合并”。

那么，如果我们先排序呢？

把区间按起点从小到大排好序，神奇的事情发生了——任何一个区间如果会和前面的区间重叠，它一定和紧挨着的前一个区间有交集。为什么？

想象一根从左到右的数轴。如果区间 A 的起点比区间 B 早，而区间 C 的起点比 B 还晚，那么 C 想要和 A 重叠，它的起点就必须小于等于 A 的终点。但既然 B 在中间，这说明 B 的起点也小于等于 A 的终点——也就是说 B 已经和 A 重叠了。换言之，只要按起点排序，”重叠”这件事就只会发生在相邻元素之间，不会出现隔空重叠的现象。

于是整个问题就变成了：排序，然后从左到右”推土机”式地一次遍历——维护一个”当前正在合并的区间”，遇到能接上的就扩展，遇到接不上的就结算当前区间，然后开启新区间。

解法：排序 + 一次遍历（贪心）

为什么选择这个策略？

按起点排序后，问题从”任意两个区间都可能重叠”退化成了”只有相邻区间才可能重叠”。这让我们可以贪心地处理：每扫到一个新区间，只和当前正在合并的区间比较即可。

其他备选方案：

方案 思路 问题
两两比较 + 反复扫描 每次找到一个有重叠的区间对就合并，重复直到稳定 最坏 $O(n^3)$，不可接受
扫描线（差分数组） 在坐标轴上标记区间起点 +1、终点 -1，扫描数轴 坐标可能非常大（题设未限制），且无法直接输出区间列表
排序 + 一次遍历 按起点排序，贪心合并 $O(n \log n)$，简洁高效

显然，排序法是我们在这个问题上的最优解——思路直观，代码短，复杂度优秀。

算法步骤

  1. 排序：将所有区间按 $start_i$ 升序排列。
  2. 初始化：用第一个区间作为”当前合并区间”（不妨叫 cur）。
  3. 遍历：从第二个区间开始，逐个与 cur 比较：
    • 如果新区间的起点 $\leq$ cur 的终点 → 重叠，更新 cur 的终点为两者终点的较大值。
    • 否则 → 不重叠，将 cur 加入结果集，然后将新区间设为新的 cur。
  4. 收尾：遍历结束后，别忘了把最后一个 cur 加入结果集。

复杂度分析

维度 分析
时间复杂度 $O(n \log n)$ — 排序占主导，一次遍历仅为 $O(n)$
空间复杂度 $O(\log n)$（不考虑结果数组）— 排序的递归栈深度；若结果数组计入则为 $O(n)$

延伸思考

学到了什么？

排序可以将”全局无序关系”转化为”局部有序关系”。这道题是区间问题的经典模板——以后遇到”合并区间””插入区间””会议室问题””删除被覆盖区间”等变体，第一反应就应该是先排序，再贪心遍历。

下次遇到类似问题，你会优先观察什么？

问自己一个问题：“如果我把数据按某个维度排序，相邻元素之间会不会呈现出某种确定性规律？” 如果答案是肯定的，那么排序大概率就是你解题的钥匙。
