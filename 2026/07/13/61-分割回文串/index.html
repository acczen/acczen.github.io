分割回文串

分割回文串

核心思路

第一反应：枚举所有切分方案

拿到这道题，第一个出现在我们脑海中的想法往往是这样的：把一个字符串切成若干段，使得每一段都是回文串，然后返回所有可能的切分方案。

对于一个长度为 n 的字符串 s，它的 n-1 个间隔位置，每个间隔我们都可以选择”切”或者”不切”——这听起来是不是很像在枚举子集？没错，总共有 2^(n-1) 种切分方式。对每一种切分方式，我们检查切出来的每一段是不是回文串。

问题是：2^(n-1) 的增长速度极快。当 n=16 时就有 32768 种切分方式，当 n=20 时超过 50 万种。而这还只是切分的数量，还没算上检查回文的时间。

我们需要一个更聪明的办法——有没有办法在构造切分的同时就判断回文，提前剪掉不合法的分支？

换一个思路：逐段构造

让我们重新思考这个问题。我们不一次性决定所有切分点，而是逐段地构造：

从左到右扫描字符串，每次尝试切出一个前缀子串。如果这个前缀是回文串，我们就把它加入当前路径，然后递归处理剩下的部分。如果前缀不是回文串，就直接跳过，因为一段不合法就意味着整个切分方案无效。

这个过程可以用一棵”决策树”来想象：根节点是整个字符串，每一层我们选择一个回文前缀作为第一段，剩下的字符串交给下一层去处理。

每次都要判断回文，能不能更快？

上面的做法中，我们每尝试一个子串 s[i:j] 都要从头判断它是不是回文。判断一次是 O(len) 的，如果同一个子串被反复判断，就会产生重复计算。

想一想：一个子串是不是回文，和它的”内层”子串有什么关系？

如果一个子串 s[i:j] 是回文，那么它的首尾字符 s[i] 和 s[j] 必须相等，并且去掉首尾后的子串 s[i+1:j-1] 也必须是回文。反过来，如果我们已经知道 s[i+1:j-1] 是回文，并且 s[i] == s[j]，那么 s[i:j] 也是回文。

这是一个典型的动态规划递推关系：

其中 j - i <= 2 处理的是长度为 1、2、3 的子串（长度 1 一定是回文，长度 2 和 3 只需首尾相等）。

于是我们可以提前把字符串所有子串的回文性质计算出来，存到一个二维布尔表中。之后在回溯过程中判断 s[i:j] 是否是回文时，只需要 O(1) 查表即可。

最终思路

把前面的思考串联起来：

  1. 预处理阶段：用动态规划计算出 isPalindrome[i][j]，表示子串 s[i:j]（包含两端）是否为回文串。
  2. 回溯阶段：从起始位置 start 开始，枚举结束位置 end，如果 s[start:end] 是回文（查表 O(1)），就将其加入当前路径，递归处理 end+1 开始的剩余部分。
  3. 终止条件：start == n 时，说明整个字符串已经切分完毕，将当前路径加入结果集。

整个过程类似于在字符串上做深度优先搜索，每次走一步（切出一个回文前缀），直到走完全程。

复杂度分析表

版本 时间复杂度 空间复杂度 说明
暴力枚举 O(n * 2^n) O(n) 枚举所有 2^(n-1) 种切分方式，每次验证回文 O(n)
回溯 + 实时判断回文 O(n * 2^n) O(n) 回溯过程中每步判断回文 O(n)，最坏仍为指数级
回溯 + DP 预处理 O(n * 2^n) O(n^2) 查表 O(1)，但最坏情况下的方案数本身就是 O(2^n) 级别

时间复杂度的补充说明：三种方案在最坏情况下（如 s = "aaaa...a"，所有子串都是回文）都会输出 O(2^n) 个方案，所以时间复杂度的主体是输出方案的数量，无法低于 O(2^n)。DP 预处理的优势体现在平均情况下——回文检查从 O(n) 降到 O(1)，避免了大量的重复计算。

空间复杂度的补充说明：DP 表占用 O(n^2) 空间。回溯过程中 path 最多存储 n 个字符串，递归深度最多 n 层，这部分是 O(n)。结果集 result 的空间不计入额外空间复杂度（题目要求返回的结果）。

延伸思考

这道题让我们学到了什么？

回溯法的本质是在决策树上做深度优先搜索。 当我们面对”枚举所有可能方案”的问题时，回溯是最自然的工具。关键是要搞清楚三个要素：路径（已经做出的选择）、选择列表（当前可以做的选择）、终止条件（什么时候记录答案）。

动态规划不仅可以求最值，还可以做预处理。 这道题的 DP 不是为了直接求出答案，而是为回溯过程提供”快速查询”的能力。这种”DP 预处理 + 回溯搜索”的组合思路在很多复杂枚举问题中都会用到——先花一些代价建立辅助数据结构，再通过查表加速搜索过程。

剪枝的核心是尽早发现不可行的分支。 我们的剪枝只有一个条件——“当前前缀不是回文就直接跳过”。这个判断越早做、代价越低，搜索的效率就越高。

下次遇到类似问题优先观察什么？

  1. 问题是”枚举所有方案”还是”求最优方案”？ 如果是前者，优先考虑回溯/DFS；如果是后者，优先考虑 DP 或贪心。

  2. 能否提前建立一个辅助结构来加速判断？ 这道题的「回文判断」是回溯中的高频操作，我们选择了 DP 表来将判断时间从 O(n) 降到 O(1)。类似的场景还有：用前缀和加速区间求和、用邻接表加速图的遍历。

  3. 回溯时如何表示”当前状态”？ 这道题用了一个 start 指针来标记当前处理到的位置，加上一个 path 列表来记录已经切出的段。这是回溯中最经典的状态表示方式。

  4. 所有子串都是回文时怎么办？ 这种最坏情况（比如输入 "aaaa...a"）下，回文子串的数量本身是指数级的，任何算法都无法避免输出 O(2^n) 个结果。认识到这一点很重要——有些问题的输出规模决定了它的下界，这时候不要追求”更快的算法”，而是要思考是否有不需要枚举所有方案的其他解法（比如题目只问”是否可能”或”最少切割次数”）。
