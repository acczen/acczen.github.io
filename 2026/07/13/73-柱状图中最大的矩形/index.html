柱状图中最大的矩形

核心思路

从问题出发

给定一个数组 heights，每个元素代表一根柱子的高度，每根柱子的宽度都是 1。这些柱子并排排列，构成了一个柱状图。我们要在这个柱状图中，找出一个最大的矩形，这个矩形必须完全由柱状图内的柱子覆盖——也就是说，矩形的宽必须是连续若干根柱子的宽度之和，矩形的高不能超过这些柱子中最矮的那一根。

直觉上，最大的矩形是中间 5 和 6 形成的区域，高为 5、宽为 2，面积 10。但我们怎么系统性地找到它呢？

第一反应：暴力枚举所有区间

最朴素的想法——枚举所有可能的左右边界对 (l, r)，然后在这个区间内找到最矮的柱子作为高，乘以宽度 (r - l + 1)，就得到了这个区间能容纳的最大矩形面积。

这样做，枚举区间是 O(n²)，每次找区间最小值又是 O(n)，总复杂度 O(n³)。即使我们优化一下——在扩大右边界时同步维护最小值——依然是 O(n²)。

自问：n 最大到 10⁵，O(n²) 显然不可接受。我们能否不枚举所有区间？

换个视角：枚举每根柱子作为”短板”

刚才我们发现，一个矩形的面积由其宽度和最矮的那根柱子共同决定。反过来想：对于每一根柱子 i，如果我们以 heights[i] 作为矩形的高度，那这个矩形能往左右各延伸多远？

向左，直到遇到第一根比 heights[i] 更矮的柱子（停下）；
向右，直到遇到第一根比 heights[i] 更矮的柱子（停下）。

这两根”更矮的柱子”之间的范围，就是 heights[i] 作为高度时能撑起的最大宽度。

$$
\text{area}_i = \text{heights}[i] \times (\text{right_boundary} - \text{left_boundary} - 1)
$$

其中 left_boundary 是左边第一个比 heights[i] 矮的柱子下标，right_boundary 是右边第一个比 heights[i] 矮的柱子下标。

这样一来，我们只需要对每个 i 求出左右边界，计算面积并取最大值即可。

自问：这个方法本质上还是需要每个柱子都向左向右扫描一遍，最坏情况（比如柱子高度单调递增）下依然是 O(n²)。瓶颈在哪里？

瓶颈在于：对每个 i，我们都要重新向左/向右扫描。这些扫描中有大量的重复工作——相邻柱子的边界信息其实是有关联的。

自问：有没有一种结构，能让我们在遍历一遍的过程中就记住所有柱子的边界信息？

优化：单调栈登场

想一想，我们关心的是”左边/右边第一个更矮的柱子”——这本质上是在寻找最近的最小值。看到”最近的最小值”这个关键词，单调栈就应该浮现在脑海中。

我们维护一个单调递增栈（从栈底到栈顶严格递增），栈中存放的是柱子的索引。这个栈的含义是：到目前为止，尚未确定右边界的柱子，它们的高度是递增排列的。

为什么是递增的？因为一旦出现一个比栈顶更矮的柱子，栈顶柱子的右边界就找到了——就是当前这根新柱子。

让我们模拟一遍：

初始化栈，放入一个虚拟下标 -1（哨兵，帮助我们统一处理边界）。

遍历 i = 0, 1, 2, ..., n-1：

  1. 如果 heights[i] >= heights[栈顶]，将 i 入栈。

  2. 如果 heights[i] < heights[栈顶]，说明栈顶柱子的右边界就是 i。弹出栈顶，计算面积：

    • 高度 h = heights[弹出下标]
    • 左边界 L = 当前栈顶（即弹出后的新栈顶，它是左边第一个更矮的柱子）
    • 右边界 R = i
    • 宽度 w = R - L - 1
    • 面积 = h * w

    不断重复，直到栈顶柱子不再比 heights[i] 高，然后将 i 入栈。

遍历结束后，栈中剩下的柱子都没有遇到右边比它们矮的柱子——它们的右边界都是 n（数组末尾）。同样地，依次弹出计算面积。

模拟过程

以 [2, 1, 5, 6, 2, 3] 为例：

i = 0, heights[0] = 2

  • 栈：[-1]
  • 2 >= 哨兵，入栈 → 栈：[-1, 0]

i = 1, heights[1] = 1

  • 栈：[-1, 0]
  • heights[1] = 1 < heights[0] = 2，弹出 0：
    • h = 2, L = -1, R = 1, w = 1 - (-1) - 1 = 1, area = 2 × 1 = 2
  • 栈：[-1]
  • 1 >= 哨兵，入栈 → 栈：[-1, 1]

i = 2, heights[2] = 5

  • 栈：[-1, 1]
  • 5 >= heights[1] = 1，入栈 → 栈：[-1, 1, 2]

i = 3, heights[3] = 6

  • 栈：[-1, 1, 2]
  • 6 >= heights[2] = 5，入栈 → 栈：[-1, 1, 2, 3]

i = 4, heights[4] = 2

  • 栈：[-1, 1, 2, 3]
  • 2 < heights[3] = 6，弹出 3：
    • h = 6, L = 2, R = 4, w = 4 - 2 - 1 = 1, area = 6 × 1 = 6
  • 栈：[-1, 1, 2]
  • 2 < heights[2] = 5，弹出 2：
    • h = 5, L = 1, R = 4, w = 4 - 1 - 1 = 2, area = 5 × 2 = 10
  • 栈：[-1, 1]
  • 2 >= heights[1] = 1，入栈 → 栈：[-1, 1, 4]

i = 5, heights[5] = 3

  • 栈：[-1, 1, 4]
  • 3 >= heights[4] = 2，入栈 → 栈：[-1, 1, 4, 5]

遍历结束，处理栈中剩余元素

  • 弹出 5：h = 3, L = 4, R = 6, w = 6 - 4 - 1 = 1, area = 3 × 1 = 3
  • 弹出 4：h = 2, L = 1, R = 6, w = 6 - 1 - 1 = 4, area = 2 × 4 = 8
  • 弹出 1：h = 1, L = -1, R = 6, w = 6 - (-1) - 1 = 6, area = 1 × 6 = 6

最大面积为 10 ✅

哨兵的作用

我们往栈里放了一个 -1。为什么？

如果没有哨兵，当栈被弹空时，左边界需要特殊处理（不存在更矮的左柱子，所以左边界为 -1）。哨兵统一了逻辑：任何时候左边界都是 stack[top]，避免了条件分支判断。

类似地，在遍历结束后的清算阶段，右边界统一为 n，和哨兵配合使用，计算宽度的公式 R - L - 1 始终成立。

复杂度分析

版本 时间复杂度 空间复杂度 说明
暴力枚举区间 O(n³) O(1) 枚举所有左右边界，每次计算区间最小值
枚举短板 + 左右扩展 O(n²) O(1) 每个柱子向两侧找到第一个更矮的柱子
单调栈（数组模拟） O(n) O(n) 每个元素最多入栈一次、出栈一次
单调栈（栈结构） O(n) O(n) 使用系统栈或 Deque，本质同上

为什么是 O(n)？ 每个柱子至多入栈一次、出栈一次，总体操作次数与 n 成正比，而非 n²。

空间上的优化：由于栈中最多同时存储 n 个下标，最坏情况（柱子高度单调递增）下确实需要 O(n) 空间。但这是输入数组本身的规模，无法避免。

延伸思考

这道题让我们学到了什么？

单调栈是一种为”寻找下一个更大/更小元素”而生的数据结构。 它通过维护栈内元素的单调性，在一次遍历中优雅地解决了看似需要多层循环的问题。

本题的核心洞察在于转换视角：从”枚举所有可能的矩形”变为”枚举每个柱子作为矩形的高度”。这一转换将问题从组合爆炸中解放出来，让每个元素只关心它自己的左右边界，从而为单调栈的应用铺平了道路。

下次遇到类似问题，优先观察什么？

当一个问题涉及以下特征时，可以优先考虑单调栈：

  1. 需要求某个元素的左边/右边第一个比它大（或小）的元素；
  2. 数组中的元素之间存在”上一个/下一个”的依赖关系；
  3. 问题的暴力解法涉及双向扩散扫描，且扫描过程存在可复用的信息。

典型的同类问题还有：

  • 接雨水（Trapping Rain Water）—— 同样是柱子数组，求能接住多少水，也可以用单调栈求解。
  • 每日温度（Daily Temperatures）—— 求每个温度之后第一个更高的温度出现在第几天。
  • 最大矩形（Maximal Rectangle, 85 题）—— 本题的二维扩展，逐行累计高度后复用本题解法。

一个更深的视角

回到”哨兵”的设计，我们往栈里放了一个 -1。这个小技巧在边界处理中反复出现：链表的虚拟头结点、字符串算法中的哨兵字符……它的本质是将特例归入通例，让边界条件不再需要单独 if 判断。写代码的时候，多想想”能不能加一个哨兵让逻辑统一”，往往能让代码更简洁、更不易出错。

从暴力到单调栈，优化的是算法的复杂度，不变的是一颗不断追问”能不能更好”的心。希望这道题的分析过程，能为你理解单调栈带来一些启发。🎯
