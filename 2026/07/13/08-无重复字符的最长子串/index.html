LeetCode 3. 无重复字符的最长子串（longest-substring-without-repeating-characters）

题目

LeetCode 3. longest-substring-without-repeating-characters

给定一个字符串 s，请你找出其中不含有重复字符的 最长子串 的长度。

提示：

  • 0 <= s.length <= 5 * 10^4
  • s 由英文字母、数字、符号和空格组成

核心思路

拿到这道题，我们先问自己一个问题：“子串”和”子序列”的区别是什么？

子串必须是连续的，而子序列可以不连续。这意味着当我们考察一个子串时，它一定是原字符串中一段连续的切片。这个看似简单的观察，实际上决定了我们后面所有的思考方向。

好，现在让我们从最朴素的想法开始。

如果我们想找”最长的不含重复字符的子串”，最直接的做法是什么？

最直接的做法是：枚举所有子串，逐个检查它是否不含重复字符，记录最长的那个。

这个做法有什么问题？

时间复杂度是 $O(n^3)$ 或至少 $O(n^2)$（枚举起点和终点是 $O(n^2)$，每次检查重复又要 $O(n)$）。当字符串长度达到 $5 \times 10^4$ 时，这显然是行不通的。

那瓶颈在哪里？我们做了哪些”无用功”？

仔细观察：当我们检查完 "abc" 知道它无重复后，再去检查 "abca" 时，我们真的需要把 "abca" 从头到尾扫描一遍吗？

不用的。我们只需要注意到：新加入的 'a' 已经在 "abc" 中出现过了，所以 "abca" 一定不是答案。而且，以第一个 'a'（索引 0）开头的所有更长的子串，也都不用再考虑了——因为它们都会被这个重复的 'a' 卡住。

这意味着什么？

意味着我们可以动态地维护一个”窗口”——它始终代表”以当前位置为右端点的、最长的无重复字符子串”。当右边界向右扩展时，如果新字符和窗口内某个字符重复了，我们就收缩左边界，直到重复消失。

这，就是滑动窗口的核心思想。

为什么滑动窗口是安全的？我们会不会因为收缩左边界而”错过”某些更长的答案？

不会。想想看：当我们发现窗口 [left, right] 中出现了重复字符时，以原 left 为起点的所有更长子串也必然包含这个重复。所以我们可以安全地将 left 右移，跳过这些不可能成为答案的起点。这本质上是一种剪枝——不是暴力枚举所有起点，而是只维护那些”还有希望”的起点。

总结一下思考链条：

  1. 暴力枚举所有子串 → 太慢，$O(n^3)$
  2. 观察发现：重复字符一旦出现，以旧起点开头的所有子串都失效 → 可以跳过无效起点
  3. 维护一个动态窗口 [left, right]，窗口内始终无重复 → $O(n)$
  4. 窗口右扩时检查重复，重复则左缩 → 滑动窗口算法

解法：滑动窗口 + 哈希表

为什么选哈希表？

窗口内需要频繁回答一个问题：“字符 c 是否已经在窗口中出现过？如果出现过，它上次出现在哪个位置？”

我们需要一个支持 $O(1)$ 查找的数据结构。哈希表（HashMap / 字典）天然适合这个场景：键是字符，值是该字符最近一次出现的索引。

用数组可以吗？

可以的。因为题目说字符集包括英文字母、数字、符号和空格，如果字符范围可控（比如纯 ASCII），用一个长度为 128 或 256 的数组也能达到 $O(1)$ 查找。但哈希表的通用性更好，我们这里用哈希表来讲解。

和其他方案的对比

方案 时间复杂度 空间复杂度 评价
暴力枚举 $O(n^3)$ $O(1)$ 不可接受
暴力 + 哈希去重 $O(n^2)$ $O(n)$ 仍然太慢
滑动窗口 + 哈希表 $O(n)$ $O(\min(n, \text{字符集大小}))$ 最优
滑动窗口 + 数组 $O(n)$ $O(\text{字符集大小})$ 字符集小时更优

算法步骤

  1. 维护两个指针 left 和 right，分别表示窗口的左右边界（左闭右闭 [left, right]）
  2. 维护一个哈希表 map，记录每个字符最近一次出现的索引
  3. right 从左到右遍历字符串：
    • 如果 s[right] 已经在 map 中，且其上一次出现的位置 >= left，说明窗口内出现了重复
    • 此时将 left 跳到 map.get(s[right]) + 1（即重复字符上一次出现的下一个位置）
    • 更新 map 中 s[right] 的最新位置为 right
    • 计算当前窗口长度 right - left + 1，更新全局最大值
  4. 返回全局最大值

注意一个细节：我们只在重复字符的位置 >= left 时才移动 left。为什么？因为如果重复字符上次出现在 left 之前（即已经在窗口外面了），那它不影响当前窗口的合法性，我们不需要收缩。

复杂度分析

维度 分析
时间复杂度 $O(n)$ — 每个字符最多被 left 和 right 各访问一次，哈希表的插入和查找都是 $O(1)$
空间复杂度 $O(\min(n, m))$ — 其中 $m$ 是字符集大小。哈希表最多存储窗口内出现过的不同字符。最坏情况下（全部字符不同）存储 $n$ 个键值对

延伸思考

学到了什么？

滑动窗口不是凭空而来的技巧——它来自于对暴力解法的观察和剪枝：当我们发现某个起点已经”没希望”时，就果断放弃它，把资源集中在仍有潜力的起点上。 这本质上是一种贪心思想：每一步都维护一个”当前最优”的局部解，而局部最优恰好能导向全局最优。

下次遇到类似问题，你会优先观察什么？

问自己一个问题：“这个问题的答案是否具有单调性？” 具体来说——当右边界向右移动时，左边界是否也只能向右移动（或至少不会向左移动）？如果是，那么滑动窗口很可能是正确的方向。这个”单调性”直觉，是判断滑动窗口是否适用的第一把钥匙。
