LeetCode 49. 字母异位词分组（group-anagrams）

题目

LeetCode 49. group-anagrams

给你一个字符串数组，请你将字母异位词组合在一起。可以按任意顺序返回结果列表。

字母异位词是由重新排列源单词的所有字母得到的一个新单词——换句话说，两个字符串包含的字母完全相同，只是排列顺序不同。

核心思路

拿到这道题，我们不妨先问自己一个问题：

如果让你来判断两个字符串是不是字母异位词，你会怎么做？

最直接的想法可能是：把两个字符串各自排序，排序后完全一样的就是字母异位词。 这个方法简洁明了，而且正确性毋庸置疑——如果两个字符串由相同字母、相同频次组成，排序结果必然相等。

但这里有个问题——你有没有注意到？

两两比较所有字符串，时间复杂度会是多少？

数组长度最坏 $10^4$，两两比较就是 $O(n^2 \cdot k\log k)$（$k$ 是字符串平均长度），这个开销太大了，跑不过去。

那么瓶颈在哪儿？在于我们花了大量时间做重复的比较——同一个字符串可能和很多其他字符串比对，而每次比对都要排序。

换个角度想：

能不能给每个字符串算出一个”身份证号”——字母异位词共用一个号，不同词拥有不同的号？

如果我们能做到，那就只需要遍历一次数组，每个字符串根据它的”身份证号”找到对应的组，直接加入即可。这就把 $O(n^2)$ 的比较降到了 $O(n)$ 的哈希查找。

剩下的问题是：这个”身份证号”怎么设计？

思路很自然——既然排序后的字符串可以作为异位词的”标准形式”，那我们干脆拿它当 key：key = sorted(str)。所有排序后相同的字符串，自然就是同一组字母异位词。

有没有更高效的方式？排序本身需要 $O(k\log k)$，能更快吗？

因为题目限定只有小写字母（26 种），我们可以用一个固定长度的计数数组来表示每个字母出现的次数，然后把它转成字符串作为 key——比如 "a1b0c2...z0" 或 "#1#0#2...#0"。这样把 key 的计算从 $O(k\log k)$ 降到了 $O(k)$。

解法：哈希表 + 排序（或计数）

核心思想一句话：为每个字符串生成一个”标准形式”作为哈希表的 key，value 就是该组异位词的列表。

方案一：排序作 key

对每个字符串的字符数组排序，排序后的字符串就是 key。这是最直观的写法。

  • 优点：代码极简，不需要关心字符范围。
  • 缺点：排序耗时 $O(k\log k)$，当字符串很长时有优化空间。

方案二：字符计数作 key（推荐）

用一个长度为 26 的 int[] 统计每个小写字母的出现次数，然后拼接成一个唯一的字符串作为 key。例如 "eat" → "a1b0c0d0e1f0g0h0i0j0k0l0m0n0o0p0q0r0s0t1u0v0w0x0y0z0"。

  • 优点：key 计算仅需 $O(k)$，在字符串较长或数据量大时更优。
  • 缺点：需要显式处理 26 个小写字母的假设，不适合字符集不确定的场景。

为什么不直接用计数数组本身当 key？

Java 中 int[] 的 equals() 比较的是引用地址，两个内容相同的数组放进 HashMap 会被当成不同的 key。所以必须转成 String 或包装类型。Python 则可以简洁地用 tuple(count) 作为 key。

复杂度分析

维度 方案一（排序） 方案二（计数）
时间复杂度 $O(n \cdot k\log k)$ — 每个字符串排序 $O(n \cdot k)$ — 每个字符串遍历计数
空间复杂度 $O(n \cdot k)$ — 哈希表存储所有字符串 $O(n \cdot k)$ — 同上，外加常数级计数数组

其中 $n$ 是字符串数量，$k$ 是字符串最大长度。

延伸思考

学到了什么？

这道题的核心技巧是为等价对象设计”标准形式”（canonical form），然后用哈希表归并。排序和计数只是两种常见的标准形式构造方法——选择哪种取决于数据特征（字符集大小、字符串长度）。

下次遇到类似问题，你会优先观察什么？

遇到”分组”、”归类”类的问题，第一时间问自己：同一组的元素有什么共同的、可计算的特征？这个特征能不能在 $O(1)$ 或 $O(k)$ 时间内算出来？ 如果能，哈希表就是最自然的选择。
