字母异位词分组

难度：🟡 中等

核心思路

第一步：如果没有任何技巧，我们第一反应会怎么做？

拿到这道题，最直接的想法是：遍历每个字符串，再遍历剩下的字符串，判断它们是不是字母异位词，是的话就放到同一组里。

这个解法的问题在哪里？

我们分析一下复杂度：n 个字符串，两两比较是 O(n²) 次比较，每次比较都要对两个字符串排序（O(k log k)，k 是字符串平均长度）。总时间复杂度是 O(n² × k log k)。当字符串数量很大时，平方级别的复杂度是不可接受的。

第二步：有没有办法减少重复计算？

我们换个角度来看这个问题。

仔细想想暴力法的问题——其实有两个：

  1. 对同一个字符串，我们在和不同字符串比较时，重复排序了多次
  2. 我们其实不需要知道”某个字符串的异位词有哪些”，我们只需要把”所有互为异位词的字符串”分成一组

那有没有一个更优雅的视角呢？

不妨这样想： 如果两个字符串是字母异位词，那么它们排序后的结果是一样的。也就是说，排序后的字符串可以看作是这个”异位词家族”的身份证——所有持有同一张身份证的字符串，都应该住进同一间屋子。

这样一来，我们只需要一次遍历：对每个字符串计算它的身份证（排序结果），然后用哈希表把持相同身份证的人归到一个房间里。最后，把每个房间里的列表取出来就行了。

这个过程的核心转变是什么？ 从”两两比较”变成了”以签名归类”。前者是 O(n²) 的比较，后者是 O(n) 的哈希表操作。这是质的飞跃。

现在的复杂度：遍历 n 个字符串，每次排序 O(k log k)，总时间 O(n × k log k)。相比暴力法的 O(n² × k log k)，有了巨大的提升。

第三步：排序还是有点慢，能不能更快？

排序 O(k log k) 真的已经是极限了吗？让我们再看看题目。

题目说：字符串只包含小写字母。小写字母一共只有 26 个！

这样一来，生成签名的时间从 O(k log k) 降到了 O(k)，因为只是简单地统计 26 个字母出现的次数，完全不需要排序。

实现上的一个小细节： 在 Java 中，数组的 hashCode 是基于引用而非内容的，所以不能直接把 int[] 当作 HashMap 的 key。我们需要把计数编码成字符串，比如用 # 分隔每个计数：”1#1#1#0#0#…”。在 Python 中就好办多了——我们可以用元组 (1, 1, 1, 0, 0, ...) 作为 key，元组是不可变的，天然可以做字典键。

小结

从暴力到最优，我们做对了三件事：

  1. 提取不变量：找到异位词的”共同签名”
  2. 用空间换时间：借助哈希表避免重复比较
  3. 利用数据特性：26 个字母的范围使计数代替排序成为可能

复杂度分析

版本 时间复杂度 空间复杂度 说明
暴力法 O(n² × k log k) O(n × k) 两两比较 + 排序判等，n 个字符串平均长度 k
排序哈希法 O(n × k log k) O(n × k) 排序作为 key，哈希表分组
计数哈希法 O(n × k) O(n × k) 字符计数代替排序，利用 26 字母的限制

其中 n 是字符串数组的长度，k 是字符串的最大长度。

关于空间复杂度的补充说明： 三者都需要存储最终结果（O(n × k)），这部分是不可避免的。额外空间主要来自哈希表的 key 存储——排序法每个 key 是排序后的字符串（长度 k），计数法每个 key 是固定长度的编码串（长度与 k 无关），所以计数法在空间上也略微占优。

延伸思考

这道题让我们学到了什么？

1. 等价关系与等价类

字母异位词本质上定义了一种等价关系。判断两个元素是否等价的通用思路是：找到它们的规范形式（canonical form），规范形式相同即等价。

这道题中，排序后的字符串和字符计数数组都是规范形式。很多看似不同的问题其实都在做同一件事——为等价类找一个唯一标识：

  • 判断两棵树是否结构相同 -> 树的序列化
  • 判断两个单词是否属于同一变位词族 -> 字符计数 / 排序
  • 判断两个句子是否语义相近 -> 语义向量

2. 哈希表不只是用来”查找”的

哈希表最常见的用途是 O(1) 的查询，但它还有一个强大的功能：归类。以某种特征值为 key 把数据分门别类存储，最后按类别输出。这种”分组聚合”的模式在算法题中非常常见——比如”按频率排序”、”按位与结果分组”等。

3. 利用问题中的约束条件

“字符串只包含小写字母”——这个约束不是随便给的。它让我们可以用固定大小的数组（int[26]）来计数，把生成签名的时间从 O(k log k) 降到了 O(k)。

在实际工程中，约束往往意味着优化的机会：数据范围小，优先考虑计数；数据有序，考虑双指针或二分；数据规模大，考虑哈希或分治。约束不是限制，而是线索。

下次遇到类似问题优先观察什么？

当你拿到一道”分类”或”分组”的题目时，可以沿着这个思考路径走一遍：

  1. 分组的依据是什么？ -> 找到”组内共同、组间不同”的特征
  2. 这个特征能否用哈希表建模？ -> key 是什么？value 是什么？
  3. 如何高效地计算这个特征？ -> 能否利用数据范围或特性来优化？
  4. 特征的计算能否更轻量？ -> 排序能否简化成计数？能否预计算或增量计算？

记住一句话：找到不变量，问题就解决了一半。
