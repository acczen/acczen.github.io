最长有效括号

最长有效括号

核心思路

第一反应：暴力枚举

给定一个只包含 '(' 和 ')' 的字符串，找出最长有效（即格式正确且连续）的括号子串的长度。

让我们先抛开技巧，从直觉出发。什么样的括号字符串是有效的？

  • "()" 是有效的
  • "()()" 是有效的，长度为 4
  • "(())" 也是有效的，长度为 4
  • "(()" 不是有效的——左括号多了一个
  • ")()())" 中最长的有效子串是 "()()"，长度为 4

有效括号串有一个关键性质：有效括号串的长度一定是偶数，且从左到右遍历时，任何时候 ‘)’ 的数量都不超过 ‘(‘ 的数量，最终两者数量相等。

那最暴力的做法就是：枚举所有偶数长度的子串，逐个检查它是否是有效的括号串。

这个解法非常直观，但效率惨不忍睹——枚举所有子串是 O(n²)，每次验证又是 O(n)，总体 O(n³)。当 n 达到 3×10⁴ 时，这完全不可行。

那么，我们能不能只遍历一次就找到答案？

关键洞察：连续有效括号串之间有什么关联？

让我们先想一个问题：如果你已经知道某个位置之前的括号情况，能帮助判断当前位置是否在有效括号串中吗？

换句话说，有没有办法利用已经计算过的结果，来避免重复扫描？

这里有两个著名的思路：栈 和 动态规划。它们虽然方式不同，但本质都在尝试回答同一个问题——“当前位置是否为一个有效括号串的结尾？如果是，这个有效串有多长？”

解法一：栈——用栈追踪”未匹配”的左括号

我们先尝试用栈的思路。

回想一下，有效的括号（LeetCode 20）这道题中，我们用一个栈来匹配括号：遇到 ‘(‘ 入栈，遇到 ‘)’ 出栈。检查最终栈是否为空即可。

但这里我们要求的是最长连续有效子串的长度，不仅仅是判断整体是否有效。那栈能帮我们追踪”连续”吗？

不妨想一下：当我们遍历字符串时，栈中始终存放着尚未匹配的左括号的索引。那么关键问题是——什么时候一个有效括号子串结束了？

答案是：当我们遇到一个 ‘)’ 且栈为空时，或者遍历结束时，当前的有效子串就断了。

更精确地说，如果我们用一个变量 start 来记录当前可能的有效子串的起始位置，那么：

  1. 遇到 ‘(‘，将其索引入栈
  2. 遇到 ‘)’：
    • 如果栈为空，说明这个 ‘)’ 没有匹配的左括号，有效子串必然在此处断裂——更新 start 到下一个位置
    • 如果栈不为空，弹出一个 ‘(‘（即与之匹配），此时：
      • 如果栈不为空，当前有效子串长度 = i - stack.peek()（栈顶是上一个未匹配的 ‘(‘，它之后的子串是有效的）
      • 如果栈为空，当前有效子串长度 = i - start + 1

这个思路的巧妙之处在于：栈中保留的是尚未匹配的左括号的索引，栈顶始终是当前有效子串的”左边界”。

这里我们用 -1 作为一个哨兵，代表”有效子串可以从下标 0 开始”。之后每次遇到无法匹配的 ‘)’ 时，也把它作为新的哨兵入栈。

这个解法的时间复杂度是 O(n)，空间复杂度 O(n)。

但，我们能不能省掉这个栈呢？

解法二：动态规划——让每个位置告诉我们它知道多少

换一个角度。如果我们定义 dp[i] 为以 s[i] 结尾的最长有效括号子串的长度，那么最终答案就是 max(dp[i])。

这个定义对我们有什么用？关键在于找出递推关系——dp[i] 如何由之前的状态推导出来？

情况一：s[i] == '('
以 ‘(‘ 结尾的子串不可能是有效的括号串，所以 dp[i] = 0。

情况二：s[i] == ')'
这才是需要认真分析的情况。

  • 子情况 A：s[i-1] == '('
    形如 "...()"。此时 s[i-1] 和 s[i] 直接构成一对，那么：

    即，以 i 结尾的最长有效子串，等于 i-2 位置的结果加上新匹配的 “()”。

  • 子情况 B：s[i-1] == ')'
    形如 "...))"。此时我们需要找到与 s[i] 匹配的左括号。这个左括号应该位于 i - dp[i-1] - 1 的位置——也就是以 i-1 结尾的最长有效子串的前一个位置。

    让我们用一个具体例子来理解：

    对于 i = 3，s[3] = ')'：

    • dp[2] = 2，意味着 s[1..2] = “()” 是有效的
    • 和 s[3] 匹配的左括号应该在 i - dp[2] - 1 = 3 - 2 - 1 = 0，即 s[0] = '('
    • 匹配成功！那么 dp[3] = dp[2] + 2 + dp[0-1]，其中 dp[0-1] 视为 0
    • 所以 dp[3] = 2 + 2 + 0 = 4

解法三：双指针计数——常数空间的最优解

有没有可能把空间降到 O(1)？

让我们回到最原始的直觉：一个有效的括号串中，’(‘ 和 ‘)’ 的数量相等，且在任何前缀中 ‘(‘ 的数量都大于等于 ‘)’ 的数量。

如果我们只统计 left（左括号数）和 right（右括号数），会怎样？

从左到右遍历一遍：

  • 遇到 ‘(‘ 则 left++
  • 遇到 ‘)’ 则 right++
  • 如果 left == right，说明当前子串是有效的，更新答案
  • 如果 right > left，说明出现了多余的右括号，当前子串必然无效，重置 left = right = 0

答案是 0，但实际最长有效子串是 “()”（下标 1-2），长度为 2。

问题出在哪？ 从左边开始，左括号始终多于右括号，left 和 right 永远相等不了——但我们知道 “()” 确实存在。这是因为 left 始终比 right 多，left == right 的条件在遍历过程中从未触发。

那如果我们再从右到左遍历一遍呢？

从右到左遍历一遍：

  • 遇到 ‘)’ 则 right++
  • 遇到 ‘(‘ 则 left++
  • 如果 left == right，更新答案
  • 如果 left > right，说明出现了多余的左括号，重置

为什么需要两遍遍历？ 因为 "(()" 这种”左括号始终偏多”的字符串，单向遍历时 left 永远大于 right，触发不了 left == right；反过来从右向左遍历时，就能捕获到右侧的有效子串。反之，"())" 这种”右括号始终偏多”的字符串，从左向右能捕获，从右向左却不行。所以两遍遍历缺一不可。

这个解法的时间复杂度仍然是 O(n)，但空间复杂度降到了 O(1)。只用了两个计数器变量——这是这道题的最优解。

几种思路的比较

回顾我们的思考过程，可以看到一条清晰的演进路径：

  1. 暴力法——最直接的思路，但 O(n³) 完全无法接受
  2. 栈解法——利用栈追踪未匹配的左括号索引，O(n) 时间 O(n) 空间，直观好写
  3. 动态规划——定义 dp[i] 为以 i 结尾的最长有效括号子串长度，推导递推关系，O(n) 时间 O(n) 空间，需要仔细分析多种情况
  4. 双指针计数——两遍遍历分别处理”左括号偏多”和”右括号偏多”的情况，O(n) 时间 O(1) 空间，最优雅

每种解法都从不同的角度回答了同一个核心问题：如何高效地确定一个括号子串是有效的，以及它有多长？

延伸思考

这道题让我们学到了什么？

1. 括号问题的本质是”平衡性”

括号匹配的核心就是计数器思想——左括号加一，右括号减一，任何时候不能小于零，最终归零。这个原理贯穿了所有合法括号问题的解法。无论题目怎么包装，只要看到括号匹配，第一反应应该是平衡计数。

2. 连续性子问题的通用思路——“以 i 结尾”

动态规划中 dp[i] 定义为”以 i 结尾的最优值”，这是处理连续子数组/子串问题的经典套路。最大子数组和、最长递增子序列、最长回文子串——都用了这个思路。下次遇到”最长连续 xxx”的问题，不妨先想想能不能定义 dp[i]。

3. 两遍遍历消除方向偏差

有些结构（比如括号）具有方向性——“左括号过多”和”右括号过多”是不同的场景。单向遍历只擅长处理其中一种。当我们发现单向遍历总是漏掉某些情况时，反方向遍历一遍往往能补上盲区。这道题的双指针解法就是这一思想的绝佳体现。

下次遇到类似问题优先观察什么？

  • 括号类问题：平衡计数（left/right）、栈匹配、是否需要记录位置
  • 求连续最优解：能否用”以 i 结尾”定义状态？能否用滑动窗口？
  • 空间优化：是否能从 O(n) 降到 O(1)？状态之间的依赖关系是否允许滚动更新或计数代替？
