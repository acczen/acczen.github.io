寻找旋转排序数组中的最小值

寻找旋转排序数组中的最小值

核心思路

暴力解法：遍历一切

假设我们并不知道数组被旋转过，问题就是”在一个数组里找最小值”——这太简单了，遍历一次，记录遇到的最小值即可。

时间复杂度 O(n)，对于这道题而言，能通过，但不够优雅。

但题目既然特意强调了”旋转排序数组”，排序 + 旋转这个条件一定蕴含着更高效的解法。让我们想想。

观察规律：旋转后的数组长什么样？

一个升序排序数组 [0,1,2,4,5,6,7] 在某个未知点旋转后，变成了两段升序子数组的拼接：

关键性质：

  • 左段的所有元素 都大于 右段的所有元素（因为最小值在右段开头）
  • 两段内部各自保持升序

最小值恰好就是右段的第一个元素，也就是”旋转点”。

如何高效找到旋转点？

让我们问自己一个问题：如果随便选一个位置 mid，我们能否判断它落在左段还是右段？

答案是肯定的——拿 nums[mid] 与数组的最后一个元素 nums[right] 比较：

  • 如果 nums[mid] > nums[right]，说明 mid 在左段（较大的那一段），旋转点在右侧，我们收缩左边界：left = mid + 1
  • 如果 nums[mid] < nums[right]，说明 mid 在右段（较小的那一段），旋转点在左侧（包括 mid 自己也可能就是最小值），我们收缩右边界：right = mid

等等，为什么和 nums[right] 比，而不是和 nums[left] 比？

不妨试一下和 nums[left] 比较：

  • 如果 nums[mid] > nums[left]，我们不能确定 mid 在左段——因为当数组没有旋转（或旋转后刚好回到原位）时，整个数组都是升序的，nums[mid] > nums[left] 可能意味着 mid 仍在左段，也可能意味着整个数组本来就是有序的，最小值在最左边。
  • 相比之下，nums[right] 是更可靠的参照物：在没有旋转的数组中 nums[mid] < nums[right] 恒成立；在旋转后的数组中，右段的最大值不会超过左段的最小值，所以 nums[mid] 与 nums[right] 的大小关系能唯一确定 mid 所在的段。

这个比较逻辑还有一个隐秘的优雅之处：当 nums[mid] == nums[right] 时怎么办？题目说明数组中的元素 互不相同（distinct），所以等号永远不会出现，我们无需处理。如果题目允许重复（如第 154 题），那才是真正需要小心的地方。

最终算法描述

  1. 初始化 left = 0，right = nums.length - 1
  2. 当 left < right 时循环：
    • 计算 mid = left + (right - left) / 2
    • 如果 nums[mid] > nums[right]，说明最小值在右半段：left = mid + 1
    • 否则，最小值在左半段（含 mid）：right = mid
  3. 循环结束时 left == right，nums[left] 即为最小值

这个算法每次将搜索范围缩小一半，时间复杂度 O(log n)。

延伸思考

这道题让我们学到了什么？

二分查找的灵魂不在于”有序”，而在于”单调性”。 很多同学一提到二分查找，脑海里浮现的就是”在一个排好序的数组里找目标值”。但旋转排序数组教会我们：二分查找真正需要的是 能够根据一个条件排除掉一半的搜索空间。

这道题里，整个数组不是全局有序的，但我们可以通过 nums[mid] 与 nums[right] 的比较来判断当前在哪一段，从而决定去哪一半继续搜索——这就是二分查找更本质的思维方式。

下次遇到类似问题优先观察什么？

  • 数组的局部有序性：虽然不是全局有序，但旋转后两段各自有序，这种”分段有序”通常可以用二分来做文章
  • 找一个”锚点”来判定位置：nums[right] 就是这道题的锚点。类似地，在搜索旋转排序数组（第 33 题）中，我们会用 nums[left] 作为锚点来判断目标值在哪一段
  • 边界收缩习惯：当我们要找旋转点（最小值）时，right = mid 而非 right = mid - 1，因为 mid 自己可能就是答案。这个细节决定了二分查找的正确性
