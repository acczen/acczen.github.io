LRU 缓存

核心思路

先想想：如果让我们设计一个缓存，最朴素的做法是什么？

缓存的核心功能无非两点：存和取。如果不考虑”淘汰”策略，最简单的实现就是用数组或列表存下所有键值对。

get(key)：遍历整个数组去找 key，找到了返回 value，没找到返回 -1。
put(key, value)：先遍历找 key，找到了更新 value；没找到就追加到末尾；如果容量超了，就删掉第一个元素。

看起来很简单，对吧？但我们马上会意识到问题——get 和 put 都是 O(n) 的。题目要求 O(1)，这个版本显然过不了关。

O(1) 的 get 让我们想到什么？

毫无疑问是哈希表（HashMap）。用哈希表存 key → value，get 就能做到 O(1) 了，put 在 key 已存在时也能做到 O(1) 更新。

但新的问题来了：当缓存满了需要淘汰时，我们淘汰谁？

题目说淘汰”最久未使用”的那个。可哈希表不维护任何顺序信息，我们根本不知道哪个 key 是什么时候被访问的。单靠哈希表，当容量满了，我们不知道踢谁出门——这不行。

所以问题转化成：我们需要同时维护”顺序”和”快速查找”

什么东西能维护顺序？链表。

不妨想一下：每次访问一个 key，就把它移到链表的头部。这样一来，链表的尾部永远是最久未使用的那个元素。当缓存满了，直接删掉链表尾部的节点就好了。

但链表的一个致命问题是：查找是 O(n) 的。我们需要 O(1) 地知道某个 key 在链表中的哪个位置，这样才能快速把它移到头部。

于是，一个自然的想法诞生了：

用哈希表存 key → 链表节点，用链表维护访问顺序。

这样：

  • get 时：从哈希表拿到节点，O(1)；然后把节点移到链表头部，O(1)。
  • put 时：如果 key 已存在，更新值并移到头部；如果不存在，创建新节点加到头部，然后检查容量是否超了，超了就删掉尾部节点（同时从哈希表里删除）。

链表中的”移节点到头部”真的是 O(1) 吗？

这里有一个非常重要的细节：如果我们用的是单向链表，要把一个中间节点移到头部，我们需要知道它的前驱节点是谁——而单向链表找前驱需要 O(n) 遍历。

所以我们必须用双向链表。每个节点既有 next 指针，也有 prev 指针。这样，当我们有了一个节点的引用，就可以在 O(1) 时间内把它从链表中”拆”下来，再插到头部。

为了方便处理边界情况（在头部插入、从尾部删除），我们还可以在链表中使用虚拟头节点（dummy head）和虚拟尾节点（dummy tail），这样就不用处理各种 null 的边界条件了。

梳理一下最终的数据结构

  • HashMap<Integer, Node> cache：从 key 到链表节点的映射
  • Node head：虚拟头节点，head.next 是真正的最常使用的节点
  • Node tail：虚拟尾节点，tail.prev 是真正的最久未使用的节点
  • int capacity：缓存容量
  • int size：当前缓存大小

核心辅助方法：

  1. addToHead(Node node)：将节点插入到虚拟头节点的后面
  2. removeNode(Node node)：将节点从链表中移除（前后指针重连）
  3. moveToHead(Node node)：removeNode + addToHead
  4. removeTail()：移除并返回尾部的真实节点（用于淘汰）

把流程图想通

get(key)：

  1. 从 HashMap 中找 key → 如果不存在，返回 -1
  2. 如果存在，拿到 node，调用 moveToHead(node)，返回 node.value

put(key, value)：

  1. 从 HashMap 中找 key
  2. 如果存在：更新 node.value，调用 moveToHead(node)
  3. 如果不存在：创建新节点，加到头部，放入 HashMap，size++
  4. 如果 size > capacity：调用 removeTail()，从 HashMap 中删除该节点的 key，size–

每一步都是 O(1)。

版本 时间复杂度 空间复杂度 说明
暴力数组 get O(n), put O(n) O(capacity) 每次操作都要遍历查找，淘汰时涉及数组元素移动
哈希表 + 双向链表 get O(1), put O(1) O(capacity) 哈希表提供 O(1) 查找，双向链表提供 O(1) 插入和删除
LinkedHashMap（Java） get O(1), put O(1) O(capacity) Java 内置实现，底层同样是哈希表加双向链表

为什么必须是双向链表？因为我们需要在 O(1) 时间内将任意节点移动到头部。单向链表无法 O(1) 获取前驱节点，也就无法 O(1) 完成”拆下再插入”的操作。

延伸思考

这道题让我们学到了什么？

1. 数据结构的组合拳

没有一个现成的数据结构能同时满足”O(1) 查找”和”O(1) 维护顺序”这两个需求。但把哈希表和双向链表组合起来，两者各司其职，问题迎刃而解。”组合”本身就是一种重要的设计思路——当单一结构无法满足所有约束时，不妨想想能否用两种结构各负责一部分。

2. 从约束反推设计

O(1) 的 get → 哈希表。
O(1) 的 put（含淘汰）→ 需要顺序结构。
顺序结构 + O(1) 任意位置删除 → 双向链表。
每一个设计约束都在帮我们缩小选择范围，而不是限制思路。

3. 虚拟头尾节点的妙用

双向链表中，利用虚拟头尾节点可以避免大量的 null 判断，代码更简洁，边界条件更少。这是一种通用技巧，在很多链表问题中都适用，不妨记住它。

下次遇到类似问题优先观察什么？

  • 时间复杂度的要求是最大的提示：O(1) 的 lookup 指向哈希表，O(1) 的插入 / 删除指向链表。
  • “最近”、”顺序”、”淘汰”这些关键词往往暗示需要维护一个有序的数据结构。
  • 如果需要在有序结构中快速定位元素，HashMap<Key, ListNode> 的模式值得牢记。

还有哪些扩展？

  • LFU 缓存（LeetCode 460）：不仅要淘汰最久未使用的，还要考虑”使用频率”。相比 LRU 多了一个频率维度，需要两层哈希表或更复杂的结构。
  • Java 的 LinkedHashMap 底层正是用我们手写的这种结构实现的，它的 removeEldestEntry 方法可以直接用来实现 LRU 缓存，只需重写该方法即可一行代码完成淘汰策略。
  • Redis 的缓存淘汰：Redis 支持多种淘汰策略，LRU 是其中之一。但 Redis 的近似 LRU 算法做了一些采样优化，并不是严格的全量 LRU——因为严格 LRU 需要维护全局链表，在大量数据下内存开销过高。
