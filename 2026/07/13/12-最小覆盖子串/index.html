LeetCode 76. 最小覆盖子串（minimum-window-substring）

题目

LeetCode 76. minimum-window-substring

给你一个字符串 s、一个字符串 t。返回 s 中涵盖 t 所有字符的最小子串。如果 s 中不存在涵盖 t 所有字符的子串，则返回空字符串 ""。

注意：

  • 对于 t 中重复字符，我们寻找的子字符串中该字符数量必须不少于 t 中该字符数量。
  • 如果 s 中存在这样的子串，我们保证它是唯一的答案。

核心思路

拿到这道题，我们先别急着想高级技巧。如果你面前只有纸和笔，你会怎么找？

想象一下：你用两根手指指着字符串 s，一根指开头，一根结尾，中间的部分就是当前的”窗口”。要判断这个窗口是否覆盖了 t，你就得数一数窗口里每种字符各出现了几次，然后和 t 需要的数量对比。

这就是最朴素的暴力思路：枚举所有的子串（$O(n^2)$ 种可能），每个子串都和 t 比对一次（$O(n)$），总复杂度 $O(n^3)$。显然在 $s$ 长度达到 $10^5$ 级别时会直接超时。

那么，瓶颈在哪？能优化哪个环节？

第一个观察：我们需要为每个子串都重新统计一遍字符吗？完全没必要。如果你把右指针往右挪一格，窗口只多了一个新字符，增量更新即可，不需要从头数。这提示我们——“滑动窗口”天然适合这类问题。

第二个观察：我们真的需要枚举所有子串吗？当窗口已经覆盖了 t，你再往右扩只会让窗口更长，不可能变短——所以此时应该收缩左边界，尝试找到一个更短的合法窗口。这和”找满足某条件的最短连续子数组”是同一个套路。

于是思路就清晰了：

  1. 右指针 right 不断向右扩张，把新字符纳入窗口。
  2. 每当窗口覆盖了 t（即窗口内每种字符数量都满足要求），就开始移动左指针 left 向右收缩。
  3. 每次收缩到”刚好还能覆盖”的临界状态时，记录此时的窗口长度。如果比之前记录的更短，就更新答案。
  4. 收缩到不能再收缩（窗口不满足覆盖条件）时，继续扩张右指针，重复上述过程。

这就是经典的滑动窗口框架，时间复杂度直接降到 $O(n)$。

解法：滑动窗口 + 双哈希表

为什么用哈希表？

t 中的字符不仅有种类要求，还有数量要求——比如 t = "AAB"，窗口里至少要有两个 'A' 和一个 'B'。哈希表（或长度为 128 的数组，因为 ASCII 字符集就 128 个）能高效地记录”还需要多少”以及”当前有多少”。

我们维护两个哈希表（或数组）：

结构 含义
need t 中每种字符各需要多少个
window 当前窗口中每种字符各有多少个

另外用一个整数 valid 表示已经满足数量要求的字符种类数。当 valid == need.size()（即 need 中每种字符的数量都被满足），说明当前窗口是一个”覆盖子串”。

为什么不直接比对整个哈希表？

每次都遍历两个哈希表判断是否覆盖，虽然也是常数时间（最多 52 个字母或 128 个 ASCII 字符），但用 valid 计数器可以 $O(1)$ 完成判断，代码也更简洁——这是一种常见的”用计数器替代全量比对”的技巧。

执行流程

  1. 统计 t 中的字符频率，填入 need。
  2. 初始化 left = 0，right = 0，valid = 0，以及记录最小窗口的 start 和 len（初始化为一个很大的值）。
  3. right 从左到右遍历 s：
    • 将 s[right] 加入窗口。如果该字符在 need 中，更新 window，并检查是否刚好满足了 need 对该字符的数量要求——如果刚好满足，valid++。
    • 内层循环：当 valid == need.size() 时，说明窗口合法，尝试收缩：
      • 如果当前窗口长度更短，更新 start 和 len。
      • 将 s[left] 移出窗口。如果该字符在 need 中，更新 window，并检查是否导致该字符不再满足要求——如果不再满足，valid--。
      • left++，继续尝试收缩，直到窗口不再合法。
  4. 遍历结束后，如果 len 仍为初始的大值，返回 ""；否则返回 s.substring(start, start + len)。

延伸思考

学到了什么？

滑动窗口解决”最短/最长覆盖子串/子数组”类问题的核心套路是：右指针无脑扩，左指针有条件收，用计数器（valid）替代每次全量比对来 $O(1)$ 判断窗口是否合法。这道题是滑动窗口题型中的”母题”，吃透它，LC 3（无重复字符的最长子串）、LC 438（找到字符串中所有字母异位词）、LC 567（字符串的排列）都能用同一套模板解决。

下次遇到类似问题，你会优先观察什么？

先问自己三个问题：① 是否要求”连续”（子串/子数组）？如果是，滑动窗口大概率是正解；② 窗口扩张和收缩的条件分别是什么？用”需求”vs”拥有”的差值来定义；③ 能否用计数器替代全量扫描来判断条件？这三问问完，模板一套，基本就通了。
