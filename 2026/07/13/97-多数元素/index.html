多数元素

多数元素

核心思路

LeetCode 169 题「多数元素」的要求非常直白：给定一个大小为 n 的数组 nums，找出其中的多数元素，也就是出现次数大于 ⌊n/2⌋ 的元素。题目还贴心地告诉了我们一个性质：数组非空，并且给定的数组总是存在多数元素。

让我们从最直觉的方法开始，一步步推演。

暴力法 —— 逐个计数

最朴素的想法是什么？既然要找一个出现次数超过半数的元素，那我们干脆遍历数组，对每个元素都统计一下它在整个数组里出现了多少次，如果某个元素的计数超过了 n/2，就返回它。

这样做的代价是 O(n²) 的时间复杂度——对于每个元素都需要再扫一遍全数组。当 n 很大时，这个代价是不可接受的。

有没有办法只遍历一次就完成统计？

哈希表 —— 空间换时间

我们真正需要的信息是「每个元素的出现次数」。如果能在一个 pass 中边遍历边记录，就能避免内层循环。这时候自然会想到哈希表。

遍历数组，用哈希表维护每个元素的计数：如果某个元素的计数超过了 n/2，立即返回。

时间复杂度降到了 O(n)，但空间复杂度是 O(n)。

能否连额外空间都省掉？

排序法 —— 利用位置的确定性

多数元素有一个非常有趣的性质：如果我们将数组排序，那么下标 n/2 处的元素一定就是多数元素。

为什么？因为多数元素占据了超过一半的位置，排序后它必然会覆盖数组的中间位置。就像在一排座位中，如果某类人占了超过一半的座位，不管怎么排列，最中间的那个座位一定属于他们。

这样我们得到了 O(n log n) 的时间复杂度和 O(1) 的空间复杂度（不考虑排序的栈空间）。

不过，排序本身带来的 O(n log n) 时间仍然不是最优的——题目只要求找到多数元素，并不需要对整个数组有序。把整个数组排序是否有点「杀鸡用牛刀」的感觉？

能不能在 O(n) 时间内，只用 O(1) 空间得出答案？

Boyer-Moore 投票算法 —— 抵消中的真相

这是本题最精妙的解法。

让我们换个角度思考：多数元素出现的次数比其它所有元素出现的次数之和还要多。

想象一个场景：一场选举中，多数派候选人获得了超过半数的选票。如果我们让每一张多数派选票和一张非多数派选票互相抵消，最后剩下的那张选票一定属于多数派。

具体算法如下：

  • 维护一个候选元素 candidate 和它的计票 count。
  • 遍历数组：
    • 如果 count == 0，将当前元素设为新的 candidate。
    • 如果当前元素等于 candidate，count++，否则 count--。

遍历结束后，candidate 就是多数元素。

为什么这个算法是正确的？因为多数元素的数量比其他所有元素的总和还多，所以在「一票对一票」的抵消战中，它最终一定会胜出。

这个算法的精妙之处在于它没有使用任何额外空间来存储频次信息，仅仅通过一个计数器就完成了多数元素的筛选。

延伸思考

这道题让我们学到了什么？

  • 当问题中出现了「超过半数」这个条件时，一定要意识到它隐含着一个强大的不等式：多数元素的个数 > 其他元素的个数之和。这个不等式的直接推论就是 Boyer-Moore 投票算法——通过计数抵消来滤出多数元素。
  • 对于「寻找出现次数最多的元素」这一类问题，思考路径可以遵循：暴力计数 → 哈希表优化 → 利用题目特殊条件进一步优化。每一步都追问自己是否需要全部信息（排序、哈希表）还是只需要部分信息（投票计数器）。
  • 有些题目表面上是统计问题，但结合「超过半数」这个额外约束后，往往存在远比通用统计方法更巧妙的解法。

下次遇到类似问题优先观察什么？

  • 题目中是否有「超过半数」「超过三分之一」之类的比例约束？
  • 是否可以不依赖额外空间，利用元素之间的「抵消」关系？
  • 如果题目变成「超过 n/3 的元素」，Boyer-Moore 投票算法能否扩展？（答案是肯定的——我们可以维护两个候选人和两个计数器。）

最后，当我们看到一个 $O(n)$ 时间、$O(1)$ 空间的解法时，不妨停下来品味一下：它不是魔法，而是对问题约束的深度利用。多数元素的「多数」二字，就是整个题目中最关键的线索。
