而且还有一个烦恼：题目要求结果中不能包含重复的三元组。我们枚举出来的结果里，[-1, 0, 1] 和 [0, 1, -1] 在题目看来是同一个东西。如何高效地去重，也不得不额外花心思。

那么问题来了——有没有办法减少不必要的枚举？能不能把三层循环降下来？

从两数之和找灵感

LeetCode 的第一道题就是”两数之和”。那道题我们是怎么做的？用哈希表把查找 partner 的时间从 O(n) 降到了 O(1)。

但是这仍然有一个小遗憾：我们用了额外空间。而且如果数组是乱序的，去重依然有些麻烦——[nums[i], nums[j], complement] 三个数需要排序后才能判断是否重复。

能不能既不用额外空间，又能方便地去重呢？

排序 + 双指针——柳暗花明

让我们换个角度思考：如果数组是排好序的，会带来什么好处？

第一，重复的元素会相邻，去重变得非常容易——跳过相邻的相同值即可。
第二，在一个有序数组中寻找两数之和，我们可以用双指针，而不需要哈希表。

具体做法是：

  1. 先将数组排序
  2. 固定一个数 nums[i]，然后用两个指针分别指向 i+1 和末尾
  3. 根据 nums[l] + nums[r] 与 -nums[i] 的比较结果，决定指针的移动方向：
    • 和等于 target → 记录答案，移动左右指针（同时跳过重复）
    • 和小于 target → 左指针右移，增大总和
    • 和大于 target → 右指针左移，减小总和

这个过程就像在有序数组中用二分的思想搜索——每次比较都能排除一大片区域。

为什么双指针不会漏解？

不妨想一个更简单的问题：在一个有序数组中，找到所有和为 target 的不重复数对。双指针为什么能把所有数对都找全？

因为数组是有序的，当我们固定左指针时，右指针在任何时刻指向的都是”当前左指针下，能凑出 target 的最大可能位置”。如果 nums[l] + nums[r] > target，那么对于这个 l，所有更大的 r 值都会让和更大，更不可能等于 target，所以只能 r–。反之，如果 nums[l] + nums[r] < target，那么对于这个 r，所有更小的 l 都会让和更小，所以只能 l++。

双指针的本质，是在一个二维搜索空间中沿着一条”等值线”前进，每个指针最多移动 n 次，所以总体是 O(n)。加上外层循环的 O(n)，整体就是 O(n²)。

去重的细节

去重是这道题的一个关键点。排序之后，我们可以在三个位置跳过重复：

  1. 外层循环跳过重复的 nums[i]：如果 nums[i] == nums[i-1]，直接跳过本轮，因为以当前值作为第一个数的所有解，都已经在上一个相同值的轮次中被找到了。
  2. 找到一个解后，跳过重复的 nums[l]：左指针向右移动，直到遇到不同的值。
  3. 找到一個解后，跳过重复的 nums[r]：右指针向左移动，直到遇到不同的值。

剪枝优化

在外层循环中，如果 nums[i] > 0，可以直接 break。因为数组已经排好序，最左边的数都大于 0 了，后面所有的数更大，三数之和不可能为 0。

复杂度分析

版本 时间复杂度 空间复杂度 说明
暴力枚举 $$O(n^3)$$ $$O(n)$$ 三重循环枚举所有三元组，用 Set 去重
排序 + 双指针 $$O(n^2)$$ $$O(\log n) \sim O(n)$$ 排序的额外空间取决于语言实现（Java 的 Arrays.sort() 对基本类型使用双轴快排，空间 $$O(\log n)$$），双指针本身只使用常数额外空间

时间复杂度 $$O(n^2)$$ 是如何计算的？
外层循环遍历 n 个元素，内层双指针遍历剩余元素——每轮双指针最多移动 n 次。最坏情况下需要执行 O(n) 轮，所以总复杂度是 O(n²)。

延伸思考

这道题让我们学到了什么？

第一，降维思想。三数之和通过固定一个元素，降维成两数之和。遇到”K 数之和”系列问题（Two Sum、Three Sum、Four Sum），这个思路可以一以贯之——从 K 个数逐步降到 2 个数。

第二，排序的威力。很多时候我们觉得排序会”破坏原始信息”，不愿意排序。但在查找类问题中，排序带来的有序性往往利远大于弊——它让双指针成为可能，也让去重变得异常简单。

第三，双指针的本质。双指针不是”两个指针在数组上瞎逛”，而是在有序空间中沿着单调方向搜索，每次移动都排除一批不可能的解。这是一种”有序性 + 单调性”的组合技巧。

下次遇到类似问题优先观察什么？

  • 问题能否通过固定部分元素来降维？ 如果是”K 数之和”，先想固定 K-2 个数，剩下两个数用双指针。
  • 原数组是否可以排序？ 如果题目不要求保留原始顺序，排序几乎总是值得考虑的预处理步骤。
  • 数据规模有多大？ O(n²) 对于 n=3000 大约是 900 万次运算，在合理范围内。如果 n 达到 10⁵，就需要另寻它路。
  • 重复元素如何处理？ 排序后相邻比较是最直接的去重手段。

三数之和是一道经典的”面试必考题”。它看起来只是两数之和的升级版，但背后蕴含的降维、排序预处理、双指针逼近等思想，是很多复杂题目的基础。掌握了这道题，四数之和、最接近的三数之和等问题也就不难应对了。
