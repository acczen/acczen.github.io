LeetCode 15. 三数之和（3sum）

题目

LeetCode 15. 3sum

给你一个整数数组 nums，判断是否存在三元组 [nums[i], nums[j], nums[k]] 满足 i != j、i != k 且 j != k，同时还满足 nums[i] + nums[j] + nums[k] == 0。请你返回所有和为 0 且不重复的三元组。

注意：答案中不可以包含重复的三元组。

核心思路

拿到这道题，我们最容易想到的是什么？

暴力枚举。三层循环，把所有 (i, j, k) 组合试一遍，和为 0 就收进结果。这完全正确，时间复杂度是 $O(n^3)$。

那么问题来了：暴力到底哪里不够好？如果把 $n = 3000$ 带进去，$n^3 \approx 2.7 \times 10^{10}$，这个量级在任何判题系统里都是灾难。瓶颈在哪？

瓶颈在于：我们在没有”方向感”地搜索。每层循环都在盲目尝试，没有利用任何数值之间的大小关系。如果数组是有序的，我们能做什么？

想一想：在有序数组中，给定一个目标值，如何快速找到两个数之和等于它？这就是经典的两数之和 II——左右双指针相向而行。当前和偏小就左指针右移（需要更大的数），偏大就右指针左移（需要更小的数）。一次遍历，$O(n)$ 搞定。

那么三数之和呢？我们只需要固定一个数，把剩下的变成两数之和问题！

这就是从暴力到优化的完整思考链：

  1. 排序，让数组拥有”方向感”。
  2. 固定第一个数 nums[i]，把问题转化为在 [i+1, n-1] 范围内寻找两数之和等于 -nums[i]。
  3. 用双指针在 $O(n)$ 时间内完成第二步。
  4. 总复杂度：排序 $O(n \log n)$ + 外层 $O(n) \times$ 内层双指针 $O(n) = O(n^2)$。

还有一个”麻烦”：去重。题目要求答案不包含重复三元组。重复来自哪里？两种情况：固定的第一个数重复（比如 [-1, ...] 之后又遇到 -1），以及双指针移动时找到的左右值重复。两次去重都不能漏，否则答案会出现重复三元组。

停下来想一想：为什么排序不会破坏正确性？因为三元组 (a, b, c) 的和与顺序无关——我们只是在找组合，不是在找下标。排序只是帮我们”整理了一下桌面”，并不改变问题的本质。

解法：排序 + 双指针

我们把这个过程拆开来看。

第一步：排序

将 nums 升序排列。排完之后，数组从左到右递增。这个性质是双指针能够”指哪打哪”的前提——我们知道左指针右移一定会让和变大，右指针左移一定会让和变小。

第二步：固定第一个数，转化问题

遍历 i 从 0 到 n-3（至少给后面留两个位置）。对于每个 nums[i]，我们需要在 [i+1, n-1] 中找两个数，它们的和等于 -nums[i]。

这里有一个剪枝：如果 nums[i] > 0，由于数组已排序，它后面的所有数都 $\ge$ nums[i] > 0，三个正数之和不可能为 0。直接结束循环。

第三步：双指针扫描

设 left = i + 1，right = n - 1。计算 sum = nums[i] + nums[left] + nums[right]：

  • 若 sum == 0：找到一个三元组，记录。然后 left++、right--，并跳过重复值。
  • 若 sum < 0：当前和太小，需要一个更大的数 → left++。
  • 若 sum > 0：当前和太大，需要一个更小的数 → right--。

双指针的核心直觉是：我们同时维护了”上界”和”下界”。左右指针把候选区间越收越窄，每次排除的不只是一个数，而是一整条不可能的方向。

第四步：去重

两处去重，缺一不可：

  1. 外层去重：if (i > 0 && nums[i] == nums[i-1]) continue;——如果当前固定的第一个数和前一个相同，跳过。因为以 nums[i-1] 为第一个数的所有三元组已经在上一轮被找全了。
  2. 内层去重：找到 sum == 0 之后，while (left < right && nums[left] == nums[left+1]) left++; 和 while (left < right && nums[right] == nums[right-1]) right--;——跳过相邻重复的左右值，避免同一个三元组被多次加入。

注意：内层去重必须放在找到答案之后，而不能放在 sum < 0 或 sum > 0 的分支里随便跳过。为什么？因为跳过重复是为了防止同一个三元组被重复记录，而不是为了加速搜索。如果在没找到答案时就乱跳，可能会漏掉某些组合。

为什么不用哈希表？

你可能在想：两数之和不是可以用哈希表 $O(n)$ 做吗？三数之和能不能也哈希？

可以，但去重会变得很痛苦。哈希表不关心顺序，面对 [-1, 0, 1, -1, 0, 1] 这样的输入，你必须额外维护一个 Set 来去重，而且判断”这个三元组是否已经出现过”本身也有开销。排序 + 双指针天然利用有序性去重，代码更简洁，而且没有哈希冲突的隐患。

复杂度分析

维度 分析
时间复杂度 $O(n^2)$ — 排序 $O(n \log n)$，外层循环 $O(n)$，内层双指针每次 $O(n)$，总体由内层主导为 $O(n^2)$
空间复杂度 $O(1)$ — 不考虑存储结果的列表，只用了几个指针变量。排序的栈空间为 $O(\log n)$，在一些分析中也视为常数级别

延伸思考

学到了什么？

排序是很多数组问题”降维打击”的第一步。一旦数组有序，搜索就从盲目扫荡变成了有方向的逼近，双指针、二分这些武器才能派上用场。三数之和的本质，就是把一个新问题降解为一个已经会做的旧问题（两数之和），这正是算法设计中”化归”思想的体现。

下次遇到类似问题，你会优先观察什么？

先问自己：如果我把数组排序，能不能利用单调性把 $O(n^k)$ 的暴力搜索变成 $O(n^{k-1})$ 的定向搜索？ 这几乎是所有 $k$-sum 类问题的统一套路——从三数之和到四数之和，再到更一般的 $k$ 数之和，排序 + 双指针（或递归降解）是那个不变的骨架。
