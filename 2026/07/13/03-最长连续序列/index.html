LeetCode 128. 最长连续序列（longest-consecutive-sequence）

题目

LeetCode 128. longest-consecutive-sequence

给定一个未排序的整数数组 nums，找出数字连续的最长序列（不要求序列元素在原数组中连续）的长度。请你设计并实现时间复杂度为 $O(n)$ 的算法解决此问题。

提示：

  • $0 \le nums.length \le 10^5$
  • $-10^9 \le nums[i] \le 10^9$

核心思路

拿到这道题，我们最容易想到的做法是什么？

排序。把数组排好序，然后一趟扫描，看相邻元素是否连续，就能轻松找出最长连续段。这个思路清晰直接，时间复杂度 $O(n \log n)$，空间复杂度 $O(1)$（如果允许原地排序）。

那么问题来了：题目为什么非要卡 $O(n)$？排序差在哪？

因为排序的 $O(n \log n)$ 在面对 $n = 10^5$ 时固然能跑，但如果我们能把时间压到 $O(n)$，就说明我们真正”理解”了连续性的本质——连续意味着 x 和 x+1 之间只差 1，我们不需要”全局排序”来获得这个信息，只需要能快速查询某个数是否存在。

这就引出了核心思考方向：能不能换一种数据结构，让我们能以 $O(1)$ 的代价判断”当前数的下一个数是否在数组中”？

答案呼之欲出——哈希集合（HashSet）。

把所有数扔进哈希集合里，查 num + 1 是否存在就是 $O(1)$ 的。现在我们从任意一个数出发，不断 +1 往后找，就能测出一段连续序列的长度。

但是——这样真的 $O(n)$ 吗？如果每个数都向后展开一遍，最坏情况是不是 $O(n^2)$？

对，这正是我们需要面对的挑战。假设数组是 [1,2,3,4,5]，如果我们对 1 展开一次得到了长度 5，然后对 2、3、4、5 又各自展开一次，就做了大量重复工作。

突破口在哪？

我们问自己一个问题：一段连续序列 [x, x+1, x+2, ..., x+k] 中，有哪一个位置是”独特的”——一旦我们从那里出发，就不会有人再从序列中间出发做无用功？

想想看：x 的前驱 x-1 不在数组中。只有起点才具备这个性质。对于 x+1，它的前驱 x 在数组中，所以它不可能是一个”新序列的起点”——它必然已经被 x 覆盖了。

于是策略变为：

只从”前驱不存在”的数开始向后延伸。每一个数如果在某个序列中但不是起点，就跳过。

这样一来，数组中的每个数最多被访问两次：一次在外层循环判断”我是不是起点”，一次在内层 while 中被前驱数”扫过”。总时间复杂度稳稳地落在 $O(n)$。

思考链条回顾

解法：哈希集合 + 贪心起点扫描

这是本题的唯一最优解，没有中间过渡方案。

算法步骤

  1. 构建哈希集合：将 nums 中所有元素加入 HashSet，自动去重。
  2. 遍历集合中的每个数 num：
    • 起点判断：检查 num - 1 是否在集合中。如果在，说明 num 不是一段连续序列的起点，直接跳过。
    • 序列延伸：如果 num - 1 不在集合中，则 num 是起点。从 num 开始不断 +1，用 while 循环向后探测，累加当前序列长度。
    • 更新全局最大值。
  3. 返回最长长度。

为什么不用排序？

方案 时间复杂度 空间复杂度 可行性
排序 + 扫描 $O(n \log n)$ $O(1)$ 可 AC，但不满足题意
哈希 + 起点扫描 $O(n)$ $O(n)$ 最优解

为什么不用并查集（Union Find）？

并查集也能做——把相邻的数合并到同一个连通分量里，最后看哪个分量最大。但并查集需要维护 parent 映射和 size 映射，额外常数较大，且不如哈希集合方案直观。对于这道题，哈希集合方案已经足够简洁高效，没有必要引入更重的数据结构。

复杂度分析

维度 分析
时间复杂度 $O(n)$ — 构建哈希集合 $O(n)$；外层循环遍历每个数，但每个数最多在内层 while 中被访问一次（被访问过的数一定满足 num-1 在集合中，之后会被跳过），所有 while 循环的总执行次数不超过 $n$。哈希查找 $O(1)$，总计 $O(n)$。
空间复杂度 $O(n)$ — 哈希集合存储所有不重复的元素，最坏情况下所有元素都不重复。

代码实现

代码要点提醒

  1. 遍历 set 而不是 nums：set 已经去重，遍历它可以避免对重复元素做无意义的起点判断。虽然遍历 nums 再用 set 做判断在渐进复杂度上一样，但遍历 set 常数更优。
  2. 空数组检查：nums.length == 0 时直接返回 0，Java 中还要检查 nums == null。
  3. num - 1 的判断是灵魂：去掉这行判断，代码逻辑仍然正确，但会退化为 $O(n^2)$——这正是本题区分 $O(n)$ 和 $O(n^2)$ 的关键一行。

延伸思考

学到了什么？

这道题教给我们一个重要的思维模式：用空间换时间不是终点，找到”不必重复计算的起点”才是真正的优化核心。 哈希集合给了我们 $O(1)$ 的查询能力，而”只从起点出发”这条贪心规则，让我们把 $O(n^2)$ 的重复劳动压缩到了 $O(n)$。二者缺一不可。

下次遇到类似问题，你会优先观察什么？

优先观察：“我能不能找到一个性质，只从某个特殊位置开始计算，而跳过所有已经被覆盖的冗余位置？” 这个启发式不仅适用于连续序列问题，也适用于任何需要”枚举区间/子数组/子序列”但想避免重复工作的场景——先思考有没有一个”唯一入口”能够代表整个区间。
