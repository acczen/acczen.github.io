LeetCode 41. 缺失的第一个正数（first-missing-positive）

题目

LeetCode 41. first-missing-positive

给你一个未排序的整数数组 nums，请你找出其中没有出现的最小的正整数。

请你实现时间复杂度为 $O(n)$ 并且只使用常数级别额外空间的解决方案。

核心思路

拿到这道题，我们先从最直接的想法出发。

如果没有任何限制，你会怎么找缺失的第一个正数？

最简单的做法是：把数组排序，然后从头扫描。时间复杂度 $O(n \log n)$，不满足题目要求的 $O(n)$。而且排序改变了原数组，但题目没说不让改。

那能不能进一步优化？我们需要比排序更快，又不想要额外空间。提示你一下——这道题的答案有没有范围限制？

考虑一个长度为 $n$ 的数组。缺失的最小正整数，最大可能的值是多少？

停下来想一想，然后再往下看。

答案：最多是 $n + 1$。为什么呢？想象最极端的情况：数组恰好包含 $1, 2, 3, \dots, n$ 这 $n$ 个正整数，此时缺失的最小正整数是 $n + 1$。只要数组里缺了 $[1, n]$ 中的任何一个数，答案就一定落在 $[1, n]$ 内。

所以，答案的范围被限定在 $[1, n+1]$ 之中。这个结论给了我们什么启发？

一个关键洞察：所有不在 $[1, n]$ 范围内的数，对我们找答案没有任何帮助。 比如负数、0、大于 $n$ 的正数——这些都可以被”忽略”，因为它们不可能成为答案，也不会影响答案。

那么，你可能会想到：能不能利用数组的”下标”做文章？

这正是这道题的灵魂所在。想象一下：如果我们能把每个在 $[1, n]$ 范围内的数 $x$，放到数组的第 $x-1$ 个位置（也就是下标 $x-1$），那么：

  • 下标 $0$ 存的是 $1$（如果 $1$ 存在的话）
  • 下标 $1$ 存的是 $2$（如果 $2$ 存在的话）
  • 下标 $2$ 存的是 $3$（如果 $3$ 存在的话）
  • …

摆完之后，我们从头扫描：第一个下标 $i$ 上存的不是 $i+1$ 的位置，就说明 $i+1$ 缺失了。 如果所有位置都是对的，答案就是 $n+1$。

这个思路叫什么？你有什么感觉？

这就是”原地哈希“（也常被称为”索引作为哈希键”或”循环交换”）。我们利用数组下标来模拟哈希表的”键”，数组元素值充当哈希表的”值”——不需要额外的空间，数组自己就是哈希表。

解法：原地哈希（索引放置）

现在我们把思路落实到具体操作上。

我们需要遍历数组的每个位置，对于当前位置 $i$：

如果 $nums[i]$ 的值在 $[1, n]$ 范围内，且它不在”自己的位置”上（即 $nums[nums[i]-1] \neq nums[i]$），就把它交换到正确的位置去。 交换后继续检查当前位置新换来的值——也就是用一个 while 循环，直到当前位置的值”安分”了为止。

为什么用 while 循环？因为换过来的数可能也是一个”合法”的正整数，也需要被放到正确的位置。每交换一次，我们至少把一个数放到了它的”最终归宿”，所以每个数最多被交换一次——这就保证了总体的 $O(n)$ 时间复杂度。

放置完毕后，再从头扫描一遍：第一个 $nums[i] \neq i+1$ 的位置，答案就是 $i+1$。全部符合则返回 $n+1$。

和备选方案的对比

方法 时间 空间 评价
排序 + 扫描 $O(n \log n)$ $O(1)$ 不满足 $O(n)$ 要求
哈希集合 $O(n)$ $O(n)$ 直观但不满足 $O(1)$ 空间
原地哈希 $O(n)$ $O(1)$ 时间空间双最优
  • 排序的瓶颈在于比较排序的下界就是 $O(n \log n)$，我们需要更聪明的方式跳过排序。
  • 哈希集合很直观：把数都丢进集合，然后从 $1$ 开始往上找。但是需要 $O(n)$ 额外空间——题目卡的就是这个。
  • 原地哈希把数组本身当成哈希表用，一举两得：既规避了排序的时间代价，又省掉了额外的哈希空间。

复杂度分析

维度 分析
时间复杂度 $O(n)$ — 每个数最多被交换一次，while 循环的总体的执行次数不会超过 $n$，加上最后的扫描，总时间线性
空间复杂度 $O(1)$ — 只用到了循环变量和交换用的临时变量，不随输入规模增长

延伸思考

学到了什么？

这道题的核心是”把数组下标当哈希键”——当答案范围被限定在 $[1, n]$ 时，数组本身就足够承载所有可能答案的”存在性”信息。这种用空间换时间（但空间就是输入本身，所以不额外花钱）的逆向思维，是很多数组难题的突破口。

下次遇到类似问题，你会优先观察什么？

遇到”在不使用额外空间的前提下，在数组上做查找 / 去重 / 存在性判断”类问题时，先问自己：“答案的值域是否比数组长度还要小？能不能通过交换或标记，把数组改造成一个自索引的结构？” 这是原地哈希和其变体（如使用正负号标记、交换归位等技巧）的通用解法框架。
