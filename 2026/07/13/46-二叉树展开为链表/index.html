二叉树展开为链表

二叉树展开为链表

核心思路

暴力解法：前序遍历 + 重连

拿到这道题，我们的第一反应通常是：展开后的顺序不就是前序遍历的顺序吗？

那最简单的方法就呼之欲出了——先做一次前序遍历，把所有结点存到一个列表里，然后再遍历这个列表，把每个结点的 left 置为 null，right 指向下一个结点。

这个解法能通过，但存在一个明显的问题：我们用了 O(n) 的额外空间来存列表。题目给的是一个树，我们能不能直接在树上动手脚，不借助额外的存储呢？

不妨想一下：我们要的结果是一棵”歪脖子树”——每个结点只有右孩子。那么能不能一边遍历一边重建？

自问自答：能不能一边遍历一边重构？

让我们设想一下：从根结点开始，如果我们先把左子树拉平成一条链，再把右子树拉平成一条链，然后让左链的尾巴接上右链的头——看起来很有希望。

不过这里有一个细节：左链的尾巴是谁？ 对一棵树做前序遍历，左子树的最后一个结点就是左子树中最右边的那个结点。我们把左链接到根结点的 right 上之后，需要找到左链的末尾，再把右链接上去。

这就引出了递归解法。

解法一：递归后序遍历

思路很清晰：

  1. 递归地将左子树展开为链表
  2. 递归地将右子树展开为链表
  3. 将根结点的 right 指向展开后的左子树，left 置为 null
  4. 沿着新的右链走到末尾，把展开后的右子树接上去

每一步都在做同一件事：把一棵树拍平成一条右链，同时返回这条链的最后一个结点（因为父结点需要知道在哪里接上右子树）。

时间 O(n)，空间 O(h)（递归栈深度，最坏 O(n)）。

但递归解法在树退化成链时递归深度依然可能很大。有没有办法完全不用递归？

解法二：迭代 + 寻找前驱（Morris 思想）

让我们回到刚才那个朴素的问题：展开链表 = 前序遍历。如果我们能在遍历的过程中就完成指针重连，是不是就能做到 O(1) 空间？

前序遍历的顺序是 根 → 左 → 右。对于一个结点 cur：

  • 如果 cur 没有左孩子，那就好办了——不用处理，直接往右走
  • 如果 cur 有左孩子，那么左子树在前序遍历中会紧跟在 cur 之后，而右子树会在整个左子树遍历完之后才轮到

那么关键来了：左子树遍历完之后的最后一个结点是谁？ 是左子树最右边的那个结点（前驱结点）。这个结点遍历完之后，下一个就该轮到右子树了。

所以我们的操作就清晰了：

  1. 找到 cur 左子树中最右侧的结点（前驱）
  2. 把这个前驱的 right 指向 cur.right（把右子树嫁接到左链末尾）
  3. 把 cur.right 指向 cur.left，cur.left 置为 null
  4. cur 移到新的 cur.right，重复上述过程

这个过程不需要额外空间，完全在指针层面完成了重构。

复杂度一览

版本 时间复杂度 空间复杂度 说明
前序遍历+重连 O(n) O(n) 最直观，但用了额外列表
递归后序 O(n) O(h) h 为树高，最坏 O(n)
Morris 迭代 O(n) O(1) 原地修改，最优

延伸思考

这道题让我们学到了什么？

前序遍历与树结构之间的关系。 这道题本质上是在问：你能在不借助额外数组的情况下，按照前序遍历的顺序重新组织树的指针吗？答案是利用前驱结点的”桥梁”作用——左子树遍历完的最后一个结点恰好就是连接左右子树的天然位置。

这也是一种”原地算法“的思维方式：当我们需要按照某种顺序重新组织数据时，先想想能不能通过调整指针本身来达成目标，而不是先复制再重建。

下次遇到类似问题优先观察什么？

如果题目要求原地修改树的结构，优先往这两个方向思考：

  • 递归后序处理：先处理左右子树，再处理当前结点。这是一种自底向上的思维，很适合于”整棵树要变成另一种形态”的问题。
  • 寻找前驱/后继：如果问题涉及到遍历顺序，找前驱或后继结点往往能在原地完成指针重排。这和 Morris 遍历的思想一脉相承。

另外，这道题也提醒我们：不要被”链表”二字迷惑——展开后的结构本质上仍然是一棵二叉树，只不过所有左孩子都为空而已。理解这一点，递归解法的正确性也就一目了然了。
