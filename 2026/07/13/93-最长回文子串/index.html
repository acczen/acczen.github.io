最长回文子串

核心思路

让我们从一个最直观的问题开始：给定一个字符串，怎样找到其中最长的回文子串？

回文的定义很简单——正着读和反着读一样，比如 "aba"、"bb"、"a"。但定义简单，不代表寻找起来也简单。

第一反应：枚举所有子串

最容易想到的方法是：枚举所有可能的子串，对每个子串判断它是不是回文。

判断一个子串是否是回文需要 O(len) 时间，枚举所有子串需要 O(n²) 个，加起来就是 O(n³)。

这个复杂度对于稍长的字符串就已经不可接受了。让我们想想，问题出在哪里？

追问：我们在重复计算什么？

判断 s[i..j] 是否是回文时，我们做了 O(j-i) 次字符比较。但判断 s[i+1..j-1] 时，我们又把中间的部分比较了一遍——大量重叠子问题被反复计算。

如果我们知道了 s[i+1..j-1] 是回文，判断 s[i..j] 是不是回文还需要 O(n) 吗？

不需要。s[i..j] 是回文，当且仅当 s[i] == s[j] 且 s[i+1..j-1] 是回文。这是一个非常优美的递推关系。

这样，我们在 O(1) 时间内就能从短子串递推出长子串。总复杂度 O(n²)。

再追问：空间能省吗？直觉上我们做了什么？

动态规划的 O(n²) 空间能否优化？让我们换个角度思考——我们本质上做的事情是从短回文向长回文生长。那回文是怎么”生长”的呢？

回文有一个几何直觉：它是对称的，有一个中心，向两边均匀扩展。

比如 "ababa"，中心是 'b'（索引 2），向两边扩展："a" → "aba" → "ababa"。偶长度的回文，比如 "abba"，中心在两个 'b' 之间。

我们能不能直接模拟这个”由中心向两边扩展”的过程？

当然可以。对每个可能的中心（共 2n-1 个，n 个字符位置 + n-1 个字符间隙），从中心向两边扩展，记录扩展能够达到的最大长度。

每个中心扩展最多 O(n) 步，总共 2n-1 个中心，总时间复杂度依然是 O(n²)，但空间降到了 O(1)。

这就是中心扩展法——代码比 DP 更简洁，空间开销也更小。它是这道题在面试中最推荐的解法。

还有更优的解法吗？

如果追求理论最优，还有 Manacher 算法，它巧妙地利用了回文的对称性，将时间复杂度降到了 O(n)。它的核心思想是：当我们已经知道一个较大的回文时，它内部的小回文可以利用镜像信息快速确定，而不必从中心重新扩展。

不过 Manacher 算法的实现细节比较精巧，面试中一般不会作为硬性要求。但如果能讲清楚它的思路，会是很好的加分项。

延伸思考

这道题让我们学到了什么？

从定义出发，理解问题的结构比死记解法更重要。

回文的核心结构是”对称”——这个直觉直接引出了中心扩展法。如果我们只停留在 DP 的递推公式上，就错过了问题背后最直观的几何意义。

当我们遇到一个问题时，不妨先问自己三个问题：

  1. 暴力怎么做？——明确问题的底线，确保没有理解偏差。
  2. 重复计算在哪里？——暴力解法中一定有重复劳动，找到它就是优化的起点。DP 通过记忆化消除了判断子串时的重复比较。
  3. 问题的结构特征是什么？——回文的对称性天然适合”从中心生长”而非”从边界枚举”。把思维方向从”检查边界”切换到”拥抱中心”，复杂度不变，但代码更简洁、空间更优。

下次遇到类似问题优先观察什么？

“子串”+”回文/对称”类问题，优先考虑：

  • 能不能从中心向两边扩展？（字符、间隙作为中心）
  • 能不能用双指针从两边向中间夹逼？（验证回文本身）
  • 对称性是否暗示了镜像复用？（Manacher 的思路）

“子串/子数组”类问题，通用的思考链条是：

  • 暴力枚举 → 找重复计算 → DP 或双指针
  • 如果数据范围是 10³ 级别，O(n²) 的 DP 或中心扩展是合理选择
  • 如果数据范围是 10⁵ 级别，需要 O(n) 算法（Manacher、KMP、Z 算法等）

最后，不要忘记回文的最简单形式——单个字符和两个相同字符——它们是一切复杂回文的基础。从最小的回文开始生长，往往比直接判断大回文更自然。这也是中心扩展法比 DP 更贴合直觉的原因。
