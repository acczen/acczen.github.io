编辑距离

核心思路

第一步：理解问题——什么是编辑距离？

LeetCode 72「编辑距离」是这样一道题：给定两个单词 word1 和 word2，请计算将 word1 转换成 word2 所使用的最少操作数。你可以对一个单词执行以下三种操作：

  • 插入一个字符
  • 删除一个字符
  • 替换一个字符

举个例子，把 “horse” 变成 “ros”，最少需要几步？让我们先感性思考一下。

第二步：从最朴素的想法出发——暴力递归

面对两个字符串，我们最直接的想法是逐字符比对。想象一下，我们手中有两个指针 i 和 j，分别指向 word1 和 word2 的当前位置。每走一步，我们面临几种选择？

如果 word1[i] == word2[j]，那太好了，不需要操作，两个指针同时前进。

如果不一样呢？我们就面临三种选择：

  1. 删除 word1[i]——然后 i 前进一位，j 不动
  2. 插入 一个 word2[j] 到 word1 当前位置——然后 j 前进一位，i 不动
  3. 替换 word1[i] 为 word2[j]——然后 i 和 j 同时前进

我们不知道哪个选择最终会带来最少操作，所以只好都试一遍，取最小值。

第三步：停下来想一想——有没有重复计算？

让我们画一下递归树。以 “abc” 和 “def” 为例，dfs(1,1) 可能在多少条路径中被调用？很多。这意味着我们一遍又一遍地在求解同一个子问题。

第四步：从记忆化到递推——能不能换个方向思考？

递归是从大问题推向小问题，再从 base case 一路返回结果。既然我们已经知道 base case 的答案（空串与任意串的编辑距离等于那个串的长度），不妨反过来想：从小问题出发，逐步递推到大问题。

定义 dp[i][j] 表示 word1 的前 i 个字符转换成 word2 的前 j 个字符所需的最少操作数。

  • 当 i = 0 时，word1 是空串，需要插入 j 个字符，所以 dp[0][j] = j
  • 当 j = 0 时，word2 是空串，需要删除 i 个字符，所以 dp[i][0] = i

对于一般情况，考虑 word1 的第 i 个字符和 word2 的第 j 个字符：

情况一：word1[i-1] == word2[j-1]
当前字符已经相等，不需要额外操作，问题规模缩小为 dp[i-1][j-1]。
即：dp[i][j] = dp[i-1][j-1]

情况二：word1[i-1] != word2[j-1]
我们需要在三种操作中选代价最小的：

  1. 删除 word1[i-1]：相当于 word1 的前 i-1 个字符去匹配 word2 的前 j 个字符，然后加上一次删除操作
    dp[i][j] = dp[i-1][j] + 1

  2. 插入 word2[j-1] 到 word1 中：相当于 word1 的前 i 个字符已经匹配了 word2 的前 j-1 个字符，插入一个字符匹配当前位
    dp[i][j] = dp[i][j-1] + 1

  3. 替换 word1[i-1] 为 word2[j-1]：相当于 word1 的前 i-1 个字符匹配 word2 的前 j-1 个字符，然后替换一次
    dp[i][j] = dp[i-1][j-1] + 1

第五步：亲手填一遍表——让抽象变具体

让我们用手算一个小例子 word1 = "cat", word2 = "car" 来感受一下填表过程。

初始表格（dp 维度为 4 x 4）：

先填 base case：第一行和第一列已经填好。

接着填 dp[1][1]：word1[0]='c'，word2[0]='c'，相等，所以 dp[1][1] = dp[0][0] = 0。

dp[1][2]：word1[0]='c'，word2[1]='a'，不等。

  • 删除 dp[0][2] + 1 = 2 + 1 = 3
  • 插入 dp[1][1] + 1 = 0 + 1 = 1
  • 替换 dp[0][1] + 1 = 1 + 1 = 2
    取最小值 1，所以 dp[1][2] = 1。

延伸思考

这道题让我们学到了什么？

编辑距离是二维动态规划的经典入门题。它教会我们一个非常通用的解题框架：

  1. 思考从暴力开始——不要一上来就想 DP 公式。先写出递归，感受问题结构，然后发现重叠子问题，自然引向优化。
  2. 定义状态就是定义子问题——dp[i][j] 的物理含义是”子串间的编辑距离”，这个定义让问题有了”无后效性”：子问题的解不依赖于如何达到这个子问题。
  3. 状态转移来自最后一步的决策——当我们思考 dp[i][j] 时，别盯着开头看，盯着最后一个字符看。两串的最后一个字符要么相等（无需操作），要么不相等（删/插/替选一个）。这个视角让递推关系变得非常自然。

下次遇到类似问题优先观察什么？

遇到字符串匹配、转换类问题时，不妨问自己几个问题：

  • 问题能否分解成更小规模的相同问题？ 如果答案是肯定的，动态规划很可能适用。
  • 状态有哪几个维度？ 两个字符串之间的操作，通常是二维 DP；如果是单个字符串上的操作（如回文子串），有时一维就够了。
  • base case 是什么？ 空串的情况常常是最简单的 base case——从空串到任意串的距离就是那个串的长度。
  • 我们能不能交换两个输入的角色？ 这个问题的三种操作是对称的：删除 word1 的字符等价于在 word2 中插入一个字符。利用对称性可以优化空间。

编辑距离在实际应用中非常广泛——从拼写检查、DNA 序列比对到 Git 的 diff 算法，背后都有它的影子。理解这一题，相当于掌握了一把处理”序列相似度”问题的钥匙。

下次当你遇到”把一个东西变成另一个东西最少需要几步”这类问题时，不妨想想编辑距离的推导过程——从暴力的分支探索，到表格化的递推记录，再到只保留必要信息的空间压缩。思路的进化往往比最终答案更值得回味。
