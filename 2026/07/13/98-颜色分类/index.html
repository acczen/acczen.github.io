核心思路

从暴力解法开始

题目给了我们一个只包含 0、1、2 的数组，让我们原地排序。

最直接的想法是什么？用一个排序算法比如快速排序、归并排序，或者直接调 Arrays.sort() —— 对吧？但是等等，题目暗示了更好的办法。数组里只有三种值，我们真的需要用通用排序算法吗？

不妨想一下：先扫描一遍数出 0、1、2 各有多少个，然后再第二遍把数组依次填回去。这样时间复杂度是 O(n)，空间复杂度 O(1)。这已经比 O(n log n) 的排序好太多了。

但还有一个小遗憾：我们遍历了两次。

那么，能不能一次遍历就完成排序呢？

推动思考：一次遍历的挑战

一次遍历就排好序，意味着我们边走边把元素放到它该去的位置。当我们看到一个 0，应该把它换到数组前面；看到一个 2，应该把它换到数组后面；看到 1，它本就在中间，不动。

这就需要三个指针来标记三个区域的分界线：

  • left：指向下一个 0 该放的位置（已排好的 0 区域的右边界）
  • right：指向上一个 2 该放的位置（已排好的 2 区域的左边界）
  • curr：当前正在扫描的元素

规则很简单：

  • nums[curr] == 0 → 和 nums[left] 交换，left++，curr++
  • nums[curr] == 1 → 已经在正确区域，curr++
  • nums[curr] == 2 → 和 nums[right] 交换，right–，但 curr 不动（因为换过来的元素还没被检查）

最后一个细节：为什么交换 2 时 curr 不前进？因为从 right 换过来的元素可能是 0 也可能是 1，它还没被处理过，需要原地再检查一次。而交换 0 时 curr 可以直接前进，因为 left 指向的元素只可能是 1（如果之前已经处理过的话），或者就是 left 和 curr 指向同一个位置——总之换过来的是安全的。

这就是鼎鼎大名的 荷兰国旗问题（Dutch National Flag Problem），由计算机科学家 Edsger Dijkstra 提出。

可视化一下

初始状态：

交换 curr 与 right 的 2，right 左移：

nums[curr] = 0，交换 curr 与 left，left 和 curr 都右移：

nums[curr] = 0，再次交换并右移：

nums[curr] = 2，交换 curr 与 right，right 左移，curr 不动：

nums[curr] = 1，curr 右移：

nums[curr] = 1，curr 右移。curr > right 时结束。

排好了。

复杂度分析表

版本 时间复杂度 空间复杂度 说明
计数排序（两趟遍历） O(n) O(1) 扫描两次：第一次计数，第二次重填
荷兰国旗（一趟遍历） O(n) O(1) 一次扫描完成，curr 最多移动 n 步
通用排序（如快排） O(n log n) O(log n) ~ O(n) 杀鸡用牛刀

荷兰国旗版本中，curr 指针虽然有时不前进，但每个元素最多被检查两次（一次放前面，一次放后面），总操作次数仍然是 O(n)。

延伸思考

这道题让我们学到了什么？

当数据范围有限时，我们可以利用这个特性设计比通用算法更高效的解法。

三个值（0、1、2）对应三个区域，三个指针恰好能维护这三个区域的边界。这种”分区”的思想并不仅限于排序：在处理需要将数组分成若干段的场景时，多指针往往是最简洁优雅的方案。

另外，注意交换 2 时 curr 不前进这个细节——这不仅仅是一道题的小技巧，它反映了状态机中”未处理完的不能放行”这一通用原则。

下次遇到类似问题优先观察什么？

  1. 数据的取值范围是否有限 —— 如果是，计数或分区方案可能比排序更优。
  2. 是否要求原地、一次遍历 —— 这是”荷兰国旗”类问题的典型信号。
  3. 是否可以分为三类或更多 —— 三指针方案可以推广到 k 个值的情况（虽然需要更多的指针和交换逻辑）。

荷兰国旗思想在后续很多题目中都会出现，比如快速排序的 partition 过程本质上就是维护两个边界（小于 pivot 和大于 pivot）。掌握这道题，等于掌握了双指针分区的核心直觉。
