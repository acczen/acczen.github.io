LeetCode 42. 接雨水（trapping-rain-water）

题目

LeetCode 42. trapping-rain-water

给定 n 个非负整数表示每个宽度为 1 的柱子的高度图，计算按此排列的柱子，下雨之后能接多少雨水。

示例 1：

示例 2：

提示：

  • n == height.length
  • 1 <= n <= 2 * 10^4
  • 0 <= height[i] <= 10^5

核心思路

拿到这道题，我们一起来想象一个场景：

面前摆了一排高低不一的木块。往上倒水，水会留在哪里？

不用急着往下翻。先聚焦在一个位置上——比如数组中间某个下标 i。想一想：这个位置能存住水，需要什么条件？

……

答案是：左右两边都要有比自己更高的柱子，把自己”兜”住。而且，这个位置能存住多少水，取决于左右两边较矮的那一边——水多了会从矮的那边溢出去。

用公式来表达：

$$water[i] = \max\bigl(0,\ \min(leftMax[i],\ rightMax[i]) - height[i]\bigr)$$

其中 $leftMax[i]$ 是 $i$ 左边最高的柱子，$rightMax[i]$ 是 $i$ 右边最高的。

有了这个公式，最直接的暴力想法就出来了：对每个位置，分别向左、向右扫描找最大值，然后套公式。

这能 AC 吗？$n$ 最大到 $2 \times 10^4$，$O(n^2)$ 意味着最坏 $4 \times 10^8$ 次操作，大概率超时。

瓶颈在哪？

很明显：每个位置 $i$ 都重新扫描一遍左右最大值。$leftMax[5]$ 和 $leftMax[6]$ 看的区间几乎重叠，但我们却算了两遍。

怎么优化？

能不能把”左右最大值”提前算好，存起来？

——这就是**动态规划（前缀最值）**的思路：

  1. 第一次从左到右遍历，算 $leftMax[i] = \max(leftMax[i-1],\ height[i])$，存入数组。
  2. 第二次从右到左遍历，算 $rightMax[i] = \max(rightMax[i+1],\ height[i])$，存入数组。
  3. 第三次遍历，对每个 $i$ 直接套公式累加。

时间 $O(n)$，空间 $O(n)$。已经是一个合格的解了。

还能更好吗？

我们盯着 DP 解法看：它维护了两个长度为 $n$ 的数组。真的需要每一个位置的完整 $leftMax$ 和 $rightMax$ 吗？

……

其实，对于位置 $i$，我们只关心 $leftMax[i]$ 和 $rightMax[i]$ 谁更小——较矮的那个才是木桶的短板，决定了水位。

这引出了双指针的核心直觉：

用两个指针 left 和 right 从两端向中间收缩。各自维护一侧”已扫描区域”的最大值。哪边的最大值更矮，就处理哪边——因为矮的那侧已经构成了该位置的天花板，另一侧再高也没用。

用一个例子感受一下：

  • leftMax = 3，rightMax = 4。左边更矮。
  • 对于 left 这个位置，左边的最高值是确定的（已经遍历完了），右边虽然可能有比 4 更高的柱子，但没关系——即使右边有更高的，leftMax = 3 已经是短板，水最高就到 3。
  • 所以我们可以安全地结算 left 处的水量，然后 left++。

严谨的论证：leftMax < rightMax 时，位置 left 的水量只取决于 leftMax，与右侧无关。 反过来 rightMax < leftMax 时也一样。

现在你可能还会想：有没有完全不同的视角？

当然有。前面我们都是”站在每个位置向上看，能存多少水”。换一个角度——“站在水往下看，水被什么形状兜住”——就引出了单调栈：

  • 维护一个严格递减的栈（存下标）。
  • 从左往右扫。遇到一个比栈顶高的柱子时，说明栈顶与当前柱子之间形成了一个”凹槽”——栈顶是槽底，栈中前一个元素是左墙，当前柱子是右墙。
  • 弹出栈顶，计算这一层凹槽的水量（宽度 $\times$ 高度差），累加。
方法 时间 空间 核心思想
暴力 $O(n^2)$ $O(1)$ 逐位向左右扫描
DP（前缀最值） $O(n)$ $O(n)$ 预计算左右最大值数组
双指针 $O(n)$ $O(1)$ 两端收缩，谁矮处理谁
单调栈 $O(n)$ $O(n)$ 按”层”找凹槽，栈维护递减序列

四种解法之间的关系很有趣：暴力暴露了重复计算 → DP 用空间换时间 → 双指针发现”不需要完整信息”，连空间也省了；单调栈则是完全不同的几何视角。

解法：双指针

这是时间和空间都最优的解法。

算法步骤

  1. 初始化 left = 0，right = n - 1，leftMax = 0，rightMax = 0，ans = 0。
  2. 当 left < right 时循环：
    • 如果 height[left] < height[right]：
      • leftMax = max(leftMax, height[left])
      • ans += leftMax - height[left]（leftMax 就是 left 的水位天花板）
      • left++
    • 否则：
      • rightMax = max(rightMax, height[right])
      • ans += rightMax - height[right]
      • right--
  3. 返回 ans。

为什么这样是对的？

关键论证：当 height[left] < height[right] 时，leftMax 来自左侧已遍历区域。而 height[right] 是右侧当前的高度，真正的 rightMax 要么就是 height[right]，要么比它更大——无论哪种，都不小于 leftMax。所以短板就是 leftMax，可以直接结算 left 的水量。

一句选边条件的说明

细心的你可能注意到循环里的条件是 height[left] < height[right]，而不是 leftMax < rightMax。两者都正确，但 height[left] < height[right] 更简洁：因为 leftMax >= height[left] 且 rightMax >= height[right]，所以 height[left] < height[right] 蕴含了 leftMax 很可能也是较小的一方，处理逻辑一致。

复杂度分析

维度 分析
时间复杂度 $O(n)$ — 每个元素最多被访问一次，left 和 right 各移动不超过 $n$ 步
空间复杂度 $O(1)$ — 只用了常数个变量

延伸思考

学到了什么？

这道题的精髓在于从”我需要完整信息”到”我只需要关键信息”的思维跃迁。DP 解法说”左右最大值都要算出来”，双指针追问：”等一下，两个值里我真的两个都需要吗？还是只需要较小的那个？”——少即是多，这正是算法优化的核心乐趣。

下次遇到类似问题，你会优先观察什么？

遇到”某个位置的状态由左右两侧信息共同决定”的问题时，先问自己一句：能不能只比较两侧的”当前最优”，而不需要完整的全局信息？ 如果能，双指针大概率是你最好的朋友。这道题和 LeetCode 11（盛最多水的容器）在直觉上有很强的互通性，推荐一并练习。
