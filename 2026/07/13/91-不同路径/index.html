不同路径

不同路径

核心思路

第一反应：暴力递归

拿到这道题，我们先理解一下问题。

一个机器人在左上角，要走到右下角，每次只能向右或向下走。问有多少条不同的路径。

让我们从最简单的情况开始想：如果网格只有一行（m = 1），机器人只能一直向右走，只有 1 条路径。同理，如果网格只有一列（n = 1），机器人只能一直向下走，也只有 1 条路径。

那么对于一般的情况呢？机器人从 (0, 0) 出发，第一步有两个选择：

  • 向右走到 (0, 1)，然后从 (0, 1) 走到右下角
  • 向下走到 (1, 0)，然后从 (1, 0) 走到右下角

也就是说：

从 (0, 0) 到终点的路径数 = 从 (0, 1) 到终点的路径数 + 从 (1, 0) 到终点的路径数

这不就是递归吗？我们把问题拆成了规模更小的子问题。

这个解法很直观，但让我们想想它的效率。

每个格子最多有两个分支，递归树是一棵二叉树。从 (0, 0) 走到 (m-1, n-1) 需要走 (m-1) + (n-1) 步，递归树的深度是 m + n - 2，而节点数是指数级的——大约是 O(2^{m+n})。

当 m = 3, n = 7 时（LeetCode 上的一个测试用例），2^{3+7} = 2^{10} = 1024，还勉强能接受。但当 m = 23, n = 12 时，2^{35} = 340 亿——这显然是不可接受的。

那么问题来了：这个递归为什么这么慢？有没有重复计算的地方？

发现重叠子问题

让我们画一个 3 x 3 的网格，看看递归调用树：

看到了吗？countPaths(1, 1) 被多次调用。对于同一个 (i, j)，它的路径数是固定的，但我们每次遇到它都要重新算一遍——这就是重复劳动。

有多少重复呢？ 如果直接递归，countPaths(1, 1) 会被调用 C(2, 1) = 2 次。随着网格变大，中间的格子会被反复调用成百上千次。

发现问题之后，解决方案就很自然了：把已经算过的结果记下来，下次直接取用。 这就是”记忆化搜索”（Memoization）。

记忆化搜索：缓存中间结果

加上缓存之后，每个格子只计算一次，总共 m * n 个格子，时间复杂度骤降到 O(m * n)。空间复杂度也是 O(m * n)，用来存储记忆化数组。

但这里有一个值得追问的地方：记忆化搜索本质上还是递归，递归调用有栈开销。能不能完全避免递归？

动态规划：从递归到递推

递归是”从顶向下”，先想终点怎么来，再一步步拆到起点。反过来想，我们能不能”从底向上”，从起点开始推？

让我们重新定义状态：

dp[i][j] 表示从 (0, 0) 出发，到达格子 (i, j) 的不同路径数。

从起点 (0, 0) 到 (0, 0)，不走就是 1 条路径，所以 dp[0][0] = 1。

要到达 (i, j)，只能从上方 (i-1, j) 走下来，或者从左方 (i, j-1) 走右来。所以：

这就是状态转移方程。

边界条件也很清晰：

  • 第一行 (0, j)：只能从左方过来，所以 dp[0][j] = 1
  • 第一列 (i, 0)：只能从上方过来，所以 dp[i][0] = 1

我们可以从 (0, 0) 开始层层递推，一行一行地填满整个表格，最终 dp[m-1][n-1] 就是答案。

暴力递归的问题彻底解决了。每次计算 dp[i][j] 只需要常数时间，总体时间复杂度 O(m * n)，空间复杂度也是 O(m * n)。

空间还能再省吗？

让我们再看一眼状态转移方程：

计算 dp[i][j] 的时候，只用到了当前行左边的格子 dp[i][j-1] 和上一行同一列的格子 dp[i-1][j]。换句话说，我们实际上不需要保留整个二维表格，只需要保留上一行的数据和当前行左边的一个值就够了。

更巧妙的是，如果我们用一个一维数组 dp[j] 来表示当前行，那么：

  • dp[j] 在更新前代表 dp[i-1][j]（上一行的值）
  • dp[j-1] 在更新后代表 dp[i][j-1]（当前行左边的值）

于是转移方程变成了：

什么意思？右边的 dp[j] 是上一行同一列的路径数，dp[j-1] 是当前行左边的路径数，加起来的和覆盖回 dp[j]，作为当前行这个格子的值。

太妙了——一行代码，同时用到了上一行和本行的信息。

空间复杂度从 O(m * n) 降到了 O(n)。如果 m < n，我们可以交换行列，保证 n 是较小的那个，这样空间就更省了。

另一种视野：组合数学

到这里我们已经找到了时空都很优秀的解法。但让我们退一步，重新审视这个问题——它有没有一个更本质的数学解释？

机器人从左上角走到右下角，总共需要走 (m-1) + (n-1) = m + n - 2 步，其中向下走 m - 1 步，向右走 n - 1 步。

关键来了：这 m + n - 2 步的顺序可以任意排列，只要总数固定。 每一种排列对应一条不同的路径。

那么问题就转化成了：从 m + n - 2 个位置中选出 m - 1 个（或 n - 1 个）放”向下走”，剩下的放”向右走”。这不就是组合数吗？

$$
\text{答案} = C_{m+n-2}^{m-1} = C_{m+n-2}^{n-1} = \frac{(m+n-2)!}{(m-1)!(n-1)!}
$$

以 m = 3, n = 7 为例：C_{3+7-2}^{3-1} = C_8^2 = 28，与动态规划的结果一致。

组合数学解法的时间复杂度是 O(min(m, n))（只算一个组合数），空间复杂度 O(1)。但需要注意整数溢出，计算过程中要用 long 并及时约分。

这里用了一个小技巧：边乘边除，避免中间结果溢出。因为 result * (total - k + i) 一定能被 i 整除（组合数一定是整数），所以不会有精度问题。

三种方案的对比

让我们梳理一下思考路径：

阶段 思路 发生了什么
暴力递归 把问题拆成两个子问题 指数级重复计算
记忆化搜索 缓存已算过的结果 时间降到 O(mn)，但保留递归栈
动态规划 自底向上递推填表 完全消除递归，O(mn) 时间
空间优化 DP 只用一维数组 O(mn) 时间，O(n) 空间
组合数学 把路径选择看成排列组合 O(min(m,n)) 时间，O(1) 空间

需要注意的是：组合数学解法虽然时空最优，但如果题目稍微变化（比如网格中出现障碍物，即 LeetCode 63「不同路径 II」），组合数学就不适用了。动态规划的可扩展性更强。

延伸思考

这道题让我们学到了什么？

「计数类」问题常常可以抽象成数学公式。 当我们需要数”有多少种方式”时，先看看能不能用排列组合直接算出来。

但更重要的收获是从递归到动态规划的思维跃迁。这道题的进化路径非常清晰：

  1. 先写出最直观的暴力递归——想清楚”递推关系”是什么
  2. 发现重叠子问题——意识到同一个 (i, j) 被反复计算
  3. 加缓存做记忆化——空间换时间，第一次优化
  4. 自底向上改递推——消除递归栈开销
  5. 观察依赖关系，压缩空间——只保留必要的数据

每一步的改进都源于一个简单的追问：”这里能不能省？”

下次遇到类似问题优先观察什么？

  1. 问题是否满足”无后效性”？ 即当前状态一旦确定，后续决策与之前如何到达这个状态无关。这道题中，到达 (i, j) 的路径数只与 (i-1, j) 和 (i, j-1) 有关，与之前具体的路径选择无关。这是能用 DP 的前提。

  2. 状态转移是几维的？依赖哪些方向？ 本题只有”上”和”左”两个方向的依赖，所以可以用一维数组滚动。如果依赖方向更多（比如”左上”也需要），空间压缩的方案就要相应调整。

  3. 能否归结为组合数学问题？ 当每一步的选择是二元的、且选择数量固定时，路径计数往往可以转化为组合数。但要注意，一旦出现障碍物、步数限制等额外约束，组合数学就会迅速复杂化，此时 DP 是更稳妥的选择。

  4. 网格 DP 的通用模板。 很多二维网格 DP 题目（不同路径、最小路径和、最大正方形等）都可以套用”逐行扫描，滚动数组”的框架。掌握这个套路，遇到类似题目就能快速起步。

这道题虽被标记为 Medium，但它其实是一扇门——推开它，后面是动态规划、组合数学、滚动数组优化等一整个知识体系。
